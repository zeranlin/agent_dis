# project

## 目录定位

这个目录用于承接面向政府采购招标文件审查场景的新一代产品实现。

它和现有 `app/` 的区别如下：

- `app/` 保留当前已经落地的 V1 最小服务骨架
- `project/` 用于承接后续按业务化解析、收缩式 `LLM` 执行和代码侧汇总重构后的正式实现

## 为什么要新建这个目录

当前仓库已经明确：

1. 现有 `app/` 更偏最小闭环与早期实现骨架
2. 后续主线需要更强的业务化解析能力
3. 后续技术架构将从“文本切块 + 审查执行”逐步升级到“业务模块 -> 审查单元 -> 风险识别 -> 结果汇总”

如果继续在 `app/` 中直接叠加，容易让：

1. 旧实现骨架
2. 收缩期修补逻辑
3. 后续业务化重构实现

混在一起，增加理解和迁移成本。

因此，后续建议：

1. `app/` 继续承担当前最小可运行实现
2. `project/` 逐步承接新的正式实现

## 后续承接范围

后续 `project/` 目录建议优先承接以下能力：

1. 文件接入与任务编排
2. 文本提取
3. 业务化解析
4. 审查输入生成
5. `LLM` 执行调度
6. 风险归并与结果输出

## 当前约束

当前只先建立目录和定位，不在本次变更中强制迁移现有 `app/` 代码。

后续如要开始迁移或重构，应继续遵守：

1. 先补文档
2. 再补目录结构
3. 最后迁移实现
