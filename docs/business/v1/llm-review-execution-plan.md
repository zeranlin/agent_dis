# V1 LLM 审查执行方案（草案）

## 文档目的

这份文档用于正式沉淀 V1 阶段面向高精度审查目标的 `LLM` 审查执行方案，统一描述：

- `LLM` 审查主流程
- `LLM` 审查输入模型
- `LLM` 审查输出模型
- 分片审查与批处理策略
- 风险归并与去重策略
- 最终结论生成策略

这份文档的定位不是替代当前最小骨架实现，而是作为后续把“关键词命中最小实现”升级为“`LLM` 高精度审查执行层”的正式方案底稿。

## 1. 当前定位

当前仓库中的审查执行链路，已经具备以下基础能力：

- 文件上传
- 文件解析与结构切分
- 审查输入装配
- 风险点与证据片段落地
- 最终结论与审查报告生成
- 页面查看与下载

但当前审查执行层仍属于最小占位实现，本质上是：

- 默认规则包
- 固定任务指令
- 关键词命中
- 结构化结果汇总

如果目标是把产品做成高精度的招标文件合规审查智能体，则后续主线应明确切换为：

`用 LLM 替换当前关键词审查执行层`

## 2. 方案边界

### 2.1 本方案要解决什么

- 让审查执行真正接入 `LLM`
- 让审查过程具备更强的语义理解能力
- 让输出仍然保持结构化、可追溯、可汇总
- 让结果能够继续复用当前结果页、报告页和下载链路

### 2.2 本方案暂不解决什么

- 多模型编排
- 多角色工作台
- 多文件联审
- 复杂审批流
- 大规模规则运营平台

## 3. 运行前提

当前 `LLM` 方案设计遵循以下前提：

- 智能体运行在独立隔离环境
- 底层以 `LLM` 算力为主
- 单并发能力大约为 `30 tokens/秒`
- 应优先控制上下文规模与调用次数
- 不默认依赖大量运营人员进行规则配置

说明：

- 运行时需要通过环境变量注入模型服务地址、模型名称和鉴权信息
- 不应在仓库文档或代码中固化明文密钥

## 4. LLM 审查主流程

### 4.1 一级主流程

面向 `LLM` 的正式审查主流程建议定义为：

1. 文件上传
2. 文件解析与结构切分
3. 审查输入装配
4. `LLM` 分片审查
5. 风险归并与去重
6. 最终结论生成
7. 人工查看结果与按需下钻证据

### 4.2 一级主流程说明

#### 1. 文件上传

- 审核人员提交待审查招标文件
- 当前仍以单文件为主
- 页面上传入口继续沿用当前上传页

#### 2. 文件解析与结构切分

- 系统对文件进行正文提取
- 完成章节、条款、段落等结构切分
- 形成可供 `LLM` 消费的片段对象

#### 3. 审查输入装配

- 装配片段正文
- 装配章节上下文
- 装配候选规则
- 装配固定任务指令
- 形成 `LLM` 批次输入

#### 4. LLM 分片审查

- 按批次将连续片段送给 `LLM`
- 由 `LLM` 输出结构化风险结果
- 每条结果都必须包含证据与审查理由

#### 5. 风险归并与去重

- 将多个批次的原始命中结果归并为风险组
- 处理重复、冲突和多证据挂接

#### 6. 最终结论生成

- 基于风险组与完整性信息自动生成总体结论
- 渲染最终结论与审查报告

#### 7. 人工查看结果与按需下钻证据

- 审核人员先看总体结论
- 再看重点风险组
- 需要时继续下钻证据片段和审查理由

## 5. LLM 审查输入模型

### 5.1 输入对象

一次 `LLM` 审查输入建议由以下 `6` 个一级对象组成：

1. 任务对象
2. 文件对象
3. 结构片段对象
4. 上下文对象
5. 规则对象
6. 任务指令对象

#### 5.1.1 任务对象字段约束

任务对象建议收成 `5` 个核心字段和 `3` 个运行约束字段。

核心字段：

1. `task_id`
2. `project_id`
3. `document_id`
4. `file_name`
5. `review_batch_id`

运行约束字段：

1. `batch_index`
2. `batch_count`
3. `batch_scope`

字段级原则：

1. 核心标识字段创建后不可变
2. 同一任务必须可追溯到项目、文件和批次
3. 必须支持单批失败后的精确重跑
4. 必须支持后续风险归并与结果汇总按任务统一聚合

收口句：

`任务对象负责定义“这是谁的任务、审哪份文件、当前是第几批、这一批覆盖哪里”。`

#### 5.1.2 文件对象字段约束

文件对象建议收成 `4` 个基础字段和 `4` 个审查上下文字段。

基础字段：

1. `document_id`
2. `document_type`
3. `document_format`
4. `source_name`

审查上下文字段：

1. `review_scope`
2. `document_summary`
3. `chapter_catalog`
4. `token_estimate`

字段级原则：

1. 文件信息必须真实来源于上传文件
2. 重点服务解析、批处理和追溯，不只服务页面展示
3. 文件对象不直接承载条款级风险判断
4. 后续风险、证据和结论都必须能反查到原始文件

收口句：

`文件对象负责定义“当前审的是哪份文件、是什么格式、处于什么业务审查范围、能够提供哪些文件级上下文”。`

#### 5.1.3 结构片段对象字段约束

结构片段对象是 `LLM` 主审查输入单位，建议收成 `5` 个核心字段和 `4` 个定位与上下文字段。

核心字段：

1. `fragment_id`
2. `fragment_type`
3. `fragment_text`
4. `chapter_title`
5. `location_label`

定位与上下文字段：

1. `order_index`
2. `clause_type`
3. `parent_section_path`
4. `token_estimate`

字段级原则：

1. 必须可直接送审
2. 必须可定位原文
3. 必须保留原文关键表述
4. 必须可排序切批
5. 必须可挂接章节与相邻上下文

收口句：

`结构片段对象是 LLM 审查的最小主输入单位，必须同时满足可送审、可定位、可排序和可补上下文。`

#### 5.1.4 上下文对象字段约束

上下文对象建议收成 `4` 个主上下文字段和 `3` 个控制字段。

主上下文字段：

1. `previous_fragments`
2. `next_fragments`
3. `chapter_summary`
4. `section_path`

控制字段：

1. `context_window_size`
2. `context_truncation_flag`
3. `context_source`

字段级原则：

1. 只做辅助，不替代当前片段原文
2. 局部优先，不做全文件长上下文
3. 上下文规模必须可控
4. 上下文来源必须可追溯

收口句：

`上下文对象只负责给当前片段补局部理解环境，不能替代当前片段原文。`

#### 5.1.5 规则对象字段约束

规则对象建议收成 `6` 个核心字段和 `4` 个增强字段。

核心字段：

1. `rule_id`
2. `rule_code`
3. `rule_name`
4. `rule_domain`
5. `execution_level`
6. `risk_level`

增强字段：

1. `rule_definition`
2. `rule_examples`
3. `rule_boundary`
4. `version`

字段级原则：

1. 必须可直接进入 `LLM` 输入
2. 必须同时服务模型理解和人工展示
3. 必须支持后续章节级规则预筛
4. 必须支持规则版本追溯

收口句：

`规则对象负责定义“查什么、按什么级别查、命中后按什么风险解释”。`

#### 5.1.6 任务指令对象字段约束

任务指令对象建议收成 `4` 个核心字段和 `3` 个约束字段。

核心字段：

1. `system_instruction`
2. `output_schema`
3. `evidence_requirement`
4. `reasoning_requirement`

约束字段：

1. `conservative_policy`
2. `json_only`
3. `instruction_version`

字段级原则：

1. 只定义模型行为护栏，不承载业务事实
2. 必须稳定且版本化
3. 必须直接约束输出结构
4. 必须让模型在证据不足时保持保守

收口句：

`任务指令对象定义“模型该怎么审、该怎么输出、证据和理由必须怎么给”。`

### 5.2 输入字段

#### 1. 任务对象

| 字段 | 字段说明 |
| --- | --- |
| `task_id` | 当前审查任务唯一标识 |
| `project_id` | 当前项目唯一标识 |
| `document_id` | 当前文件唯一标识 |
| `file_name` | 原始文件名称 |
| `review_batch_id` | 当前 `LLM` 分批审查批次标识 |

#### 2. 文件对象

| 字段 | 字段说明 |
| --- | --- |
| `document_type` | 文件业务类型，如招标文件正文 |
| `document_format` | 文件格式，如 `PDF`、`DOC`、`DOCX` |
| `review_scope` | 本轮审查范围说明 |
| `source_name` | 页面展示或上传来源名称 |

#### 3. 结构片段对象

| 字段 | 字段说明 |
| --- | --- |
| `fragment_id` | 当前审查片段唯一标识 |
| `fragment_type` | 片段类型，如条款片段、段落片段 |
| `chapter_title` | 当前片段所属章节标题 |
| `location_label` | 当前片段位置标签 |
| `fragment_text` | 当前片段原文正文 |
| `clause_type` | 当前片段的结构类型补充说明 |
| `order_index` | 当前片段在全文中的顺序位置 |

#### 4. 上下文对象

| 字段 | 字段说明 |
| --- | --- |
| `previous_fragments` | 当前片段前置相邻片段摘要或原文 |
| `next_fragments` | 当前片段后置相邻片段摘要或原文 |
| `chapter_summary` | 当前章节摘要说明 |
| `section_path` | 当前片段在章节结构中的路径 |
| `related_fragments` | 与当前片段关联的补充片段列表 |

#### 5. 规则对象

| 字段 | 字段说明 |
| --- | --- |
| `rule_id` | 规则唯一标识 |
| `rule_code` | 规则编码 |
| `rule_name` | 规则名称 |
| `rule_domain` | 规则所属风险域 |
| `execution_level` | 规则执行级别 |
| `risk_level` | 规则默认风险等级 |
| `rule_definition` | 规则业务定义文本 |
| `rule_examples` | 规则示例或典型命中样例 |
| `rule_boundary` | 规则适用边界与不适用边界 |

#### 6. 任务指令对象

| 字段 | 字段说明 |
| --- | --- |
| `system_instruction` | 固定系统审查指令 |
| `output_schema` | 规定模型输出字段结构 |
| `evidence_requirement` | 规定证据必须引用原文和位置 |
| `reasoning_requirement` | 规定必须输出审查理由 |
| `conservative_policy` | 证据不足时应保持保守 |
| `json_only` | 是否只允许输出 `JSON` |

#### 5.2.1 必须字段

V1 必须字段建议只保留“没有它就无法稳定执行审查”的字段，分为四组：

任务与文件标识字段：

1. `task_id`
2. `document_id`
3. `review_batch_id`
4. `file_name`
5. `document_format`

主审查片段字段：

1. `fragment_id`
2. `fragment_type`
3. `fragment_text`
4. `chapter_title`
5. `location_label`

规则字段：

1. `rule_id`
2. `rule_code`
3. `rule_name`
4. `rule_domain`
5. `execution_level`
6. `risk_level`
7. `rule_definition`

指令字段：

1. `system_instruction`
2. `output_schema`
3. `evidence_requirement`
4. `reasoning_requirement`
5. `json_only`

收口句：

`必须字段 = 标识任务与文件、能直接审片段、能直接读规则、能直接约束模型输出。`

#### 5.2.2 增强字段

增强字段用于提升模型理解稳定性和后续系统编排能力，建议包括：

文件级增强字段：

1. `review_scope`
2. `document_summary`
3. `chapter_catalog`
4. `token_estimate`

片段级增强字段：

1. `clause_type`
2. `order_index`
3. `parent_section_path`

上下文增强字段：

1. `previous_fragments`
2. `next_fragments`
3. `chapter_summary`
4. `section_path`
5. `related_fragments`
6. `context_window_size`
7. `context_truncation_flag`
8. `context_source`

规则增强字段：

1. `rule_examples`
2. `rule_boundary`
3. `version`

指令增强字段：

1. `conservative_policy`
2. `instruction_version`

收口句：

`增强字段 = 提升模型理解稳定性、降低误判率、支撑批处理和追溯。`

#### 5.2.3 字段分层原则

输入字段建议固定成 `三层分层原则`：

第一层：必跑层

- 缺少该层字段则不应执行 `LLM` 审查

第二层：增强层

- 缺少该层字段仍可执行，但准确率和稳定性下降

第三层：控制层

- 不直接参与业务判断，但直接影响批处理、上下文预算、重试和排障

收口句：

`输入字段不追求越多越好，而是先保证可执行，再逐步增强精度与稳定性。`

### 5.3 输入批次结构

一次 `LLM` 调用建议固定为：

- `batch_meta`
- `document_context`
- `fragments`
- `rules`
- `instructions`

建议结构示意：

```json
{
  "batch_meta": {
    "task_id": "task_xxx",
    "project_id": "project_xxx",
    "document_id": "document_xxx",
    "file_name": "招标文件.docx",
    "review_batch_id": "batch_001",
    "batch_index": 1
  },
  "document_context": {
    "document_type": "招标文件正文",
    "document_format": "DOCX",
    "review_scope": "公开招标-货物与服务",
    "chapter_title": "第一章 资格要求",
    "chapter_summary": "本章主要描述供应商资格与基本要求。"
  },
  "fragments": [],
  "rules": [],
  "instructions": {}
}
```

收口原则：

- 同一批只放一个主章节上下文
- 同一批只放连续片段
- 同一批共享候选规则与固定任务指令

#### 5.3.1 批次对象结构

建议一次 `LLM` 输入批次固定成：

1. 批次主对象
2. 文件上下文子对象
3. 片段列表子对象
4. 规则列表子对象
5. 指令子对象

批次主对象建议至少包含：

- `task_id`
- `project_id`
- `document_id`
- `review_batch_id`
- `batch_index`
- `batch_scope`

收口句：

`批次对象本身负责标识与控制，具体业务内容分到文件上下文、片段列表、规则列表和指令对象中。`

#### 5.3.2 片段列表组织结构

片段列表建议采用有序数组，且每个片段对象至少包含：

- `fragment_id`
- `fragment_type`
- `chapter_title`
- `location_label`
- `fragment_text`
- `order_index`
- `clause_type`

组织原则：

1. 按 `order_index` 升序
2. 片段必须连续
3. 同章优先
4. 默认每批 `3` 个片段，最多 `5` 个
5. 正文优先，章节标题不作为主审查片段

收口句：

`片段列表必须是有序、连续、同章优先的小批量正文片段集合。`

#### 5.3.3 共享规则与指令结构

建议采用：

- 批次共享规则
- 批次共享指令

共享规则建议字段：

- `rule_id`
- `rule_code`
- `rule_name`
- `rule_domain`
- `execution_level`
- `risk_level`
- `rule_definition`
- `rule_examples`
- `rule_boundary`
- `version`

共享指令建议字段：

- `system_instruction`
- `output_schema`
- `evidence_requirement`
- `reasoning_requirement`
- `conservative_policy`
- `json_only`
- `instruction_version`

收口句：

`规则和指令必须批次共享，这样才能减少 token 浪费并提升输出稳定性。`

## 6. LLM 审查输出模型

### 6.1 输出对象

一次 `LLM` 审查输出建议由以下 `5` 个一级对象组成：

1. 批次输出对象
2. 片段审查结果对象
3. 风险结果对象
4. 证据对象
5. 审查说明对象

#### 6.1.1 批次输出对象

批次输出对象用于回答“这一批是否成功返回可用结果”，建议字段包括：

- `task_id`
- `review_batch_id`
- `batch_index`
- `output_status`
- `error_message`
- `model_name`
- `finished_at`

#### 6.1.2 片段审查结果对象

片段审查结果对象用于回答“这个片段是否命中、是否需要人工确认”，建议字段包括：

- `fragment_id`
- `fragment_type`
- `location_label`
- `review_status`
- `risk_count`
- `need_human_confirm`

#### 6.1.3 风险结果对象

风险结果对象用于回答“命中了什么规则、是什么风险、后续如何归并”，建议字段包括：

- `rule_id`
- `rule_code`
- `rule_name`
- `hit`
- `risk_title`
- `risk_level`
- `execution_level`
- `risk_description`
- `risk_category`
- `merge_key`
- `duplicate_flag`

#### 6.1.4 证据对象

证据对象用于回答“为什么说这里有问题”，建议字段包括：

- `evidence_text`
- `evidence_location`
- `evidence_note`
- `evidence_source_type`
- `evidence_fragment_id`

#### 6.1.5 审查说明对象

审查说明对象用于回答“为什么命中、把握有多大、哪里仍不确定”，建议字段包括：

- `review_reasoning`
- `confidence`
- `need_human_confirm`
- `uncertainty_note`
- `boundary_note`
- `decision_basis`

### 6.2 输出字段

#### 1. 批次输出对象

| 字段 | 字段说明 |
| --- | --- |
| `task_id` | 所属审查任务 `ID` |
| `review_batch_id` | 当前 `LLM` 批次 `ID` |
| `batch_index` | 当前批次序号 |
| `output_status` | 当前批次输出状态 |
| `error_message` | 当前批次异常说明 |
| `model_name` | 本次调用使用的模型名称 |
| `finished_at` | 批次输出完成时间 |

#### 2. 片段审查结果对象

| `review_status` | 该片段审查状态 |

#### 6.2.1 必须字段

输出必须字段建议分为五组：

批次级必须字段：

1. `task_id`
2. `review_batch_id`
3. `batch_index`
4. `output_status`

片段级必须字段：

1. `fragment_id`
2. `location_label`
3. `review_status`

风险级必须字段：

1. `rule_id`
2. `rule_code`
3. `rule_name`
4. `hit`
5. `risk_title`
6. `risk_level`
7. `risk_description`

证据级必须字段：

1. `evidence_text`
2. `evidence_location`

说明级必须字段：

1. `review_reasoning`
2. `need_human_confirm`

收口句：

`必须字段 = 批次状态 + 片段定位 + 风险识别 + 原文证据 + 审查理由。`

#### 6.2.2 增强字段

增强字段用于提升归并、排序、追溯和人工复核体验，建议包括：

批次级增强字段：

1. `error_message`
2. `model_name`
3. `finished_at`

片段级增强字段：

1. `fragment_type`
2. `risk_count`

风险级增强字段：

1. `execution_level`
2. `risk_category`
3. `merge_key`
4. `duplicate_flag`

证据级增强字段：

1. `evidence_note`
2. `evidence_source_type`
3. `evidence_fragment_id`

说明级增强字段：

1. `confidence`
2. `uncertainty_note`
3. `boundary_note`
4. `decision_basis`

收口句：

`增强字段 = 让结果更适合归并、排序、追溯和人工复核。`

#### 6.2.3 字段分层原则

输出字段建议固定成三层：

第一层：入库必需层

- 没有这些字段，这条结果不应进入主链路

第二层：归并增强层

- 没有这些字段也能入库，但归并效果会下降

第三层：人工解释层

- 没有这些字段也能生成结果，但人工查看体验会变差

收口句：

`输出字段必须先保证系统能接，再增强系统能归并，最后增强人工能理解。`

### 6.3 输出批次结构

一次调用的整体输出建议固定成：

- `batch_result`
- `fragment_results`
- `risk_items`
- `meta`

收口原则：

- 必须返回合法 `JSON`
- 不允许混入解释性自然语言前后缀
- 每个命中项必须带 `evidence` 和 `reasoning`
- 即使没有命中，也应返回 `fragment_results`

#### 6.3.1 批次结果层

批次结果层建议至少包含：

- `task_id`
- `review_batch_id`
- `batch_index`
- `output_status`
- `error_message`

作用：

- 先判断这一批能不能进入后续主链路

#### 6.3.2 片段结果层

片段结果层建议通过 `fragment_results` 承载，每个元素至少包含：

- `fragment_id`
- `fragment_type`
- `location_label`
- `review_status`
- `risk_count`
- `need_human_confirm`

作用：

- 让系统知道每个片段各自的审查状态

#### 6.3.3 风险结果层

风险结果层建议通过 `risk_items` 承载，每个元素应包含：

- 风险字段
- 证据对象
- 审查说明对象

作用：

- 作为后续风险归并、报告生成和结果排序的主输入

#### 6.3.4 元信息层

元信息层建议通过 `meta` 承载，至少包含：

- `model_name`
- `json_valid`
- `schema_version`

作用：

- 支撑排障、追溯和版本管理

#### 6.3.5 输出批次约束

建议固定以下约束：

1. 必须输出合法 `JSON`
2. 不允许带前后解释性自然语言
3. 每个 `risk_item` 必须绑定 `fragment_id`
4. 每个命中项必须带 `evidence` 和 `reasoning`
5. 即使无命中，也必须返回 `fragment_results`

## 7. 分片审查与批处理策略

### 7.1 分片单位

分片单位建议固定成 `3` 层结构：

1. 章节
2. 条款片段
3. 段落片段

其中：

- `章节` 只作为上下文，不直接送审
- `条款片段` 是主审查单位
- `段落片段` 是补充审查单位

#### 7.1.1 章节

章节对象只负责提供上下文，主要承担：

- `chapter_title`
- 章节摘要
- 章节路径

约束：

1. 不直接作为主审查单元
2. 必须能稳定挂接到条款片段或段落片段
3. 必须支持同章节批处理

#### 7.1.2 条款片段

条款片段是 V1 主审查单位，适用于：

- 有明确编号
- 有明确位置
- 有完整语义单元

约束：

1. 条款编号优先
2. 保留条款标题和紧随正文
3. 必须能直接引用原文

#### 7.1.3 段落片段

段落片段用于补足没有明确条款编号但正文独立完整的区域。

约束：

1. 必须是完整自然段
2. 不应把多个独立要求混成一个片段
3. 必须能定位回原文位置

#### 7.1.4 分片优先级

建议固定优先级：

1. `条款片段`
2. `段落片段`
3. `章节仅上下文`

收口句：

`章节负责上下文，条款片段优先主审，段落片段负责补充覆盖。`

### 7.2 批次组织方式

批次组织建议采用：

`同章节 + 连续片段 + 小窗口 + 规则预筛`

#### 7.2.1 同章节组织

建议一批尽量只放同一章节的片段。

原因：

1. 章节语义更一致
2. 规则预筛更稳定
3. 模型不易分散注意力

#### 7.2.2 连续片段组织

建议同一批必须由相邻片段组成。

原因：

1. 相邻条款常有补充关系
2. 远距离片段混装会放大误判风险

#### 7.2.3 小窗口组织

建议：

1. 默认 `3` 个片段一批
2. 最多 `5` 个片段一批
3. 超长片段可降到 `1-2` 个

#### 7.2.4 规则预筛组织

建议先按章节模块预筛候选规则，再送 `LLM`。

例如：

- 资格要求章节优先资格与公平竞争类规则
- 评分办法章节优先评审解释性规则
- 采购需求章节优先需求合理性规则

#### 7.2.5 批次形成顺序

建议固定顺序：

1. 按章节分组
2. 章节内按顺序排序
3. 连续窗口切批
4. 为每批预筛规则
5. 形成最终 `LLM` 输入批次

收口句：

`批次组织的关键不是多，而是同章、连续、小批、先筛规则。`

具体口径：

1. 同一批尽量只放同一个章节的片段
2. 同一批片段必须连续
3. 默认每批 `3` 个片段，最多 `5` 个
4. 单批规则需先做预筛选，不直接喂全部规则

### 7.3 调用与重试策略

`LLM` 调用策略建议采用：

`串行小批次 + 批次级重试 + 失败批次局部降级`

#### 7.3.1 调用策略

建议：

1. 默认按批次串行调用
2. 一批成功后再处理下一批
3. V1 不优先追求复杂并发

原因：

1. 当前环境吞吐有限
2. 串行更稳、更容易排障
3. 更适合早期高精度方案收口

#### 7.3.2 重试策略

建议每批最多：

1. 首次调用 `1` 次
2. 失败后重试 `2` 次
3. 总计最多 `3` 次

失败判定包括：

1. HTTP 调用失败
2. 请求超时
3. 返回非 `JSON`
4. `JSON` 结构缺字段
5. 输出为空或不可解析

#### 7.3.3 缩批策略

建议：

1. 第一次失败：原参数重试
2. 第二次失败：缩小批次
3. 例如从 `5` 个片段降到 `2-3` 个片段

#### 7.3.4 局部失败策略

建议：

1. 某个批次失败，不直接把整份文件判死
2. 成功批次结果先保留
3. 失败批次记录失败原因
4. 最终任务允许出现：
   - `completed`
   - `completed_with_partial_failure`
   - `failed`

#### 7.3.5 输出校验策略

每次模型返回后，系统必须做三层校验：

1. 协议校验
2. 格式校验
3. 结构校验

只有全部通过，结果才进入后续归并。

#### 7.3.6 V1 收口建议

收口句：

`V1 调用策略优先稳定，不优先吞吐，失败批次允许局部降级而不拖垮整任务。`

建议规则：

1. 单批首次调用 `1` 次
2. 失败后最多重试 `2` 次
3. 第二次重试时可自动缩小批次大小
4. 单批失败不应直接拖垮整任务

输出状态建议分为：

- `success`
- `partial`
- `failed`

## 8. 风险归并与去重策略

### 8.1 风险归并对象

归并后的最小单位应是：

`风险组对象`

每个风险组对象建议包含：

- 风险组对象本身
- 证据集合对象
- 关联片段集合对象
- 归并来源对象

#### 8.1.1 风险组对象

风险组对象是最终给审核人员查看和给系统汇总的核心对象。

建议字段：

- `risk_group_id`
- `risk_title`
- `risk_level`
- `risk_category`
- `rule_code`
- `main_risk_description`
- `main_review_reasoning`
- `need_human_confirm`

作用：

- 表示一个最终可展示、可汇总、可人工复核的审核问题

##### 8.1.1.1 主键与唯一性规则

- 每个风险组必须有唯一 `risk_group_id`
- `risk_group_id` 由系统生成，不由模型直接生成
- 同一任务内唯一即可，不要求跨任务稳定
- 建议格式：`rg_{task_id}_{seq}`

##### 8.1.1.2 基础展示字段约束

- `risk_title` 用于页面和报告主标题
- `risk_title` 必须短句化，建议不超过 `30` 字
- `main_risk_description` 用于解释“是什么问题”
- `main_review_reasoning` 用于解释“为什么判成这个问题”

##### 8.1.1.3 风险等级字段约束

- 取值固定为：`high`、`medium`、`low`
- 不允许自由文本等级
- 页面展示时再映射为“高 / 中 / 低”

##### 8.1.1.4 风险分类字段约束

- 取值来自固定枚举，不允许模型临时发明新类
- V1 建议枚举：
  - `资格条件`
  - `采购需求`
  - `评分办法`
  - `程序条款`
  - `合同履约`
  - `政策功能`
  - `其他`

##### 8.1.1.5 主描述与主理由生成规则

- 主描述回答“问题是什么”
- 主理由回答“为什么判定有问题”
- 两者都必须能回指到证据，不允许脱离证据写抽象判断

##### 8.1.1.6 人工确认标记规则

- `need_human_confirm` 为布尔值
- 证据冲突、规则边界模糊、输出不一致时置为 `true`
- 页面和报告都要显式展示

#### 8.1.2 证据集合对象

证据集合对象挂在风险组下，表示这个问题由哪些证据支撑。

建议字段：

- `evidence_items`
- `evidence_count`
- `main_evidence_text`
- `main_evidence_location`

作用：

- 表示这个风险组有哪些证据
- 明确最核心的主证据是什么

##### 8.1.2.1 证据项对象字段定义

每个证据项建议包含：

- `evidence_id`
- `fragment_id`
- `evidence_text`
- `location_label`
- `chapter_title`

##### 8.1.2.2 主证据选取规则

- 优先选择表达最完整、最直接、位置最明确的证据作为主证据

##### 8.1.2.3 证据数量统计规则

- `evidence_count` 统计去重后的证据项数量

##### 8.1.2.4 证据去重规则

- 文本完全相同且位置相同的证据合并为一条
- 文本近似但位置不同的不合并

##### 8.1.2.5 证据展示顺序规则

- 先主证据，再其他证据
- 其余按文档出现顺序展示

#### 8.1.3 关联片段集合对象

关联片段集合对象用于表示这个风险覆盖了哪些片段和位置。

建议字段：

- `fragment_ids`
- `location_labels`
- `chapter_titles`
- `fragment_count`

作用：

- 帮助系统和人工理解问题分布在哪些条款或段落

##### 8.1.3.1 片段引用字段定义

建议包含：

- `fragment_ids`
- `location_labels`
- `chapter_titles`

##### 8.1.3.2 位置标签生成规则

- 优先使用“章节 + 条款号”
- 无条款号时退化为“章节 + 顺序号”

##### 8.1.3.3 章节标题汇总规则

- 去重后保留章节标题列表

##### 8.1.3.4 片段数量统计规则

- `fragment_count` 统计关联片段总数

##### 8.1.3.5 片段排序规则

- 按原文顺序排序，不按风险强弱排序

#### 8.1.4 归并来源对象

归并来源对象用于保留风险组的形成来源。

建议字段：

- `source_risk_ids`
- `source_batch_ids`
- `source_rule_ids`
- `source_count`

作用：

- 保证风险组不是黑盒结果
- 保留全部来源，支持排查和追溯

##### 8.1.4.1 来源风险引用规则

- 保留所有来源 `source_risk_ids`

##### 8.1.4.2 来源批次引用规则

- 保留 `source_batch_ids`，支持追溯是哪个分片命中的

##### 8.1.4.3 来源规则引用规则

- 保留 `source_rule_ids`，支持追溯用了哪些规则依据

##### 8.1.4.4 来源计数规则

- `source_count` 为去重后的来源命中数

##### 8.1.4.5 追溯保留边界

- 页面不必全展示
- 底层结果对象必须保留

收口句：

`风险归并后的最小展示和汇总单位不是原始命中项，而是风险组对象。`

### 8.2 归并键设计

V1 主归并键建议采用：

`rule_code|risk_category|chapter_title|evidence_keyword`

辅助判断字段建议包括：

- `location_label`
- `fragment_id`
- `evidence_text` 相似度
- `risk_title`
- `need_human_confirm`

#### 8.2.1 主归并键

V1 主归并键建议由以下字段组成：

- `rule_code`
- `risk_category`
- `chapter_title`
- `evidence_keyword`

建议格式：

`rule_code|risk_category|chapter_title|evidence_keyword`

示例：

`R1|公平竞争|第一章资格要求|本地注册`

作用：

- 作为 V1 最核心的归并依据

##### 8.2.1.1 `rule_code` 标准化规则

- 必须来自规则包编码
- 无规则编码时用 `UNKNOWN_RULE`

##### 8.2.1.2 `risk_category` 标准化规则

- 必须映射到固定分类枚举

##### 8.2.1.3 `chapter_title` 标准化规则

- 去首尾空格
- 统一全半角
- 统一常见噪声写法

##### 8.2.1.4 `evidence_keyword` 生成规则

- 从证据文本中抽取最核心触发词
- 词长建议 `2` 到 `8` 字

##### 8.2.1.5 主归并键拼接格式约束

- 固定格式：`rule_code|risk_category|chapter_title|evidence_keyword`

#### 8.2.2 归并键设计原因

主归并键这样设计的原因是：

1. 同一个问题通常会命中同一规则或同类规则
2. 同一个问题通常处于同一风险域
3. 同一个问题通常集中在同一章节
4. 同一个问题通常会反复出现相同或相近的证据关键词

因此 V1 不建议只靠：

- `rule_code`
- 文本相似度
- 位置标签

中的单一维度做判断。

##### 8.2.2.1 为什么不能只靠规则编码

- 同一规则可能命中多个不同问题

##### 8.2.2.2 为什么不能只靠文本相似度

- 文本相似不等于同一风险，容易误并

##### 8.2.2.3 为什么要引入章节维度

- 同一问题通常集中在同一章节上下文

##### 8.2.2.4 为什么要引入证据关键词

- 便于把重复表述收束成同一触发点

##### 8.2.2.5 V1 与后续版本的归并差异边界

- V1 用可解释硬规则
- 后续版本再引入语义相似度和学习型归并

#### 8.2.3 辅助归并键

当主归并键不能完全确定时，再看以下辅助字段：

- `location_label`
- `fragment_id`
- `risk_title`
- `evidence_text` 相似度
- `need_human_confirm`

作用：

- 避免误拆
- 避免误合并
- 给后续软归并提供支持

##### 8.2.3.1 `location_label` 使用规则

- 用于进一步确认是否为同一位置问题

##### 8.2.3.2 `fragment_id` 使用规则

- 同片段优先判为重复

##### 8.2.3.3 `risk_title` 使用规则

- 仅辅助，不作为主归并依据

##### 8.2.3.4 `evidence_text` 相似度使用规则

- 可作为软判断，不作为唯一依据

##### 8.2.3.5 `need_human_confirm` 兜底规则

- 边界模糊时优先升为人工确认

#### 8.2.4 证据关键词来源

V1 建议采用可控方式生成 `evidence_keyword`：

1. 由系统从 `evidence_text` 中抽取触发词
2. 或由模型直接输出 `evidence_keyword`
3. 再作为 `merge_key` 的最后一段

示例关键词：

- 本地注册
- 综合评价
- 品牌型号
- 专利要求

##### 8.2.4.1 系统抽取方式

- 用规则抽取常见限制词、歧视词、排他词

##### 8.2.4.2 模型输出方式

- 模型可给 `evidence_keyword`，但必须经过系统清洗

##### 8.2.4.3 关键词清洗与归一化

- 去空白
- 去标点
- 统一同义表达

##### 8.2.4.4 缺失关键词时的兜底策略

- 使用截短后的主证据文本片段

##### 8.2.4.5 多关键词竞争时的主关键词选取

- 优先选最能表达风险触发点的词，而不是最常见词

收口句：

`V1 归并键先用“规则 + 风险域 + 章节 + 证据关键词”做主键，再用位置和文本相似度做辅助判断。`

### 8.3 冲突与重复处理

处理原则建议为：

1. 重复结果优先归并
2. 冲突结果优先保守
3. 无法自动一致时统一上升为“需人工重点确认”

优先级建议：

1. 有明确证据优先于证据不足
2. 高风险优先于中低风险
3. 需要人工确认优先保留
4. 理由更完整优先保留

#### 8.3.1 重复处理规则

重复的典型情况包括：

- 同一条款被不同批次重复命中
- 同一章节多个连续片段表达同一个问题
- 模型用不同表述重复输出同一个风险

V1 建议：

1. 主归并键一致时，直接归并成一个风险组
2. 主归并键不同，但位置接近、证据相似、语义明显一致时，作为疑似重复归并
3. 无法确定时，暂不强合并，但标记 `duplicate_flag = true`

##### 8.3.1.1 完全重复判定规则

- 主归并键一致且位置接近，直接归并

##### 8.3.1.2 疑似重复判定规则

- 主归并键不同，但证据和位置高度接近，可标为疑似重复

##### 8.3.1.3 不可自动归并的边界

- 风险分类不同
- 章节明显不同
- 证据含义相反时不自动归并

##### 8.3.1.4 `duplicate_flag` 标记规则

- 无法确定是否重复时标记 `duplicate_flag = true`

##### 8.3.1.5 重复归并后的证据保留规则

- 合并后保留全部来源证据，不只留一条

#### 8.3.2 冲突处理规则

冲突的典型情况包括：

- 一批判定高风险，另一批认为证据不足
- 不同批次给出不同风险等级
- 不同批次对是否需要人工确认判断不一致

V1 建议按以下优先级处理：

1. 有明确证据优先于证据不足
2. 高风险优先于中低风险
3. 需要人工确认优先保留
4. 理由更完整优先保留
5. 无法自动决断时统一转为 `need_human_confirm = true`

##### 8.3.2.1 风险等级冲突处理

- 取更高等级

##### 8.3.2.2 证据充分性冲突处理

- 明确证据优先于模糊证据

##### 8.3.2.3 人工确认标记冲突处理

- 任一来源要求人工确认，则结果保留人工确认

##### 8.3.2.4 审查理由冲突处理

- 保留更完整、可回指证据的理由

##### 8.3.2.5 无法自动决断的升级规则

- 统一升级为 `need_human_confirm = true`

#### 8.3.3 主版本保留规则

每个风险组建议只保留一个“主版本”用于页面与报告展示。

主版本建议按以下顺序选择：

1. 风险等级更高
2. 证据更完整
3. 审查理由更完整
4. 位置更明确
5. 输出结构更完整

其他版本不丢弃，应保留到：

- `source_risk_ids`
- `source_batch_ids`

中。

##### 8.3.3.1 主版本选择排序规则

- 高等级优先
- 证据完整优先
- 理由完整优先

##### 8.3.3.2 主版本字段覆盖规则

- 主版本负责页面和报告主展示字段

##### 8.3.3.3 非主版本保留规则

- 非主版本只退到底层追溯对象，不丢弃

##### 8.3.3.4 页面展示与底层追溯分离规则

- 页面只展示主版本
- 底层保留全来源

##### 8.3.3.5 主版本切换触发条件

- 新来源证据更强、风险更高时可切换主版本

#### 8.3.4 V1 最保守原则

在政府采购审查场景里，冲突处理必须优先保守：

1. 不强行自动得出过于肯定的判断
2. 冲突明显时优先提示人工重点确认
3. 保留冲突来源，避免错误自信

##### 8.3.4.1 保守原则适用边界

- 适用于所有自动归并和结论汇总阶段

##### 8.3.4.2 哪些情形必须上升人工确认

- 证据冲突
- 结果冲突
- 规则边界不清

##### 8.3.4.3 哪些情形允许系统直接归并

- 规则、章节、证据、位置都明显一致

##### 8.3.4.4 保守原则对页面展示的影响

- 页面需明确展示“需人工重点确认”

##### 8.3.4.5 保守原则对总体结论的影响

- 有人工确认项时，总体结论需更保守

收口句：

`重复结果优先归并，冲突结果优先保守，无法自动一致时统一上升为“需人工重点确认”。`

## 9. 最终结论生成策略

### 9.1 总体结论对象

最终结论建议由：

- 总体结论对象
- 风险统计对象
- 重点关注对象
- 运行完整性对象

共同组成。

#### 9.1.1 总体结论对象

总体结论对象用于回答“这份文件整体怎么看”。

建议字段：

- `conclusion_id`
- `task_id`
- `overall_conclusion`
- `overall_risk_level`
- `need_human_priority_review`
- `conclusion_summary`

##### 9.1.1.1 主键与关联关系

- `conclusion_id` 唯一关联 `task_id`

##### 9.1.1.2 `overall_conclusion` 取值范围

- 为标准化总结文案，不允许自由长文

##### 9.1.1.3 `overall_risk_level` 取值范围

- 固定为：`high`、`medium`、`low`

##### 9.1.1.4 `need_human_priority_review` 判定规则

- 高风险、人工确认项、失败批次任一成立即为真

##### 9.1.1.5 `conclusion_summary` 生成规则

- 用 `1` 到 `2` 句概括整体情况，不展开证据细节

#### 9.1.2 风险统计对象

风险统计对象用于回答“问题大概有多少、严重程度如何”。

建议字段：

- `high_risk_count`
- `medium_risk_count`
- `low_risk_count`
- `need_human_confirm_count`
- `risk_group_count`

##### 9.1.2.1 高中低风险计数口径

- 按归并后的风险组计数，不按原始命中项计数

##### 9.1.2.2 人工确认计数口径

- 统计 `need_human_confirm = true` 的风险组数量

##### 9.1.2.3 风险组总数计数口径

- 为最终去重后的总风险组数

##### 9.1.2.4 去重后统计规则

- 所有统计都基于归并结果，不基于原始批次结果

##### 9.1.2.5 统计对象与页面展示的映射关系

- 页面摘要区直接消费该统计对象

#### 9.1.3 重点关注对象

重点关注对象用于回答“审核人员最先要看哪些风险组”。

建议字段：

- `top_risk_group_ids`
- `top_risk_titles`
- `priority_review_notes`

##### 9.1.3.1 Top 风险组筛选规则

- 优先高风险，再人工确认，再证据更完整的项

##### 9.1.3.2 Top 数量上限

- V1 建议最多展示 `3` 到 `5` 个

##### 9.1.3.3 标题生成规则

- 直接使用 `risk_title`

##### 9.1.3.4 优先复核说明生成规则

- 解释“为什么建议先看这几项”

##### 9.1.3.5 无重点风险时的空结果规则

- 可为空，但结构字段仍保留

#### 9.1.4 运行完整性对象

运行完整性对象用于回答“这份结论是不是完整可靠”。

建议字段：

- `review_completed`
- `partial_failure_flag`
- `failed_batch_count`
- `completeness_note`

##### 9.1.4.1 `review_completed` 判定规则

- 所有批次完成且无失败时为真

##### 9.1.4.2 `partial_failure_flag` 判定规则

- 有部分失败批次时为真

##### 9.1.4.3 `failed_batch_count` 统计规则

- 统计失败分片数量

##### 9.1.4.4 `completeness_note` 生成规则

- 用标准说明表达结果是否完整

##### 9.1.4.5 完整性对象对页面提示的影响

- 不完整时页面必须显示提醒

收口句：

`总体结论对象不是一句总结，而是“总体结论 + 风险统计 + 重点关注 + 完整性说明”的组合对象。`

### 9.2 结论生成规则

总体结论建议由系统规则生成，而不是交给 `LLM` 自由生成。

建议规则：

1. 先判断是否存在未完成审查批次
2. 再判断高风险数量
3. 再判断中风险数量
4. 最后判断是否只有低风险或无命中

总体风险等级建议采用保守规则：

- 有高风险则整体高风险
- 无高风险但有中风险则整体中风险
- 仅低风险或无命中则整体低风险

#### 9.2.1 总体结论生成原则

总体结论建议完全由系统规则生成，优先保证：

- 稳定
- 可解释
- 可复核
- 不漂移

原则：

1. 不新增事实
2. 不扩展模型未给出的业务判断
3. 只基于风险组、人工确认标记和完整性信息汇总

##### 9.2.1.1 系统生成与模型生成的边界

- 模型负责识别风险
- 系统负责最终汇总结论

##### 9.2.1.2 可解释性约束

- 每个结论都必须能解释来源

##### 9.2.1.3 不新增事实约束

- 不允许写出证据中不存在的结论

##### 9.2.1.4 输入依赖边界

- 只依赖风险组、统计对象、完整性对象

##### 9.2.1.5 规则优先级总表

- 完整性优先于等级
- 等级优先于文案润色

#### 9.2.2 生成顺序

建议按以下顺序生成：

1. 先判断是否存在失败批次
2. 再判断高风险数量
3. 再判断中风险数量
4. 最后判断是否只有低风险或无命中

建议输出口径：

- 有高风险：存在明显风险，建议优先复核高风险条款
- 无高风险但中风险较多：存在多处疑点，建议重点复核相关条款
- 只有低风险：未发现明显高风险问题，但仍建议人工复核提示项
- 无命中：当前未发现明显风险，但不替代人工最终判断

##### 9.2.2.1 失败批次优先判断

- 先判断是否存在部分失败

##### 9.2.2.2 高风险优先判断

- 有高风险则优先输出高风险口径

##### 9.2.2.3 中风险优先判断

- 无高风险再判断中风险

##### 9.2.2.4 低风险与无命中判断

- 只剩低风险或无命中时输出保守低风险口径

##### 9.2.2.5 输出文案模板规则

- 文案必须模板化，不能任意漂移

#### 9.2.3 总体风险等级规则

建议采用保守规则：

- 有高风险组：`overall_risk_level = 高`
- 无高风险但有中风险：`overall_risk_level = 中`
- 只有低风险：`overall_risk_level = 低`
- 无风险命中：`overall_risk_level = 低`

##### 9.2.3.1 高风险触发条件

- 存在任一高风险组

##### 9.2.3.2 中风险触发条件

- 无高风险但存在中风险组

##### 9.2.3.3 低风险触发条件

- 只有低风险组或无风险命中

##### 9.2.3.4 无命中归低风险的业务原因

- 无命中不代表无问题，只代表未发现明显风险

##### 9.2.3.5 风险等级与展示色标映射

- `high = 红`
- `medium = 橙`
- `low = 蓝 / 灰`

#### 9.2.4 是否需要人工优先复核

满足以下任一条件，则：

- `need_human_priority_review = true`

条件包括：

1. 存在高风险组
2. `need_human_confirm_count > 0`
3. 存在失败批次

##### 9.2.4.1 高风险触发人工优先复核

- 任何高风险组都触发

##### 9.2.4.2 人工确认标记触发人工优先复核

- 任一风险组要求人工确认都触发

##### 9.2.4.3 失败批次触发人工优先复核

- 存在失败批次即触发

##### 9.2.4.4 多条件并存时的说明生成

- 说明文案应覆盖多个触发原因

##### 9.2.4.5 页面提示与下载报告口径统一

- 页面和下载文件用同一逻辑生成

#### 9.2.5 完整性说明规则

如果存在失败批次，应明确写出：

- 部分内容未完成审查
- 当前结论不完整
- 建议人工补充复核

如果不存在失败批次，则明确写出：

- 当前审查已完成
- 结果可用于后续人工复核

##### 9.2.5.1 部分失败时的标准说明

- 明确提示“部分内容未完成审查”

##### 9.2.5.2 全部完成时的标准说明

- 明确提示“当前审查已完成”

##### 9.2.5.3 说明文案长度控制

- 一般控制在 `1` 到 `2` 句

##### 9.2.5.4 说明与总体结论的关系

- 完整性说明是总体结论的补充，不替代总体结论

##### 9.2.5.5 说明与人工复核提示的关系

- 不完整时必须同时提示人工补充复核

收口句：

`V1 总体结论由规则引擎基于风险组数量、等级、人工确认标记和审查完整性稳定生成。`

### 9.3 报告输出结构

V1 输出建议采用：

- `最终结论.md`
- `审查报告.md`

两份输出共用同一套结构化结果对象渲染，不分别维护两套业务逻辑。

其中：

#### 最终结论

建议包含：

1. 文件信息
2. 总体结论
3. 风险统计
4. 重点关注
5. 完整性说明

#### 审查报告

建议包含：

1. 文件信息
2. 总体结论
3. 风险统计
4. 风险组列表
5. 证据明细
6. 审查说明
7. 运行说明

#### 9.3.1 两份输出

V1 输出建议固定成两份：

1. `最终结论`
2. `审查报告`

定位：

- 最终结论负责快读
- 审查报告负责深读

##### 9.3.1.1 最终结论定位

- 给审核人员快速读

##### 9.3.1.2 审查报告定位

- 给审核人员深度读、留档、下载

##### 9.3.1.3 两份输出的共享数据源

- 必须共用同一结构化结果对象

##### 9.3.1.4 两份输出的差异边界

- 只允许展示层差异，不允许业务逻辑差异

##### 9.3.1.5 下载对象命名规则

- 统一命名，如 `最终结论.md`、`审查报告.md`

#### 9.3.2 最终结论结构

建议包含以下 `5` 段：

1. 文件信息
2. 总体结论
3. 风险统计
4. 重点关注
5. 完整性说明

##### 9.3.2.1 文件信息段

- 文件名
- 任务编号
- 生成时间

##### 9.3.2.2 总体结论段

- 一句话结论和整体风险等级

##### 9.3.2.3 风险统计段

- 高中低风险数量、人工确认数量

##### 9.3.2.4 重点关注段

- 最需要优先查看的风险组

##### 9.3.2.5 完整性说明段

- 是否完整、是否有失败批次

#### 9.3.3 审查报告结构

建议包含以下 `7` 段：

1. 文件信息
2. 总体结论
3. 风险统计
4. 风险组列表
5. 证据明细
6. 审查说明与人工确认提示
7. 运行说明与完整性说明

##### 9.3.3.1 文件信息段

- 与最终结论一致

##### 9.3.3.2 总体结论段

- 与最终结论一致

##### 9.3.3.3 风险统计段

- 与最终结论一致

##### 9.3.3.4 风险组列表段

- 展示全部风险组

#### 9.3.4 风险组展示结构

每个风险组建议统一展示：

- 风险标题
- 风险等级
- 规则编码
- 风险分类
- 主风险描述
- 主审查理由
- 是否需要人工重点确认
- 关联条款位置
- 证据片段列表

##### 9.3.4.1 风险标题展示规则

- 用简洁标题，不用长句堆叠

##### 9.3.4.2 风险等级与分类展示规则

- 风险等级和分类必须同时展示

##### 9.3.4.3 规则编码与理由展示规则

- 每项都显示 `rule_code` 和主理由

##### 9.3.4.4 关联位置展示规则

- 显示章节标题和位置标签

##### 9.3.4.5 证据片段展示规则

- 先主证据，后补充证据

#### 9.3.5 统一输出原则

两份输出必须来自同一套结构化结果对象，不分别维护两套业务逻辑。

建议顺序：

1. 先生成统一结果对象
2. 再渲染成 `最终结论.md`
3. 再渲染成 `审查报告.md`

##### 9.3.5.1 统一结果对象定义

- 页面、接口、下载都基于同一结果对象

##### 9.3.5.2 页面与下载共用口径

- 不允许页面和下载出现不同结论

##### 9.3.5.3 先结构化后渲染原则

- 先生成结构化数据，再渲染成 Markdown 或页面

##### 9.3.5.4 不允许双份业务逻辑

- 不允许各自维护一套生成逻辑

##### 9.3.5.5 后续扩展字段兼容规则

- 新字段只加不破，不影响既有展示口径

收口句：

`V1 报告输出采用“统一结构化结果对象 + 最终结论 + 审查报告”两层渲染方式。`

## 10. 当前阶段结论

当前关于 `LLM` 审查执行层，已经形成以下统一认识：

1. 当前关键词命中最小实现只能作为占位，不是最终目标
2. 若目标是高精度审查，必须正式接入 `LLM`
3. `LLM` 应替换的是“审查执行层”，而不是推翻整条主链路
4. 审查执行必须采用“结构切分 + 批次输入 + 结构化输出 + 系统归并”的方式
5. 总体结论与报告输出应由系统稳定生成，而不是依赖自由文本生成
