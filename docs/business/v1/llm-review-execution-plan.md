# V1 LLM 审查执行方案（草案）

## 文档目的

这份文档用于正式沉淀 V1 阶段面向高精度审查目标的 `LLM` 审查执行方案，统一描述：

- `LLM` 审查主流程
- `LLM` 审查输入模型
- `LLM` 审查输出模型
- 分片审查与批处理策略
- 风险归并与去重策略
- 最终结论生成策略

这份文档的定位不是替代当前最小骨架实现，而是作为后续把“关键词命中最小实现”升级为“`LLM` 高精度审查执行层”的正式方案底稿。

## 1. 当前定位

当前仓库中的审查执行链路，已经具备以下基础能力：

- 文件上传
- 文件解析与结构切分
- 审查输入装配
- 风险点与证据片段落地
- 最终结论与审查报告生成
- 页面查看与下载

但当前审查执行层仍属于最小占位实现，本质上是：

- 默认规则包
- 固定任务指令
- 关键词命中
- 结构化结果汇总

如果目标是把产品做成高精度的招标文件合规审查智能体，则后续主线应明确切换为：

`用 LLM 替换当前关键词审查执行层`

## 2. 方案边界

### 2.1 本方案要解决什么

- 让审查执行真正接入 `LLM`
- 让审查过程具备更强的语义理解能力
- 让输出仍然保持结构化、可追溯、可汇总
- 让结果能够继续复用当前结果页、报告页和下载链路

### 2.2 本方案暂不解决什么

- 多模型编排
- 多角色工作台
- 多文件联审
- 复杂审批流
- 大规模规则运营平台

## 3. 运行前提

当前 `LLM` 方案设计遵循以下前提：

- 智能体运行在独立隔离环境
- 底层以 `LLM` 算力为主
- 单并发能力大约为 `30 tokens/秒`
- 应优先控制上下文规模与调用次数
- 不默认依赖大量运营人员进行规则配置

说明：

- 运行时需要通过环境变量注入模型服务地址、模型名称和鉴权信息
- 不应在仓库文档或代码中固化明文密钥

## 4. LLM 审查主流程

### 4.1 一级主流程

面向 `LLM` 的正式审查主流程建议定义为：

1. 文件上传
2. 文件解析与结构切分
3. 审查输入装配
4. `LLM` 分片审查
5. 风险归并与去重
6. 最终结论生成
7. 人工查看结果与按需下钻证据

### 4.2 一级主流程说明

#### 1. 文件上传

- 审核人员提交待审查招标文件
- 当前仍以单文件为主
- 页面上传入口继续沿用当前上传页

#### 2. 文件解析与结构切分

- 系统对文件进行正文提取
- 完成章节、条款、段落等结构切分
- 形成可供 `LLM` 消费的片段对象

#### 3. 审查输入装配

- 装配片段正文
- 装配章节上下文
- 装配候选规则
- 装配固定任务指令
- 形成 `LLM` 批次输入

#### 4. LLM 分片审查

- 按批次将连续片段送给 `LLM`
- 由 `LLM` 输出结构化风险结果
- 每条结果都必须包含证据与审查理由

#### 5. 风险归并与去重

- 将多个批次的原始命中结果归并为风险组
- 处理重复、冲突和多证据挂接

#### 6. 最终结论生成

- 基于风险组与完整性信息自动生成总体结论
- 渲染最终结论与审查报告

#### 7. 人工查看结果与按需下钻证据

- 审核人员先看总体结论
- 再看重点风险组
- 需要时继续下钻证据片段和审查理由

## 5. LLM 审查输入模型

### 5.1 输入对象

一次 `LLM` 审查输入建议由以下 `6` 个一级对象组成：

1. 任务对象
2. 文件对象
3. 结构片段对象
4. 上下文对象
5. 规则对象
6. 任务指令对象

### 5.2 输入字段

#### 1. 任务对象

| 字段 | 字段说明 |
| --- | --- |
| `task_id` | 当前审查任务唯一标识 |
| `project_id` | 当前项目唯一标识 |
| `document_id` | 当前文件唯一标识 |
| `file_name` | 原始文件名称 |
| `review_batch_id` | 当前 `LLM` 分批审查批次标识 |

#### 2. 文件对象

| 字段 | 字段说明 |
| --- | --- |
| `document_type` | 文件业务类型，如招标文件正文 |
| `document_format` | 文件格式，如 `PDF`、`DOC`、`DOCX` |
| `review_scope` | 本轮审查范围说明 |
| `source_name` | 页面展示或上传来源名称 |

#### 3. 结构片段对象

| 字段 | 字段说明 |
| --- | --- |
| `fragment_id` | 当前审查片段唯一标识 |
| `fragment_type` | 片段类型，如条款片段、段落片段 |
| `chapter_title` | 当前片段所属章节标题 |
| `location_label` | 当前片段位置标签 |
| `fragment_text` | 当前片段原文正文 |
| `clause_type` | 当前片段的结构类型补充说明 |
| `order_index` | 当前片段在全文中的顺序位置 |

#### 4. 上下文对象

| 字段 | 字段说明 |
| --- | --- |
| `previous_fragments` | 当前片段前置相邻片段摘要或原文 |
| `next_fragments` | 当前片段后置相邻片段摘要或原文 |
| `chapter_summary` | 当前章节摘要说明 |
| `section_path` | 当前片段在章节结构中的路径 |
| `related_fragments` | 与当前片段关联的补充片段列表 |

#### 5. 规则对象

| 字段 | 字段说明 |
| --- | --- |
| `rule_id` | 规则唯一标识 |
| `rule_code` | 规则编码 |
| `rule_name` | 规则名称 |
| `rule_domain` | 规则所属风险域 |
| `execution_level` | 规则执行级别 |
| `risk_level` | 规则默认风险等级 |
| `rule_definition` | 规则业务定义文本 |
| `rule_examples` | 规则示例或典型命中样例 |
| `rule_boundary` | 规则适用边界与不适用边界 |

#### 6. 任务指令对象

| 字段 | 字段说明 |
| --- | --- |
| `system_instruction` | 固定系统审查指令 |
| `output_schema` | 规定模型输出字段结构 |
| `evidence_requirement` | 规定证据必须引用原文和位置 |
| `reasoning_requirement` | 规定必须输出审查理由 |
| `conservative_policy` | 证据不足时应保持保守 |
| `json_only` | 是否只允许输出 `JSON` |

### 5.3 输入批次结构

一次 `LLM` 调用建议固定为：

- `batch_meta`
- `document_context`
- `fragments`
- `rules`
- `instructions`

建议结构示意：

```json
{
  "batch_meta": {
    "task_id": "task_xxx",
    "project_id": "project_xxx",
    "document_id": "document_xxx",
    "file_name": "招标文件.docx",
    "review_batch_id": "batch_001",
    "batch_index": 1
  },
  "document_context": {
    "document_type": "招标文件正文",
    "document_format": "DOCX",
    "review_scope": "公开招标-货物与服务",
    "chapter_title": "第一章 资格要求",
    "chapter_summary": "本章主要描述供应商资格与基本要求。"
  },
  "fragments": [],
  "rules": [],
  "instructions": {}
}
```

收口原则：

- 同一批只放一个主章节上下文
- 同一批只放连续片段
- 同一批共享候选规则与固定任务指令

## 6. LLM 审查输出模型

### 6.1 输出对象

一次 `LLM` 审查输出建议由以下 `5` 个一级对象组成：

1. 批次输出对象
2. 片段审查结果对象
3. 风险结果对象
4. 证据对象
5. 审查说明对象

### 6.2 输出字段

#### 1. 批次输出对象

| 字段 | 字段说明 |
| --- | --- |
| `task_id` | 所属审查任务 `ID` |
| `review_batch_id` | 当前 `LLM` 批次 `ID` |
| `batch_index` | 当前批次序号 |
| `output_status` | 当前批次输出状态 |
| `error_message` | 当前批次异常说明 |
| `model_name` | 本次调用使用的模型名称 |
| `finished_at` | 批次输出完成时间 |

#### 2. 片段审查结果对象

| 字段 | 字段说明 |
| --- | --- |
| `fragment_id` | 被审查片段 `ID` |
| `fragment_type` | 片段类型 |
| `location_label` | 片段位置标签 |
| `review_status` | 该片段审查状态 |
| `risk_count` | 该片段命中的风险数量 |
| `need_human_confirm` | 该片段是否需要人工重点确认 |

#### 3. 风险结果对象

| 字段 | 字段说明 |
| --- | --- |
| `rule_id` | 命中的规则 `ID` |
| `rule_code` | 命中的规则编码 |
| `rule_name` | 命中的规则名称 |
| `hit` | 是否命中规则 |
| `risk_title` | 风险标题 |
| `risk_level` | 风险等级 |
| `execution_level` | 执行级别 |
| `risk_description` | 风险描述 |
| `risk_category` | 风险分类 |
| `merge_key` | 后续归并使用的归并键 |
| `duplicate_flag` | 是否疑似重复风险 |

#### 4. 证据对象

| 字段 | 字段说明 |
| --- | --- |
| `evidence_text` | 原文证据片段 |
| `evidence_location` | 证据位置标签 |
| `evidence_note` | 证据说明 |
| `evidence_source_type` | 证据来源类型 |
| `evidence_fragment_id` | 证据对应片段 `ID` |

#### 5. 审查说明对象

| 字段 | 字段说明 |
| --- | --- |
| `review_reasoning` | 模型审查理由 |
| `confidence` | 命中置信度 |
| `need_human_confirm` | 是否需要人工重点确认 |
| `uncertainty_note` | 不确定性说明 |
| `boundary_note` | 适用边界说明 |
| `decision_basis` | 判断依据摘要 |

### 6.3 输出批次结构

一次调用的整体输出建议固定成：

- `batch_result`
- `fragment_results`
- `risk_items`
- `meta`

收口原则：

- 必须返回合法 `JSON`
- 不允许混入解释性自然语言前后缀
- 每个命中项必须带 `evidence` 和 `reasoning`
- 即使没有命中，也应返回 `fragment_results`

## 7. 分片审查与批处理策略

### 7.1 分片单位

分片单位建议固定成 `3` 层结构：

1. 章节
2. 条款片段
3. 段落片段

其中：

- `章节` 只作为上下文，不直接送审
- `条款片段` 是主审查单位
- `段落片段` 是补充审查单位

### 7.2 批次组织方式

批次组织建议采用：

`同章节 + 连续片段 + 小窗口 + 规则预筛`

具体口径：

1. 同一批尽量只放同一个章节的片段
2. 同一批片段必须连续
3. 默认每批 `3` 个片段，最多 `5` 个
4. 单批规则需先做预筛选，不直接喂全部规则

### 7.3 调用与重试策略

`LLM` 调用策略建议采用：

`串行小批次 + 批次级重试 + 失败批次局部降级`

建议规则：

1. 单批首次调用 `1` 次
2. 失败后最多重试 `2` 次
3. 第二次重试时可自动缩小批次大小
4. 单批失败不应直接拖垮整任务

输出状态建议分为：

- `success`
- `partial`
- `failed`

## 8. 风险归并与去重策略

### 8.1 风险归并对象

归并后的最小单位应是：

`风险组对象`

每个风险组对象建议包含：

- 风险组对象本身
- 证据集合对象
- 关联片段集合对象
- 归并来源对象

### 8.2 归并键设计

V1 主归并键建议采用：

`rule_code|risk_category|chapter_title|evidence_keyword`

辅助判断字段建议包括：

- `location_label`
- `fragment_id`
- `evidence_text` 相似度
- `risk_title`
- `need_human_confirm`

### 8.3 冲突与重复处理

处理原则建议为：

1. 重复结果优先归并
2. 冲突结果优先保守
3. 无法自动一致时统一上升为“需人工重点确认”

优先级建议：

1. 有明确证据优先于证据不足
2. 高风险优先于中低风险
3. 需要人工确认优先保留
4. 理由更完整优先保留

## 9. 最终结论生成策略

### 9.1 总体结论对象

最终结论建议由：

- 总体结论对象
- 风险统计对象
- 重点关注对象
- 运行完整性对象

共同组成。

### 9.2 结论生成规则

总体结论建议由系统规则生成，而不是交给 `LLM` 自由生成。

建议规则：

1. 先判断是否存在未完成审查批次
2. 再判断高风险数量
3. 再判断中风险数量
4. 最后判断是否只有低风险或无命中

总体风险等级建议采用保守规则：

- 有高风险则整体高风险
- 无高风险但有中风险则整体中风险
- 仅低风险或无命中则整体低风险

### 9.3 报告输出结构

V1 输出建议采用：

- `最终结论.md`
- `审查报告.md`

两份输出共用同一套结构化结果对象渲染，不分别维护两套业务逻辑。

其中：

#### 最终结论

建议包含：

1. 文件信息
2. 总体结论
3. 风险统计
4. 重点关注
5. 完整性说明

#### 审查报告

建议包含：

1. 文件信息
2. 总体结论
3. 风险统计
4. 风险组列表
5. 证据明细
6. 审查说明
7. 运行说明

## 10. 当前阶段结论

当前关于 `LLM` 审查执行层，已经形成以下统一认识：

1. 当前关键词命中最小实现只能作为占位，不是最终目标
2. 若目标是高精度审查，必须正式接入 `LLM`
3. `LLM` 应替换的是“审查执行层”，而不是推翻整条主链路
4. 审查执行必须采用“结构切分 + 批次输入 + 结构化输出 + 系统归并”的方式
5. 总体结论与报告输出应由系统稳定生成，而不是依赖自由文本生成

## 11. 后续继续细化顺序

后续若继续细化，应遵循既定分层准则，按以下顺序推进：

1. `1. LLM 审查输入模型` 的更下一级字段和对象关系
2. `2. LLM 审查输出模型` 的技术入库结构
3. `3. 分片审查与批处理策略` 的执行编排实现
4. `4. 风险归并与去重策略` 的技术规则与数据结构
5. `5. 最终结论生成策略` 的模板与渲染实现
