# V1 验收标准（草案）

## 文档目的

这份文档用于定义 V1 产品的最小验收标准，帮助技术负责人、开发人员和后续测试人员对齐“做到什么程度算可交付”。

## 1. 验收原则

V1 验收优先关注三件事：

1. 主流程是否跑通
2. 核心结果是否可用
3. 输出是否与既定方案一致

V1 不以复杂运营后台、规则在线配置或高覆盖率地方适配作为验收前提。

## 2. 主流程验收

以下四步主流程必须可跑通：

1. 文件进入系统
2. AI 发起审查
3. 生成审查结果
4. 人工查看结果

### 2.1 文件进入系统

验收标准：

- 支持上传 1 份文件
- 支持 `PDF` 和 `Word`
- 文件上传成功后可形成唯一任务标识

### 2.2 AI 发起审查

验收标准：

- 文件上传后自动触发审查
- 审查使用解析后的文件内容
- 审查依据包含默认规则包和固定审查任务指令

### 2.3 生成审查结果

验收标准：

- 系统能够自动汇总中间结果
- 至少生成最终结论和审查报告
- 两类结果均可被程序读取和展示

### 2.4 人工查看结果

验收标准：

- 结果页可在线查看最终结论
- 结果页可在线查看审查报告
- 支持下载最终结论和审查报告

## 3. 页面验收

V1 页面只要求覆盖三个状态页面：

1. 上传页
2. 审核中状态
3. 结果页

验收标准：

- 上传页能提交文件
- 审核中状态能体现任务处理中
- 结果页能展示最终结论与审查报告

## 4. 结果对象验收

V1 至少需要稳定输出以下对象：

1. 上传结果对象
2. 审核状态对象
3. 审查结果对象

验收标准：

- 三类对象字段结构稳定
- 状态字段取值清晰
- 审查结果对象包含 `overall_conclusion` 和 `report_markdown`

## 5. 审查内容验收

V1 的中间产物至少应覆盖：

1. 风险点
2. 证据片段
3. 风险说明

验收标准：

- 每个风险点可追溯到命中的规则
- 每个风险点可定位到原文位置
- 风险说明能够表达“为什么这里有问题”

## 6. 规则侧验收

V1 应默认内置规则包 1.0。

验收标准：

- 默认规则包可被系统加载
- 规则元数据可映射到输出结果
- 至少覆盖当前约定的 12 条默认规则

## 7. 结果文件验收

V1 下载结果建议统一采用 `Markdown`。

验收标准：

- 能下载最终结论 `.md`
- 能下载审查报告 `.md`
- 文件内容和页面展示保持一致

## 8. 工程验收

技术交付至少应满足以下要求：

- 代码仓库中可以找到默认规则包配置
- 代码仓库中可以找到固定审查任务指令
- 审查执行链路具备基础日志或状态标识
- 单次任务失败时能够看到失败状态

## 9. 非验收项

以下内容不作为 V1 验收前提：

- 多文件联审
- 合同草案强制联审
- 在线规则维护后台
- 复杂审批流
- 精细化地方规则差异适配
- 全自动正式定稿

## 10. 当前结论

V1 的验收标准应坚持“主流程跑通、结果可用、交付边界清晰”。

只要能够稳定完成上传、自动审查、结果生成和结果查看下载，就可以视为达到第一版可交付门槛。
