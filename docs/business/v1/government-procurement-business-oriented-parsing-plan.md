# 政府采购招标文件业务化解析方案草案

## 文档目的

这份文档用于把当前已讨论并收口的业务边界、招标文件结构要求和审查对象要求，进一步整理成一版“业务化解析方案草案”。

这份方案要解决的不是“如何把文件切得更碎”，而是：

1. 如何按政府采购招标文件的天然结构去解析
2. 如何把解析结果直接服务于后续合规审查
3. 如何避免为了适配 `LLM` 而把文件切成业务上不自然的碎片

## 一、方案背景

当前项目的目标，不是通用文档理解，而是：

- 面向政府采购招标文件合规审查

因此，解析层不能只按技术角度追求：

1. 分块更细
2. 文本更短
3. 片段更多

而应优先满足业务审查的需要：

1. 审核人员能理解
2. 规则能命中到自然业务单元
3. 风险、证据、结论能稳定回溯到文件中的真实位置

## 二、核心判断

当前解析策略应从“技术切分优先”调整为“业务结构优先”。

正式判断如下：

1. 招标文件不是普通长文，而是结构高度稳定的业务文件
2. 真正需要被审查的，不是任意文本块，而是带有业务语义的条款单元
3. 解析层应先识别文件中的业务模块，再识别模块中的可审查单元
4. 只有在业务单元过长、内部包含多个独立判断点时，才继续做轻量技术切分

一句话收口：

- 先按业务结构解析，再按技术需要轻量切分

## 三、解析目标

业务化解析层应服务以下目标：

1. 识别文件中的主要业务模块
2. 将文件拆为可直接审查的业务单元
3. 让每个审查单元都能被规则命中、被证据引用、被结果回溯
4. 减少因碎片化切分带来的重复命中和语义断裂

## 四、业务化解析原则

### 1. 章节优先，不按整页切

政府采购招标文件的目录和章节通常具有稳定结构。

因此，解析时优先按：

1. 一级章节
2. 二级章节
3. 条款编号

来建立结构，而不是按页码或固定字数切块。

### 2. 条款优先，不按自然段盲切

真正承载审查意义的，通常是条款，而不是自然段。

因此，优先把以下内容识别为条款：

1. 资格条件单条要求
2. 技术参数单条要求
3. 评分项单条规则
4. 合同责任单条约定
5. 验收、付款、违约、质保等单条约定

### 3. 表格按“业务行”解析，不只转成连续文本

招标文件中很多高价值风险并不在普通段落里，而在：

1. 技术参数表
2. 评分表
3. 商务响应表
4. 偏离表
5. 资格条件列表或对照表

因此，表格不应只做“整表转文本”，而应尽量按：

1. 表头
2. 行
3. 参数项
4. 评分项

来恢复业务单元。

### 4. 轻切分只作为补充，不作为主策略

当一个业务单元过长时，允许继续细切分。

但细切分的前提必须是：

1. 原业务单元已经建立
2. 细切分不会破坏原业务单元归属
3. 细切分后仍能回溯到原条款和原位置

### 5. 审查可追溯性优先

每个解析结果必须保留足够定位信息，至少能回答：

1. 它属于哪个章节
2. 它属于哪类业务模块
3. 它对应哪个条款或哪一行表格
4. 它在文件中大致位于哪里

## 五、政府采购招标文件的优先业务模块

结合前面已讨论的本体和审查框架，当前解析层应优先识别以下模块。

### 1. 项目与文件基础信息模块

例如：

1. 项目名称
2. 采购人
3. 代理机构
4. 采购方式
5. 项目编号
6. 预算金额

这部分主要服务于项目上下文，不是高优先审查区，但应被稳定提取。

### 2. 资格条件模块

这是高优先审查区。

重点承载以下规则：

1. 地域限制
2. 所有制 / 规模歧视
3. 资格条件过高
4. 业绩要求定向或过高

推荐最小解析单元：

1. 单条资格条件
2. 单条业绩要求
3. 单条资质 / 证书要求

### 3. 采购需求模块

这是高优先审查区。

重点承载以下内容：

1. 技术参数
2. 服务范围
3. 功能要求
4. 交付要求
5. 配置要求

推荐最小解析单元：

1. 单条技术参数
2. 单条功能要求
3. 单条服务要求
4. 单个参数表行

### 4. 评分办法模块

这是高优先审查区。

重点承载以下规则：

1. 评分项未量化
2. 主观评分过多
3. 分值结构失衡
4. 重复加码

推荐最小解析单元：

1. 单个评分项
2. 单个评分表行
3. 单条价格分 / 商务分 / 技术分规则

### 5. 合同条款模块

这是高优先审查区。

重点承载以下内容：

1. 付款条款
2. 验收条款
3. 违约责任
4. 质保条款
5. 风险承担条款

推荐最小解析单元：

1. 单条付款约定
2. 单条验收约定
3. 单条违约责任
4. 单条费用承担约定

### 6. 程序与公开模块

重点包括：

1. 投标文件提交要求
2. 开标评标程序
3. 澄清修改
4. 质疑投诉路径

推荐最小解析单元：

1. 单条程序要求
2. 单条时点要求
3. 单条救济路径说明

### 7. 政策功能模块

重点包括：

1. 中小企业条款
2. 绿色采购
3. 节能环保
4. 进口产品
5. 地方政策适用

推荐最小解析单元：

1. 单条政策适用说明
2. 单条声明要求
3. 单条政策评分或价格扣除规则

## 六、建议的解析层级

当前建议采用“模块 -> 单元 -> 子片段”的三级解析结构。

### 第一级：业务模块

用于回答：

- 这部分内容属于招标文件中的什么区域

建议模块类型至少包括：

1. 项目基础信息
2. 资格条件
3. 采购需求
4. 评分办法
5. 合同条款
6. 程序条款
7. 政策条款
8. 其他

### 第二级：业务单元

这是后续审查的主工作单元。

建议业务单元类型至少包括：

1. 条款
2. 表格行
3. 评分项
4. 参数项
5. 合同项
6. 偏离项

这里应作为默认审查输入，而不是再默认退回自然段。

### 第三级：子片段

只有当业务单元过长、且内部明显包含多个独立判断点时，才进入子片段层。

例如：

1. 一个超长资格条款中同时包含 3 个独立限制条件
2. 一个技术参数行中串联了多个并列指标
3. 一个合同责任条款中同时规定了多项费用承担和责任分配

子片段层必须保留：

1. 原业务单元归属
2. 原始条款编号
3. 原始位置锚点

## 七、推荐的业务化解析对象

当前建议后续在解析结果中显式保留以下对象。

### 1. `module_block`

表示业务模块块。

建议字段：

1. `module_id`
2. `module_type`
3. `module_title`
4. `document_id`
5. `chapter_id`
6. `source_anchor`

### 2. `review_unit`

表示真正进入审查的业务单元。

建议字段：

1. `unit_id`
2. `unit_type`
3. `module_type`
4. `chapter_id`
5. `clause_id`
6. `table_id`
7. `row_index`
8. `title`
9. `text`
10. `location_label`
11. `parent_unit_id`

### 3. `unit_fragment`

表示业务单元内部因过长而继续拆出的子片段。

建议字段：

1. `fragment_id`
2. `unit_id`
3. `fragment_order`
4. `fragment_text`
5. `fragment_anchor`
6. `origin_location_label`

## 八、不同内容类型的解析建议

### 1. 正文条款

按条款编号优先切分。

如果条款标题存在，应优先保留标题。

### 2. 评分表

优先识别：

1. 评分项名称
2. 分值
3. 评分标准
4. 量化条件

最小审查单元应是单个评分项，而不是整张评分表。

### 3. 技术参数表

优先识别：

1. 参数名称
2. 指标要求
3. 单位
4. 备注

最小审查单元应是单个参数项或单行参数，而不是整段表格文本。

### 4. 偏离表

偏离表应被识别为独立数据源，不应和正文参数简单重复拼接。

建议：

1. 先保留偏离表行级对象
2. 审查时标记其与正文参数项的关联
3. 避免正文与偏离表被当成两个独立但重复的普通文本块

### 5. 合同条款

优先按责任条款、付款条款、验收条款、违约条款分开。

不要把整段合同连续正文直接作为一个大块进入审查。

## 九、反对的解析方式

当前明确不建议采用以下方式作为主策略：

1. 按页切分
2. 按固定字数机械切分
3. 把整章直接送审
4. 将整表简单拼成一段长文本
5. 先切成大量碎片，再靠结果层反向归并

这些方式的共同问题是：

1. 业务语义弱
2. 重复命中多
3. 审查解释性差
4. 证据定位和人工复核成本高

## 十、V1 实施建议

当前 V1 不要求一步做到“完整业务结构恢复”，但建议按以下顺序推进。

### 1. 第一优先

优先识别 4 个高价值模块：

1. 资格条件
2. 采购需求
3. 评分办法
4. 合同条款

### 2. 第二优先

在上述模块内优先形成 4 类审查单元：

1. 单条资格要求
2. 单个参数项
3. 单个评分项
4. 单条合同责任或付款 / 验收条款

### 3. 第三优先

只有在单元过长时，才继续补：

1. 单元内部子片段
2. 子片段锚点
3. 轻量去重

## 十一、与后续 `LLM V1` 收缩方案的关系

这份业务化解析方案，与 `LLM V1` 收缩方案是配套关系。

两者分工如下：

1. 业务化解析方案负责回答“应该把什么当作审查单元”
2. `LLM V1` 收缩方案负责回答“这些审查单元应该怎样轻量送给 `LLM`”

因此，后续技术实现不应再沿着“越切越碎”的方向推进，而应沿着：

1. 更像业务单元
2. 更少重复
3. 更容易判断
4. 更容易回溯

这条路径推进。

## 十二、当前正式结论

当前正式结论如下：

1. 招标文件解析应优先遵循政府采购招标文件的业务结构，而不是技术上的固定长度切分
2. 后续解析主线应从“切得更细”转为“按业务单元切得更准”
3. V1 的主审查单元应优先落在资格条件、需求参数、评分项、合同条款等自然业务单元上
4. 表格、评分表、参数表、偏离表不应只被当作普通文本，而应尽量恢复为行级或项级业务单元
5. 只有在业务单元过长时，才允许继续做轻量技术切分
