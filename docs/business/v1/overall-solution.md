# V1 总体方案（草案）

## 文档目的

这份文档用于沉淀当前 V1 阶段已经形成的总体方案，统一描述产品定位、业务边界、主流程、子流程、规则使用方式和数据支撑方式，作为后续继续细化的基础底稿。

## 1. 产品定位

当前产品定位为：

- 面向政府采购招标文件合规检查的审查智能体
- 面向公开招标场景的自动初审系统
- 以最终结论和审查报告为主查看层
- 以风险点、证据片段和规则依据为支撑层

这意味着 V1 的重点不是做一个复杂的规则运营平台，而是做一个能够默认运行、默认输出、默认可解释的审查智能体。

## 2. V1 边界

### 2.1 纳入范围

- 文件范围：招标文件正文
- 文件格式：PDF / Word
- 提交方式：一次提交 1 份文件
- 采购类型：货物 + 服务
- 采购方式：公开招标

### 2.2 暂不纳入范围

- 工程类采购
- 多文件联审
- 合同草案强制联审
- 澄清文件、修改文件、补遗文件联审
- 复杂地方特殊规则深度适配
- 在线审批流转

## 3. 运行前提

当前方案设计遵循以下运行前提：

- 智能体运行在独立隔离环境
- 底层主要依赖 LLM 算力
- 算力能力大约为单并发 30 tokens/秒
- 不假设存在大量运营人员进行规则配置

因此，V1 必须优先：

- 控制上下文规模
- 减少无效轮次
- 使用默认规则包
- 使用稳定输出结构

## 4. 审查依据

V1 的审查依据采用：

- 默认规则包为主
- 固定审查任务指令为辅

其中：

- 默认规则包负责定义优先审查范围、稳定输出框架和主要风险边界
- 固定审查任务指令负责统一审查目标、输出要求和智能体行为边界

## 5. 默认规则包定位

V1 的默认规则包不是全功能规则引擎，而是：

- 主审查框架
- 输出护栏
- 默认关注清单

当前默认规则包 1.0 已覆盖 12 条规则，优先面向：

- 公平竞争风险
- 需求合理性风险
- 评分规则风险
- 合同与结构完整性风险

## 6. 主流程

当前 V1 主流程定义为：

1. 文件进入系统
2. AI 发起审查
3. 生成审查结果
4. 人工查看结果

本版本暂不纳入“结果留存与流转”作为主流程节点。

## 7. 子流程设计

### 7.1 文件进入系统

目标：

- 审核人员提交一份待审核文件
- 系统形成可进入审查的输入对象

子流程：

1. 文件提交格式
2. 文件接收
3. 文件基础校验
4. 文件解析
5. 结构切分

当前已明确：

- 审核人员提交文件
- 支持 PDF / Word
- 当前一次只支持 1 份文件
- 如文件有问题，直接重新提交

### 7.2 AI 发起审查

目标：

- 系统自动开始对文件执行审查

子流程：

1. 自动触发
2. 输入装配
3. 审查依据加载
4. 审查执行
5. 中间产物生成

当前已明确：

- 文件进入系统后自动触发
- 用户提交原始文件，AI 使用解析后的文件内容
- 审查依据为默认规则包和固定审查任务指令
- 审查输出目标为风险点、证据片段、风险说明

### 7.3 生成审查结果

目标：

- 把中间审查产物汇总成审核人员可直接查看的结果

子流程：

1. 风险结果归并
2. 证据绑定
3. 结论汇总
4. 报告生成
5. 结果对象落地

当前已明确：

- 最终输出内容为最终结论和审查报告
- 由系统自动汇总生成
- 只要最终结论和审查报告生成成功即可视为完成

### 7.4 人工查看结果

目标：

- 审核人员查看系统输出并完成当前 V1 的人工消费动作

子流程：

1. 结果展示
2. 重点问题浏览
3. 按需下钻
4. 结果下载

当前已明确：

- 人工优先查看最终结论和审查报告
- 人工查看后的动作是下载最终结论和审查报告

## 8. 输出方式

V1 输出采用“报告驱动”模式。

主查看层：

- 最终结论
- 审查报告

支撑层：

- 风险点
- 证据片段
- 风险说明
- 规则依据

审核人员默认先看主查看层，仅在需要时才进一步下钻支撑层。

## 9. 数据支撑

当前 V1 数据模型已收敛出 8 个核心对象：

- 采购项目
- 采购文件
- 章节
- 条款
- 审查规则
- 风险点
- 证据片段
- 审查结论

主链路为：

`采购项目 -> 采购文件 -> 章节 -> 条款 -> 风险点 -> 证据片段 -> 审查结论`

规则对象作为贯穿判断依据存在：

`审查规则 -> 风险点`

## 10. 当前结论

当前 V1 总体方案已经形成以下统一认识：

1. 产品应定位为审查智能体，而不是规则运营平台
2. V1 必须优先收窄边界，先做单文件、公开招标、货物与服务场景
3. 审查依据以默认规则包为主、固定任务指令为辅
4. 主流程采用“文件进入系统 -> AI 发起审查 -> 生成审查结果 -> 人工查看结果”
5. 人工侧以查看和下载结果为主，不引入复杂流转

## 11. 后续继续细化方向

后续如继续往下细化，应遵循分层准则，从已确定主流程的某一个一级节点继续拆分，不跳级。

当前较适合优先继续细化的方向包括：

1. 文件解析与结构切分
2. 审查执行过程
3. 审查报告内部结构
4. 规则库字段模型和版本升级机制
