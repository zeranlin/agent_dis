# V1 默认规则包按业务模块重排（草案）

## 文档目的

这份文档用于把当前 V1 默认规则包中的 12 条规则，按产品讨论更常用的业务模块视角重排为：

- 资格
- 商务
- 技术
- 评分
- 合同

这样整理的目的，不是替代原有规则编号体系，而是帮助产品、总负责人和技术负责人在讨论“当前主要审什么、还缺什么、哪些规则容易混类”时，有一张更接近业务语义的对照表。

## 使用原则

- 规则编号、规则名称和执行级别仍以默认规则包为准
- 本文档只做业务讨论视角重排，不改现有规则边界
- 某些规则会跨两个业务模块，本文按当前更主要的讨论入口归类

## 2. 五类模块的当前讨论重点

### 2.1 资格类

资格类当前重点不是“抽更多字段”，而是先稳定识别以下风险：

- 不当限制供应商进入的准入条件
- 资质、人员、证书、年限要求明显偏高
- 业绩要求疑似定向

当前更适合优先讨论的规则组合是：

- `R1`
- `R2`
- `R3`
- `R4`

### 2.2 商务类

商务类在当前默认规则包中规则数量不多，主要承担“和资格、评分交界处的问题收口”。

当前重点不是单独扩一大组商务规则，而是先看：

- 商务要求中是否混入不当限制
- 商务能力是否被重复设门槛和重复加分

当前主要依赖：

- `R1`
- `R11`

### 2.3 技术类

技术类当前是最容易出现高价值风险的模块之一，重点关注：

- 明确品牌/型号指向
- 参数组合疑似定向
- 需求表述不明确
- 需求明显超配

当前主要规则组合是：

- `R5`
- `R6`
- `R7`
- `R8`

### 2.4 评分类

评分类当前重点是“评分标准是否可解释、是否可执行、是否对竞争结构产生明显影响”。

当前主要关注：

- 评分项是否量化
- 分值结构是否失衡
- 同一能力是否重复加码

当前主要规则组合是：

- `R9`
- `R10`
- `R11`

### 2.5 合同类

合同类当前还没有扩成完整合同审查体系，先收敛为：

- 关键条款是否缺失
- 条款之间是否前后冲突
- 履约责任、付款、验收、质保、费用承担等是否存在明显风险

当前主要规则是：

- `R12`

## 3. 当前产品讨论中的几个判断

### 3.1 当前规则最密集的模块是资格、技术、评分

这三类是当前默认规则包的主体，也最符合 V1 “先识别高价值典型风险”的策略。

### 3.2 商务类当前仍偏薄

商务类目前还没有独立成组的多条规则，更多是通过资格和评分交界问题来覆盖。

这意味着后续若继续扩规则，商务类是一个可能的补强方向，但当前不建议脱离真实样例单独大扩。

### 3.3 合同类当前先用一条总规则兜底

合同类现在主要依靠 `R12` 承担缺失和冲突检查，这更像第一版兜底策略，而不是完整合同规则库。

## 4. 当前结论

从产品讨论角度看，V1 默认规则包已经可以稳定映射到：

- 资格
- 商务
- 技术
- 评分
- 合同

这五类业务模块。

其中：

- 资格、技术、评分是当前默认规则包的主覆盖区
- 商务是当前相对薄的补充区
- 合同是当前以兜底规则为主的收口区

后续如继续扩规则，应优先基于真实样例命中质量和漏报情况推进，而不是脱离当前五类模块另起一套分类体系。
