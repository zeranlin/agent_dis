# V1 产品范围定义（草案）

## 文档目的

这份文档用于沉淀当前已经讨论达成一致的 V1 产品范围，作为后续继续细化规则、输出结构、角色分工和数据模型的统一底稿。

## V1 定位

V1 的目标不是覆盖全部政府采购审查场景，而是在范围清晰、规则相对稳定的场景下，先把自动审查、结果输出和人机协作流程跑通。

当前 V1 的产品定位是：

- 面向公开招标场景的自动初审系统
- 以最终结论和审查报告为主查看层
- 以风险卡片、证据片段和规则依据为下钻支撑层

## 1. V1 产品边界

### 1.1 纳入范围

- 文件范围：招标文件正文 + 合同草案
- 采购类型：货物 + 服务
- 采购方式：公开招标
- 输出目标：风险点 + 依据 + 证据定位 + 整改建议

### 1.2 暂不纳入范围

- 工程类采购
- 澄清文件、修改文件、补遗文件联审
- 复杂地方特殊规则的深度适配
- 全自动正式审查报告定稿

### 1.3 边界含义

这意味着 V1 优先解决的是“稳定发现典型风险”的问题，而不是一开始覆盖所有采购场景和所有规则差异。

## 2. 规则执行分层

V1 的规则需要明确分成三层，不同层的输出强度和业务含义不同。

### 2.1 自动判定

适用于文本特征明确、争议较小、命中条件稳定的规则。

典型场景包括：

- 明显地域限制
- 明显品牌或型号指向
- 明显缺少关键模块
- 明显结构冲突
- 部分规则明确的政策条款缺失

这一层可以直接给出较强提示，并附带证据和整改建议。

### 2.2 辅助提示

适用于存在较强可疑信号，但是否成立仍需业务判断的规则。

典型场景包括：

- 资格条件明显偏高
- 参数组合疑似定向
- 分值结构失衡
- 需求疑似超配
- 合同条款疑似失衡

这一层更适合输出“疑似风险”“建议关注”之类的结论。

### 2.3 人工复核

适用于依赖采购目标、地方口径、裁量判断或业务背景的规则。

典型场景包括：

- 采购方式适用是否恰当
- 条件设置是否与项目实质相关
- 技术需求是否与采购目的相称
- 地方特殊规则是否适用
- 某些评分办法是否实质影响公平竞争

这一层不应直接给出强裁定，应更多承担问题整理和依据组织的作用。

### 2.4 统一字段建议

每条规则建议统一带一个 `执行级别` 字段，取值为：

- 自动判定
- 辅助提示
- 人工复核

## 3. 输出结果结构

V1 的输出应分为“主查看层”和“下钻支撑层”。

### 3.1 主查看层

审核人员优先看到的内容包括：

- 最终结论
- 审查报告
- 风险总览
- 优先处理问题摘要

这一层的目标是帮助审核人员快速形成整体判断，而不是先进入细碎证据。

### 3.2 下钻支撑层

当审核人员需要确认某个问题时，再展开查看：

- 风险卡片列表
- 命中条款位置
- 证据片段
- 规则依据
- 整改建议明细

这意味着风险卡片仍然重要，但它属于支撑层，而不是主入口。

## 4. 角色定义

为了避免“人工”这个概念过于模糊，V1 至少区分以下四类角色。

### 4.1 文件提交人

职责：

- 上传或导入招标文件正文和合同草案
- 确认文件版本和基础信息
- 发起审查

该角色不负责最终合规定性。

### 4.2 审核人员

职责：

- 查看最终结论或审查报告
- 必要时下钻查看风险卡片、证据片段和规则依据
- 判断是否采纳系统结论
- 形成最终审核意见

这是当前业务中的核心使用角色。

### 4.3 运营人员

职责：

- 跟踪任务状态
- 协调任务流转
- 处理异常任务
- 确保任务和报告流转到正确的人

该角色关注流程管理，而不是法规判断本身。

### 4.4 规则/系统维护人员

职责：

- 维护规则库
- 处理误报和漏报反馈
- 更新规则和输出逻辑
- 优化系统准确率与可解释性

该角色关注系统能力演进，而不是单个项目审核。

## 5. 人机协作流程

基于上述边界、规则分层、输出结构和角色定义，V1 的推荐流程为：

`导入文件 -> AI 自动审查 -> 生成最终结论/报告 -> 人工查看结论 -> 按需下钻证据`

### 5.1 导入文件

由文件提交人上传：

- 招标文件正文
- 合同草案

系统在这一阶段完成：

- 文件解析
- 章节识别
- 条款切分
- 基础结构校验

### 5.2 AI 自动审查

系统根据规则执行分层完成：

- 自动判定类检测
- 辅助提示类检测
- 人工复核类问题整理

并形成以下中间结果：

- 风险总览
- 风险列表
- 证据定位
- 初步整改建议

### 5.3 生成最终结论/报告

系统把审查结果汇总成审核人员优先查看的主输出：

- 最终结论
- 审查报告
- 优先问题摘要

这一阶段的关键点是：审核人员的第一入口不是所有中间过程，而是最终输出。

### 5.4 人工查看结论

审核人员主要动作包括：

- 查看最终结论
- 阅读审查报告
- 判断是否采纳系统意见

默认情况下，审核人员不需要逐条查看所有风险点。

### 5.5 按需下钻证据

只有在审核人员需要确认某个问题时，才继续展开：

- 风险卡片
- 条款原文
- 证据片段
- 规则依据
- 整改建议细项

因此，V1 更适合定义为“报告驱动”的使用模式，而不是“逐条复核驱动”的使用模式。

## 6. 当前结论

当前已经形成的统一认识是：

- V1 应先做范围收敛，而不是功能泛化
- 规则必须分层输出，不能所有问题都用同一种强度表达
- 审核人员优先看最终结论和报告，而不是先逐条看风险卡片
- 风险卡片、证据和规则依据属于必要的支撑层
- 不同人工角色必须拆开定义，不能混成一个“人工”概念

## 7. 后续继续细化的方向

这份草案之后，最值得继续展开的方向包括：

1. V1 审查报告的结构定义
2. 风险卡片字段定义
3. 规则库字段模型
4. 对象与关系的字段级数据模型
