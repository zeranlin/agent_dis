# V1 文件解析与结构切分最小可用化

## 文档目的

这份文档用于把 V1 的下一主题正式收口为一个新的一级讨论主题，作为结果页线程收口后的下一轮主线。

## 1. 主题结论

下一主题确定为：

- 文件解析与结构切分最小可用化

选择这个主题的原因是：

1. 结果页线程已经收口，当前最大缺口不再是展示层
2. 当前解析能力仍是最小文本解码占位实现，还不足以支撑稳定审查
3. 如果输入侧不稳定，后续真实 LLM 审查能力就无法稳定落地

## 2. 当前目标

这一主题的目标不是一步做到高精度解析，而是先把 V1 输入侧推进到“可用于最小稳定审查”的程度。

V1 当前只追求：

- 能稳定接住 `PDF` / `Word`
- 能抽出可供审查的正文文本
- 能做最小结构切分
- 能把切分结果稳定提供给后续审查阶段

## 3. 一级拆分方案

当前一级拆分建议固定为以下 3 项：

### 1.1 文件接入边界

讨论重点：

- V1 到底接哪些 `PDF` / `Word`
- 哪些文件先视为可处理，哪些先不处理
- 哪些失败场景直接判定为解析失败

本节目标：

- 收口 V1 可接收文件边界
- 收口失败判定口径

本节已确认结论：

#### 1.1.1 接入范围

- V1 只接收 1 份文件
- 文件格式只支持 `PDF`、`DOCX`、`DOC`
- 文件内容目标限定为招标文件正文
- 文件来源默认为审核人员手工上传
- 不要求上传项目信息、合同草案、附件包、压缩包

#### 1.1.2 先视为可处理的文件

- 文本型 `PDF`
- 常规导出的 `DOCX`
- 结构不太复杂、正文连续的 `DOC`
- 文件能被系统成功读取字节流
- 文件能抽出连续正文文本

这里的判断重点不是“看起来像招标文件”，而是“系统能稳定抽出可审查正文”。

#### 1.1.3 先判定为不可处理的文件

- 加密或受权限控制的 `PDF`
- 扫描件图片型 `PDF`
- 只有图片、几乎没有可提取文本的文件
- 损坏文件
- 空文件
- 扩展名伪装文件
- 压缩包、图片、`Excel`、压缩后的材料包
- 正文极少、基本无法形成审查输入的文件

#### 1.1.4 V1 明确不处理的范围

- 不做 `OCR`
- 不做多文件合并解析
- 不做附件与正文联审
- 不做版式高度还原
- 不做复杂表格精细结构恢复
- 不做地方模板差异适配

#### 1.1.5 最小失败口径

建议统一为两层：

1. 上传阶段失败
2. 解析阶段失败

上传阶段失败示例：

- 文件格式不支持
- 文件损坏
- 文件为空
- 无法保存

解析阶段失败示例：

- 文件已接收，但无法提取出可审查正文

面向审核人员的最小提示语建议统一为：

- 文件格式不支持
- 文件无法读取
- 文件无可审查正文
- 当前文件暂不支持解析，请重新提交更清晰的正文文件

### 1.2 正文提取

讨论重点：

- 从文件里到底提什么内容
- 标题、目录、页眉页脚、附件说明怎么处理
- 什么叫“足够可审查”的正文文本

本节目标：

- 收口 V1 正文提取口径
- 收口最小清洗规则

本节已确认结论：

#### 1.2.1 提取目标

- 目标不是还原原文件版式
- 目标是拿到“可供审查的连续正文文本”
- 输出优先服务后续规则匹配、风险识别、证据定位
- 正文提取以“保留审查意义”优先，不以“页面长得完全一样”优先

#### 1.2.2 V1 需要提取的内容

- 文件主标题
- 章节标题
- 条款正文
- 列表项正文
- 与资格条件、评分办法、采购需求、合同条款、履约验收、政策功能有关的正文段落
- 对审查有直接意义的表格文本内容

这里的表格内容在 V1 中先按“表格转文本”理解，不做复杂单元格结构恢复。

#### 1.2.3 V1 先不重点提取的内容

- 页眉
- 页脚
- 连续页码
- 水印提示
- 装饰性封面文字
- 纯导航型目录
- 重复出现的单位名称、打印时间、系统页脚
- 附件区里的非正文材料

#### 1.2.4 特殊内容处理建议

- 标题：保留
- 目录：默认不作为正文
- 页眉页脚：默认去掉
- 页码：默认去掉
- 表格：先转成顺序文本
- 附件说明：如只是“详见附件”，先保留该提示语，但附件内容本身不进入 V1 正文提取主范围

如果目录、页眉页脚与正文强混排，且无法稳定剔除，则允许保留原文，以避免误删真实正文。

#### 1.2.5 可审查正文判断标准

建议统一为以下最小标准：

- 文本连续可读
- 不是大段乱码
- 不是几乎全是空白或重复页眉页脚
- 能识别出基本的标题和正文关系
- 能支撑后续切分出审查块
- 至少能让系统判断出采购需求、资格条件、评分办法、合同条款等区域中的一部分

V1 不要求“提得特别干净”，但至少要“能审”。

#### 1.2.6 最小清洗规则

建议只做轻清洗，不做重清洗：

- 去掉明显空白行
- 合并连续多空格
- 去掉重复页眉页脚
- 去掉孤立页码
- 保留标题与段落换行
- 保留列表编号
- 保留条款编号
- 表格内容转成可读文本行
- 不做激进改写
- 不做语义重写
- 不做自动总结式清洗

### 1.3 结构切分

讨论重点：

- 提取出来的正文如何切成后续可审查的结构块
- 是按页、按段、按章节还是混合切分
- 怎样兼顾 LLM 上下文成本和审查可追溯性

本节目标：

- 收口 V1 结构切分单位
- 收口切分结果最小输出对象

## 4. 当前顺序约束

后续讨论顺序固定为：

1. 先讨论 `1.1`
2. 再讨论 `1.2`
3. 再讨论 `1.3`

只有当前一级节点收口后，才进入它的二级节点讨论。

## 5. 当前结论

结果页线程已经正式收口。

V1 下一主题正式切换为“文件解析与结构切分最小可用化”，后续按 `1.1 -> 1.2 -> 1.3` 的顺序逐层推进。

其中：

- `1.1 文件接入边界` 已完成收口
- `1.2 正文提取` 已完成收口
