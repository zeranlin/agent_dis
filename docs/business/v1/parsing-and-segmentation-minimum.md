# V1 文件解析与结构切分最小可用化

## 文档目的

这份文档用于把 V1 的下一主题正式收口为一个新的一级讨论主题，作为结果页线程收口后的下一轮主线。

## 1. 主题结论

下一主题确定为：

- 文件解析与结构切分最小可用化

选择这个主题的原因是：

1. 结果页线程已经收口，当前最大缺口不再是展示层
2. 当前解析能力仍是最小文本解码占位实现，还不足以支撑稳定审查
3. 如果输入侧不稳定，后续真实 LLM 审查能力就无法稳定落地

## 2. 当前目标

这一主题的目标不是一步做到高精度解析，而是先把 V1 输入侧推进到“可用于最小稳定审查”的程度。

V1 当前只追求：

- 能稳定接住 `PDF` / `Word`
- 能抽出可供审查的正文文本
- 能做最小结构切分
- 能把切分结果稳定提供给后续审查阶段

## 3. 一级拆分方案

当前一级拆分建议固定为以下 3 项：

### 1.1 文件接入边界

讨论重点：

- V1 到底接哪些 `PDF` / `Word`
- 哪些文件先视为可处理，哪些先不处理
- 哪些失败场景直接判定为解析失败

本节目标：

- 收口 V1 可接收文件边界
- 收口失败判定口径

### 1.2 正文提取

讨论重点：

- 从文件里到底提什么内容
- 标题、目录、页眉页脚、附件说明怎么处理
- 什么叫“足够可审查”的正文文本

本节目标：

- 收口 V1 正文提取口径
- 收口最小清洗规则

### 1.3 结构切分

讨论重点：

- 提取出来的正文如何切成后续可审查的结构块
- 是按页、按段、按章节还是混合切分
- 怎样兼顾 LLM 上下文成本和审查可追溯性

本节目标：

- 收口 V1 结构切分单位
- 收口切分结果最小输出对象

## 4. 当前顺序约束

后续讨论顺序固定为：

1. 先讨论 `1.1`
2. 再讨论 `1.2`
3. 再讨论 `1.3`

只有当前一级节点收口后，才进入它的二级节点讨论。

## 5. 当前结论

结果页线程已经正式收口。

V1 下一主题正式切换为“文件解析与结构切分最小可用化”，后续按 `1.1 -> 1.2 -> 1.3` 的顺序逐层推进。
