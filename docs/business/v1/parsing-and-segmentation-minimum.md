# V1 文件解析与结构切分最小可用化

## 文档目的

这份文档用于把 V1 的下一主题正式收口为一个新的一级讨论主题，作为结果页线程收口后的下一轮主线。

## 1. 主题结论

下一主题确定为：

- 文件解析与结构切分最小可用化

选择这个主题的原因是：

1. 结果页线程已经收口，当前最大缺口不再是展示层
2. 当前解析能力仍是最小文本解码占位实现，还不足以支撑稳定审查
3. 如果输入侧不稳定，后续真实 LLM 审查能力就无法稳定落地

## 2. 当前目标

这一主题的目标不是一步做到高精度解析，而是先把 V1 输入侧推进到“可用于最小稳定审查”的程度。

V1 当前只追求：

- 能稳定接住 `PDF` / `Word`
- 能抽出可供审查的正文文本
- 能做最小结构切分
- 能把切分结果稳定提供给后续审查阶段

## 3. 一级拆分方案

当前一级拆分建议固定为以下 3 项：

### 1.1 文件接入边界

讨论重点：

- V1 到底接哪些 `PDF` / `Word`
- 哪些文件先视为可处理，哪些先不处理
- 哪些失败场景直接判定为解析失败

本节目标：

- 收口 V1 可接收文件边界
- 收口失败判定口径

本节已确认结论：

#### 1.1.1 接入范围

- V1 只接收 1 份文件
- 文件格式只支持 `PDF`、`DOCX`、`DOC`
- 文件内容目标限定为招标文件正文
- 文件来源默认为审核人员手工上传
- 不要求上传项目信息、合同草案、附件包、压缩包

#### 1.1.2 先视为可处理的文件

- 文本型 `PDF`
- 常规导出的 `DOCX`
- 结构不太复杂、正文连续的 `DOC`
- 文件能被系统成功读取字节流
- 文件能抽出连续正文文本

这里的判断重点不是“看起来像招标文件”，而是“系统能稳定抽出可审查正文”。

#### 1.1.3 先判定为不可处理的文件

- 加密或受权限控制的 `PDF`
- 扫描件图片型 `PDF`
- 只有图片、几乎没有可提取文本的文件
- 损坏文件
- 空文件
- 扩展名伪装文件
- 压缩包、图片、`Excel`、压缩后的材料包
- 正文极少、基本无法形成审查输入的文件

#### 1.1.4 V1 明确不处理的范围

- 不做 `OCR`
- 不做多文件合并解析
- 不做附件与正文联审
- 不做版式高度还原
- 不做复杂表格精细结构恢复
- 不做地方模板差异适配

#### 1.1.5 最小失败口径

建议统一为两层：

1. 上传阶段失败
2. 解析阶段失败

上传阶段失败示例：

- 文件格式不支持
- 文件损坏
- 文件为空
- 无法保存

解析阶段失败示例：

- 文件已接收，但无法提取出可审查正文

面向审核人员的最小提示语建议统一为：

- 文件格式不支持
- 文件无法读取
- 文件无可审查正文
- 当前文件暂不支持解析，请重新提交更清晰的正文文件

### 1.2 正文提取

讨论重点：

- 从文件里到底提什么内容
- 标题、目录、页眉页脚、附件说明怎么处理
- 什么叫“足够可审查”的正文文本

本节目标：

- 收口 V1 正文提取口径
- 收口最小清洗规则

### 1.3 结构切分

讨论重点：

- 提取出来的正文如何切成后续可审查的结构块
- 是按页、按段、按章节还是混合切分
- 怎样兼顾 LLM 上下文成本和审查可追溯性

本节目标：

- 收口 V1 结构切分单位
- 收口切分结果最小输出对象

## 4. 当前顺序约束

后续讨论顺序固定为：

1. 先讨论 `1.1`
2. 再讨论 `1.2`
3. 再讨论 `1.3`

只有当前一级节点收口后，才进入它的二级节点讨论。

## 5. 当前结论

结果页线程已经正式收口。

V1 下一主题正式切换为“文件解析与结构切分最小可用化”，后续按 `1.1 -> 1.2 -> 1.3` 的顺序逐层推进。

其中：

- `1.1 文件接入边界` 已完成收口
- 下一步进入 `1.2 正文提取`
