# V1 文件解析与结构切分最小可用化

## 文档目的

这份文档用于把 V1 的下一主题正式收口为一个新的一级讨论主题，作为结果页线程收口后的下一轮主线。

## 1. 主题结论

下一主题确定为：

- 文件解析与结构切分最小可用化

选择这个主题的原因是：

1. 结果页线程已经收口，当前最大缺口不再是展示层
2. 当前解析能力仍是最小文本解码占位实现，还不足以支撑稳定审查
3. 如果输入侧不稳定，后续真实 LLM 审查能力就无法稳定落地

## 2. 当前目标

这一主题的目标不是一步做到高精度解析，而是先把 V1 输入侧推进到“可用于最小稳定审查”的程度。

V1 当前只追求：

- 能稳定接住 `PDF` / `Word`
- 能抽出可供审查的正文文本
- 能做最小结构切分
- 能把切分结果稳定提供给后续审查阶段

## 3. 一级拆分方案

当前一级拆分建议固定为以下 3 项：

### 1.1 文件接入边界

讨论重点：

- V1 到底接哪些 `PDF` / `Word`
- 哪些文件先视为可处理，哪些先不处理
- 哪些失败场景直接判定为解析失败

本节目标：

- 收口 V1 可接收文件边界
- 收口失败判定口径

本节已确认结论：

#### 1.1.1 接入范围

- V1 只接收 1 份文件
- 文件格式只支持 `PDF`、`DOCX`、`DOC`
- 文件内容目标限定为招标文件正文
- 文件来源默认为审核人员手工上传
- 不要求上传项目信息、合同草案、附件包、压缩包

#### 1.1.2 先视为可处理的文件

- 文本型 `PDF`
- 常规导出的 `DOCX`
- 结构不太复杂、正文连续的 `DOC`
- 文件能被系统成功读取字节流
- 文件能抽出连续正文文本

这里的判断重点不是“看起来像招标文件”，而是“系统能稳定抽出可审查正文”。

#### 1.1.3 先判定为不可处理的文件

- 加密或受权限控制的 `PDF`
- 扫描件图片型 `PDF`
- 只有图片、几乎没有可提取文本的文件
- 损坏文件
- 空文件
- 扩展名伪装文件
- 压缩包、图片、`Excel`、压缩后的材料包
- 正文极少、基本无法形成审查输入的文件

#### 1.1.4 V1 明确不处理的范围

- 不做 `OCR`
- 不做多文件合并解析
- 不做附件与正文联审
- 不做版式高度还原
- 不做复杂表格精细结构恢复
- 不做地方模板差异适配

#### 1.1.5 最小失败口径

建议统一为两层：

1. 上传阶段失败
2. 解析阶段失败

上传阶段失败示例：

- 文件格式不支持
- 文件损坏
- 文件为空
- 无法保存

解析阶段失败示例：

- 文件已接收，但无法提取出可审查正文

面向审核人员的最小提示语建议统一为：

- 文件格式不支持
- 文件无法读取
- 文件无可审查正文
- 当前文件暂不支持解析，请重新提交更清晰的正文文件

### 1.2 正文提取

讨论重点：

- 从文件里到底提什么内容
- 标题、目录、页眉页脚、附件说明怎么处理
- 什么叫“足够可审查”的正文文本

本节目标：

- 收口 V1 正文提取口径
- 收口最小清洗规则

本节已确认结论：

#### 1.2.1 提取目标

- 目标不是还原原文件版式
- 目标是拿到“可供审查的连续正文文本”
- 输出优先服务后续规则匹配、风险识别、证据定位
- 正文提取以“保留审查意义”优先，不以“页面长得完全一样”优先

#### 1.2.2 V1 需要提取的内容

- 文件主标题
- 章节标题
- 条款正文
- 列表项正文
- 与资格条件、评分办法、采购需求、合同条款、履约验收、政策功能有关的正文段落
- 对审查有直接意义的表格文本内容

这里的表格内容在 V1 中先按“表格转文本”理解，不做复杂单元格结构恢复。

#### 1.2.3 V1 先不重点提取的内容

- 页眉
- 页脚
- 连续页码
- 水印提示
- 装饰性封面文字
- 纯导航型目录
- 重复出现的单位名称、打印时间、系统页脚
- 附件区里的非正文材料

#### 1.2.4 特殊内容处理建议

- 标题：保留
- 目录：默认不作为正文
- 页眉页脚：默认去掉
- 页码：默认去掉
- 表格：先转成顺序文本
- 附件说明：如只是“详见附件”，先保留该提示语，但附件内容本身不进入 V1 正文提取主范围

如果目录、页眉页脚与正文强混排，且无法稳定剔除，则允许保留原文，以避免误删真实正文。

#### 1.2.5 可审查正文判断标准

建议统一为以下最小标准：

- 文本连续可读
- 不是大段乱码
- 不是几乎全是空白或重复页眉页脚
- 能识别出基本的标题和正文关系
- 能支撑后续切分出审查块
- 至少能让系统判断出采购需求、资格条件、评分办法、合同条款等区域中的一部分

V1 不要求“提得特别干净”，但至少要“能审”。

#### 1.2.6 最小清洗规则

建议只做轻清洗，不做重清洗：

- 去掉明显空白行
- 合并连续多空格
- 去掉重复页眉页脚
- 去掉孤立页码
- 保留标题与段落换行
- 保留列表编号
- 保留条款编号
- 表格内容转成可读文本行
- 不做激进改写
- 不做语义重写
- 不做自动总结式清洗

### 1.3 结构切分

讨论重点：

- 提取出来的正文如何切成后续可审查的结构块
- 是按页、按段、按章节还是混合切分
- 怎样兼顾 LLM 上下文成本和审查可追溯性

本节目标：

- 收口 V1 结构切分单位
- 收口切分结果最小输出对象

本节已确认结论：

#### 1.3.1 切分原则

- 不按整页切
- 不只按自然段切
- 采用“章节/条款优先，段落补充”的混合切分
- 目标是让每个切分块既能被审查，又能被追溯

#### 1.3.2 推荐切分单位

优先级建议为：

1. 章节块
2. 条款块
3. 段落块

具体执行时：

- 能识别章节和条款时，优先按章节或条款切
- 章节过大时，再向下拆成多个段落块
- 识别不稳定时，退化成连续段落块

V1 不追求“所有文件都完美结构化”，而是追求“有结构就用结构，没结构就退化可用”。

#### 1.3.3 每个切分块的最小字段

建议至少保留：

- `block_id`
- `block_type`
- `title`
- `text`
- `source_page_start`
- `source_page_end`
- `order_index`
- `parent_block_id`
- `source_anchor`

其中：

- `block_type` 用于表达 `section`、`clause`、`paragraph`
- `source_anchor` 用于保留原标题、条款号、编号等定位线索

#### 1.3.4 块大小控制原则

- 文本足够表达一个相对完整的审查语义单元
- 不能大到把多个无关规则主题混在一起
- 也不能碎到一句话一个块

一个块最好只承载一个相对稳定的审查主题，例如：

- 资格条件一组
- 评分标准一组
- 合同条款一组

#### 1.3.5 大段内容处理建议

- 超长章节要拆
- 长表格转出的长文本要拆
- 连续条款很多时可按条款分块
- 如果同一章节同时包含资格、评分、合同等不同主题，也要拆开

#### 1.3.6 对切分结果的最小要求

V1 切分结果至少要做到：

- 后续 `LLM` 能直接消费
- 同一个块能回指原文
- 块和块之间保留顺序
- 能支持后面把风险点挂回具体块
- 能支持报告里引用来源位置

## 6. 下一层讨论进展

### 1.2.1 可审查正文判定标准

本节已确认结论：

#### 1.2.1.1 判定目标

- 不是判断文件是否完整
- 而是判断提取出来的正文，是否足以支撑最小审查
- 重点看“能不能审”，不是看“漂不漂亮”

#### 1.2.1.2 判定级别

V1 建议统一采用三级判定：

- `可审查`
- `可降级审查`
- `不可审查`

#### 1.2.1.3 可审查标准

满足大部分以下条件即可：

- 文本连续可读
- 没有大段乱码
- 正文占比明显高于噪声
- 能识别出部分标题或条款层次
- 能提取出与采购需求、资格条件、评分办法、合同条款等有关的正文片段
- 切分后能形成稳定审查块
- 原文定位信息基本可保留

这类文件进入正常审查流程。

#### 1.2.1.4 可降级审查标准

存在缺陷，但仍可勉强审查：

- 文本能读，但结构较乱
- 标题识别不稳定
- 页眉页脚、目录、杂质较多
- 表格转文本效果一般
- 部分页面提取异常，但整体仍有可用正文
- 切分效果不理想，但还能形成部分审查块

这类文件可以继续审查，但需要标记输入质量较弱。

#### 1.2.1.5 不可审查标准

满足任一类严重问题即可判定：

- 大部分内容无法提取文本
- 大段乱码或无意义字符
- 基本全是图片、空白、页眉页脚、页码、目录噪声
- 无法形成连续正文
- 无法切出最小审查块
- 提取结果不足以支撑任何主要审查主题
- 原文定位严重缺失，无法追溯

这类文件不进入正式审查，直接进入解析失败或不可审查状态。

#### 1.2.1.6 判定方式

建议采用“强规则 + 少量阈值”的组合，不做复杂打分模型。

强规则直接失败：

- 空文本
- 近乎全乱码
- 无法形成正文
- 无法切分

弱规则用于降级：

- 标题识别弱
- 噪声占比高
- 表格文本质量一般
- 部分页提取失败

也就是：先拦明显失败，再区分“正常审查”还是“降级审查”。

#### 1.2.1.7 对后续流程的影响

- `可审查`：正常进入后续审查
- `可降级审查`：进入后续审查，但打上输入质量标记
- `不可审查`：不进入后续审查，直接返回失败说明

### 1.3.1 切分块数据结构

本节已确认结论：

#### 1.3.1.1 设计目标

- 不做很重的文档模型
- 而是做一个能直接服务后续审查的最小块对象
- 这个对象要同时满足：
  - `LLM` 可消费
  - 风险点可挂接
  - 原文可追溯
  - 报告可引用

#### 1.3.1.2 对象名称

V1 建议统一命名为：

- `document_block`

#### 1.3.1.3 必须字段

建议必须字段固定为：

- `block_id`
- `task_id`
- `block_type`
- `title`
- `text`
- `order_index`
- `source_page_start`
- `source_page_end`
- `source_anchor`
- `parent_block_id`

字段说明：

- `block_id`：块唯一标识
- `task_id`：所属任务
- `block_type`：`section` / `clause` / `paragraph`
- `title`：块标题，没有则可为空
- `text`：块正文
- `order_index`：在全文中的顺序
- `source_page_start` / `source_page_end`：页码范围
- `source_anchor`：原标题、条款号、编号等定位线索
- `parent_block_id`：父块，没有则为空

#### 1.3.1.4 建议字段

建议再保留以下可选字段：

- `block_level`
- `path_titles`
- `path_ids`
- `extracted_from`
- `quality_flag`
- `token_estimate`

字段说明：

- `block_level`：层级深度
- `path_titles`：从根到当前块的标题路径
- `path_ids`：从根到当前块的块 id 路径
- `extracted_from`：来自章节、条款还是段落退化切分
- `quality_flag`：如 `normal` / `degraded`
- `token_estimate`：供后续 `LLM` 编排做预算

#### 1.3.1.5 关系表达

V1 只保留 3 类基础关系：

- 父子关系：`parent_block_id`
- 顺序关系：`order_index`
- 路径关系：`path_titles` / `path_ids`

这已经足够支撑后续：

- 块内审查
- 上下文回溯
- 报告引用
- 风险归类

#### 1.3.1.6 页码与定位表达

V1 不做字符级 `offset`，先统一采用：

- 页码范围
- 标题锚点
- 条款号或编号锚点

目标是先保证“人能找回去”，不追求字符级精确高亮。

#### 1.3.1.7 与后续对象的连接方式

后续至少应允许以下对象直接挂到 `document_block`：

- 风险点
- 证据片段
- 审查规则命中
- 报告引用来源

因此，后续对象统一优先通过：

- `block_id`

完成挂接，而不要反向依赖原始页码拼接。

### 1.1.1 文件失败分类细化

本节已确认结论：

#### 1.1.1.1 分类目标

- 不把所有异常都分得很细
- 先建立一套 V1 可运营、可提示、可追踪的最小失败分类
- 这套分类同时服务：
  - 审核人员提示
  - 系统状态流转
  - 技术排查
  - 后续统计分析

#### 1.1.1.2 一级分类

V1 失败分类固定为 4 类：

- `上传校验失败`
- `文件读取失败`
- `正文提取失败`
- `正文不可审查`

#### 1.1.1.3 各类含义

`上传校验失败`

- 文件类型不支持
- 空文件
- 扩展名伪装
- 上传请求本身不合法

`文件读取失败`

- 文件损坏
- 文件无法打开
- 加密或权限限制导致无法读取
- 物理字节可接收，但解析器无法正常读取

`正文提取失败`

- 文件可读，但几乎提不出正文
- 只有图片或杂乱字符
- 大量内容无法转成连续文本

`正文不可审查`

- 已提取出部分文本
- 但不足以支撑最小审查
- 无法形成正文块或无法追溯
- 这类更接近“输入质量失败”，而不是纯技术异常

#### 1.1.1.4 阶段归属

上传阶段失败：

- `上传校验失败`

解析阶段失败：

- `文件读取失败`
- `正文提取失败`
- `正文不可审查`

#### 1.1.1.5 是否允许降级

建议如下：

- `上传校验失败`：直接终止
- `文件读取失败`：直接终止
- `正文提取失败`：一般终止
- `正文不可审查`：终止，不进入正式审查

只有“可降级审查”保留在成功主链路中，不进入失败分类。

#### 1.1.1.6 用户口径与系统口径分层

面向审核人员的提示建议为：

- 文件格式不支持
- 文件无法读取
- 文件正文提取失败
- 文件正文不足以支持审查

面向系统内部的错误码建议为：

- `UPLOAD_VALIDATION_FAILED`
- `FILE_READ_FAILED`
- `TEXT_EXTRACTION_FAILED`
- `TEXT_NOT_REVIEWABLE`
