# 给技术负责人的开发启动说明

## 文档目的

这份文档用于帮助负责开发 V1 产品的技术负责人快速进入状态，明确：

- 这次要做的是什么
- 不做什么
- 先做哪些模块
- 交付到什么程度算第一版完成

## 1. 先说结论

如果你已经阅读了 `docs/business/v1/` 目录下现有文档，那么当前信息已经足够你启动：

- 技术方案设计
- 核心数据结构设计
- 接口草案设计
- 原型或最小可用版本开发

但如果要直接进入多人协作开发，仍建议先把以下内容在技术侧补成实现物：

1. 固定审查任务指令
2. 默认规则包配置文件
3. 审查执行链路设计
4. 页面接口和结果对象定义
5. V1 验收清单

本次文档就是把这些开发启动要点集中说清楚。

## 2. 你现在接手的产品是什么

当前产品不是一个规则运营平台，而是一个审查智能体。

它的目标不是让运营人员长期手配规则，而是：

- 审核人员上传 1 份招标文件
- 系统自动发起审查
- 系统自动生成最终结论和审查报告
- 审核人员在线查看并下载结果

V1 交互非常简单：

1. 上传页
2. 审核中状态
3. 结果页

## 3. 当前已经确定的业务边界

V1 当前只覆盖以下范围：

- 文件格式：PDF / Word
- 文件数量：一次 1 份
- 文件范围：招标文件正文
- 采购方式：公开招标
- 采购类型：货物 + 服务

V1 当前不做：

- 工程类采购
- 多文件联审
- 强依赖人工运营配规则
- 复杂地方规则深适配
- 在线审批流

这意味着技术实现要优先围绕“极简闭环”来做，而不是提前做大而全平台。

## 4. 当前已经确定的主流程

V1 主流程已经收敛为：

1. 文件进入系统
2. AI 发起审查
3. 生成审查结果
4. 人工查看结果

对应的最小系统能力应理解为：

- 接收文件
- 解析文件
- 装配审查输入
- 调用审查智能体
- 生成结果对象
- 展示与下载结果

## 6. 推荐开发顺序

建议按以下顺序启动开发。

### 6.1 第一阶段：跑通主链路

目标：

- 把上传 -> 审查 -> 结果生成 -> 查看 下载 先跑通

建议先实现：

1. 单文件上传
2. 文件解析
3. 审查任务创建
4. 审查结果对象生成
5. 结果页展示

### 6.2 第二阶段：接入审查能力

目标：

- 让审查智能体能够在默认规则包约束下稳定输出中间结果

建议先实现：

1. 固定审查任务指令管理
2. 默认规则包加载
3. 章节或条款切分
4. 风险点输出
5. 证据片段输出
6. 风险说明输出

### 6.3 第三阶段：收口结果层

目标：

- 把中间结果稳定汇总成审核人员能直接消费的结果

建议先实现：

1. 最终结论生成
2. 审查报告 Markdown 生成
3. 下载文件生成
4. 状态流转

## 7. 推荐最小模块划分

技术上建议先拆成以下最小模块：

1. 文件接入模块
2. 文件解析模块
3. 审查编排模块
4. 规则加载模块
5. 智能体执行模块
6. 结果汇总模块
7. 页面展示模块

如果团队规模小，也可以先收敛成三块：

1. 输入层
2. 审查层
3. 输出层

## 8. 审查智能体实现原则

当前运行环境是独立隔离环境，底层主要依赖 LLM，单并发能力大约为 30 tokens/秒。

因此实现时要特别注意：

- 控制上下文大小
- 避免整份长文一次性塞入
- 优先按章节或条款切分
- 减少多轮往返
- 输出结构尽量固定

技术上不建议一开始做：

- 很重的 Agent 编排框架
- 很复杂的自治流程
- 很多动态工具调用

V1 更适合：

- 稳定输入模板
- 稳定规则包
- 稳定任务指令
- 稳定输出格式

## 9. 你需要先产出的技术材料

如果要正式拉开发，建议你先补齐以下技术侧材料：

1. 系统模块图
2. 任务状态流转图
3. 接口定义草案
4. 数据表或对象结构草案
5. 异常处理策略
6. 最小测试方案

其中最优先的是：

- 接口定义
- 数据结构
- 状态流转

## 10. 开发完成的第一版判断标准

只要满足以下条件，就可以认为第一版已经可交付：

1. 能上传 1 份 PDF 或 Word
2. 上传后能自动触发审查
3. 系统能生成最终结论和审查报告
4. 结果页能在线查看
5. 能下载 Markdown 结果文件

不要求第一版就解决：

- 地方规则全覆盖
- 高精度复杂推理
- 多角色工作台
- 全流程审批流

## 11. 当前建议

如果你是这个产品的技术负责人，建议你现在按下面顺序启动：

1. 先基于 `docs/business/v1/` 完成技术方案草图
2. 先画模块和状态，不急着上复杂页面
3. 先把固定任务指令和默认规则包做成可加载资产
4. 先实现最小主链路，再逐步补充准确率和可解释性

## 当前结论

`docs/business/v1/` 目录下的业务方案，已经足够支持技术负责人进入开发启动阶段。

这套资料更适合的使用方式不是“继续空谈”，而是立即转成：

- 技术方案
- 数据与接口设计
- 最小可用产品开发
