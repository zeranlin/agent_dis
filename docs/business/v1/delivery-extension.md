# V1 开发交付补充方案（草案）

## 文档目的

这份文档用于沉淀 V1 阶段为产品开发交付补充的三类关键设计：

1. 审查报告模板
2. 审查执行流程
3. 规则库配置格式

它们共同用于支撑技术负责人从“业务方案理解”走向“可执行的技术方案和原型设计”。

## 1. 审查报告模板

### 1.1 报告头信息

目标：

- 说明这份报告对应哪份文件
- 保证报告和待审文件可以一一对应

建议内容：

- 报告编号
- 文件名称
- 文件格式
- 审查时间
- 审查版本

更细一层可拆为：

- 文件标识信息
- 报告标识信息

### 1.2 总体结论

目标：

- 让审核人员第一眼看到整体判断

建议内容：

- 总体判断
- 结论摘要

更细一层可拆为：

- 总体判断
- 一到三句话的结论摘要

### 1.3 风险总览

目标：

- 快速展示问题分布

建议内容：

- 高风险数量
- 中风险数量
- 低风险数量
- 自动判定数量
- 辅助提示数量
- 一级规则域分布

更细一层可拆为：

- 风险等级统计
- 执行级别统计
- 规则域分布

### 1.4 重点问题摘要

目标：

- 把最需要先关注的问题提到前面

建议内容：

- 重点问题列表
- 每条问题的简短摘要

每条摘要建议至少包含：

- 风险标题
- 风险级别
- 命中位置
- 一句话说明

更细一层可拆为：

- 重点问题筛选规则
- 重点问题摘要字段

### 1.5 审查明细

目标：

- 提供完整可追溯的问题清单

建议内容：

- 风险标题
- 风险级别
- 执行级别
- 所属规则域
- 所属文件模块
- 命中条款位置
- 原文证据
- 风险说明
- 规则依据

更细一层可拆为：

- 基础识别信息
- 定位与证据
- 解释信息

### 当前结论

V1 审查报告模板固定为 5 个部分：

1. 报告头信息
2. 总体结论
3. 风险总览
4. 重点问题摘要
5. 审查明细

## 2. 审查执行流程

### 2.1 输入装配

目标：

- 把进入 AI 的输入准备完整

建议内容：

- 解析后的文件内容
- 文件结构信息
- 默认规则包
- 固定审查任务指令
- 固定输出格式要求

更细一层可拆为：

- 文件内容输入
- 审查约束输入

### 2.2 审查切分方式

目标：

- 控制上下文规模，适应当前 LLM 环境

建议方式：

- 先按章节切分
- 再按条款聚焦
- 最后做一次汇总

更细一层可拆为：

- 章节级初筛
- 条款级审查
- 全文汇总

### 2.3 默认规则包加载方式

目标：

- 明确规则包如何参与 AI 审查

建议方式：

- 默认规则包作为主审查框架和输出护栏
- 不是完全替代 AI 判断的硬编码规则引擎

更细一层可拆为：

- 风险关注清单
- 输出约束模板

### 2.4 风险识别

目标：

- 从条款中识别结构化问题

建议输出：

- 风险标题
- 风险级别
- 执行级别
- 所属规则域
- 所属文件模块

更细一层可拆为：

- 问题命中
- 风险分类
- 风险分级

### 2.5 证据抽取

目标：

- 让风险判断可解释、可定位

建议内容：

- 命中条款位置
- 原文证据
- 证据说明

更细一层可拆为：

- 条款定位
- 原文截取
- 证据说明

### 2.6 风险说明生成

目标：

- 让审核人员理解为什么有风险

建议内容：

- 风险说明
- 规则依据

更细一层可拆为：

- 风险说明生成
- 规则依据绑定

### 2.7 中间结果落地

目标：

- 为第 3 步生成结果做准备

建议产出：

- 风险点对象
- 证据片段对象
- 风险说明对象

更细一层可拆为：

- 风险点落地
- 证据片段落地
- 汇总输入准备

### 当前结论

V1 审查执行流程固定为 7 个部分：

1. 输入装配
2. 审查切分方式
3. 默认规则包加载方式
4. 风险识别
5. 证据抽取
6. 风险说明生成
7. 中间结果落地

## 3. 规则库配置格式

### 3.1 规则标识信息

目标：

- 说明规则是谁

建议内容：

- 规则编号
- 规则名称
- 规则版本
- 规则状态

更细一层可拆为：

- 唯一标识
- 管理信息

### 3.2 规则分类信息

目标：

- 说明规则属于哪一类

建议内容：

- 一级规则域
- 二级细则名称
- 所属文件模块
- 执行级别
- 风险级别

更细一层可拆为：

- 业务分类
- 输出分类
- 文件定位分类

### 3.3 适用范围信息

目标：

- 说明规则适用于什么场景

建议内容：

- 适用采购类型
- 适用采购方式
- 适用区域

更细一层可拆为：

- 场景范围
- 区域范围

### 3.4 审查目标信息

目标：

- 说明规则到底要检测什么

建议内容：

- 检测目标
- 触发条件描述
- 触发模式

更细一层可拆为：

- 检测目标
- 触发描述
- 触发模式

### 3.5 输出映射信息

目标：

- 说明规则命中后如何生成统一输出

建议内容：

- 风险标题模板
- 风险说明模板
- 规则依据
- 证据抽取要求

更细一层可拆为：

- 标题与说明
- 依据与证据

### 3.6 版本升级信息

目标：

- 支持默认规则包可持续迭代

建议内容：

- 规则包版本
- 变更说明
- 启用状态

更细一层可拆为：

- 包版本信息
- 规则变更记录

### 当前结论

V1 规则库配置格式固定为 6 个部分：

1. 规则标识信息
2. 规则分类信息
3. 适用范围信息
4. 审查目标信息
5. 输出映射信息
6. 版本升级信息

## 4. 当前阶段结论

这份开发交付补充方案，补全了技术负责人从业务方案走向技术方案所需要的三类关键支撑：

1. 报告怎么长
2. 审查怎么跑
3. 规则怎么配
