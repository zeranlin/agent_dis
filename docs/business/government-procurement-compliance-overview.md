# 政府采购招标文件合规审查业务结论

## 业务方向

当前拟讨论和建设的业务方向是：政府采购招标文件合规性检查。

核心目标不是替代审核人员作出最终裁定，而是帮助审核人员快速、系统、可追溯地找出招标文件中的风险点。

## 核心问题

这个业务要解决的核心问题是：

- 审核人员面对的招标文件内容复杂、篇幅长、规则散
- 风险点往往分散在资格条件、技术参数、评分办法、合同条款等多个位置
- 人工审查容易遗漏跨章节矛盾、隐性歧视、程序漏洞和政策条款缺失
- 审查结论需要能说明“为什么有问题、依据是什么、证据在哪里、建议怎么改”

因此，产品目标不应只是“找错”，而应是：

1. 找出风险点
2. 给出规则依据
3. 锚定证据位置
4. 形成整改建议
5. 支持人工复核和留痕

## 什么是政府采购

在本业务语境中，政府采购可以定义为一种受强约束的公共支出交易行为，至少包含以下要素：

- 主体：国家机关、事业单位、团体组织等采购人
- 资金：财政性资金
- 客体：货物、工程、服务
- 目标：满足履职或公共服务需要
- 约束：必须遵守法定方式、程序、信息公开、政策功能和监督救济要求

它不是单纯的采购行为，也不是普通商业交易，而是一类带有公共治理目标和法定程序要求的交易活动。

## 法规与政策支撑

本业务的规范依据应分层理解，而不是只盯住某一部法律。

### 基础法层

- 《政府采购法》
- 《招标投标法》

### 实施层

- 《政府采购法实施条例》

### 部门规章与业务办法层

- 《政府采购货物和服务招标投标管理办法》
- 《政府采购需求管理办法》
- 《政府采购信息发布管理办法》
- 《政府采购质疑和投诉办法》

### 政策功能层

- 促进中小企业发展相关规则
- 绿色采购、节能环保相关规则
- 进口产品采购相关规则
- 框架协议、集采目录、政策采购等专项规则

### 地方规则层

除全国统一适用的规则外，各省、各地市还会叠加本地管理办法、标准文本、负面清单、操作细则和平台要求。

因此，后续产品中的每条审查规则都应有“适用区域”属性，至少区分：

- 全国适用
- 省级适用
- 市级适用

## 招标文件是什么

招标文件不是一份普通文书，而是一个将采购需求、竞争规则、评审方法、合同条件和权利义务一次性公开固化的法律化业务对象。

它至少承担以下五项功能：

1. 表达采购需求
2. 建立供应商竞争边界
3. 固化评审方法和评分规则
4. 约束采购人、代理机构和投标人行为
5. 为质疑、投诉、监督和审计提供证据基础

因此，招标文件应当具有完整性、明确性、可执行性、可验证性和可救济性。

## 招标文件为什么必须这样设计

招标文件之所以需要长成今天这种结构，不是为了形式完整，而是因为它要同时满足以下治理目标：

- 合法合规
- 公平竞争
- 公开透明
- 需求可执行
- 评审可解释
- 履约可验收
- 争议可处理

一旦某个部分缺失或表达不清，就可能带来歧视性限制、评审失衡、合同风险或程序风险。

## 招标文件的核心组成

后续业务建模时，可将招标文件理解为由以下核心模块构成：

- 项目基本信息
- 采购需求
- 供应商资格条件
- 投标响应要求
- 评审办法与评分标准
- 开评标与定标规则
- 异议、质疑、投诉路径
- 合同主要条款
- 履约与验收要求
- 政策功能条款

产品不应只判断它“像不像模板”，而应判断这些模块是否完整、协调、合法并可执行。

## 政府采购招标文件审查，到底审查什么

招标文件审查，本质上是在审查：

- 条款是否合法
- 竞争边界是否公平
- 采购需求是否明确合理
- 评审办法是否客观可执行
- 合同和履约条款是否存在失衡
- 程序设计是否留下监督和救济漏洞
- 政策功能是否落实到位

这意味着审查工作不是简单的文本校对，而是一种“规则约束下的文档风险识别”。

## 审查规则是否分大规则和细节规则

是，而且非常适合分成两层。

### 第一层：大规则

适合作为一级风险域或一级标签：

1. 合法性规则
2. 公平竞争规则
3. 需求合理性规则
4. 评审可解释性规则
5. 合同与履约风险规则
6. 程序与公开规则
7. 政策功能规则

### 第二层：细节规则

适合作为具体可执行的检测点，例如：

- 资格条件是否越权、过高或与项目无关
- 技术参数是否指向特定品牌、型号或少数供应商
- 评分办法是否主观性过强、重复加码或明显失衡
- 商务条款是否存在不合理付款、违约、保证金安排
- 程序条款是否前后矛盾、越权设定废标情形
- 政策条款是否遗漏或适用错误

因此，本业务应同时建设：

- 一级规则框架，用于快速总览风险版图
- 二级细则库，用于落地审查和生成证据链

## 为什么适合用本体论讨论

这个业务适合用本体论，不是因为概念听起来高级，而是因为它天然包含多个稳定对象、层级关系和推理关系。

如果只做一个问答式大模型系统，结论往往不稳定，也难以留痕和复核。

如果采用本体建模，则至少可以形成三层核心结构：

- 规则本体：法律规则、地方规则、政策规则
- 文档本体：招标文件结构、条款类型、章节关系
- 风险本体：风险类别、触发条件、证据、风险等级、整改建议

最终让系统输出的不只是“可能有问题”，而是：

- 问题是什么
- 依据是什么
- 证据在哪
- 风险等级多高
- 应该如何修改

## 当前结论

现阶段最重要的结论是：

这个业务最适合做成“规则本体 + 文档结构本体 + 风险本体”的三层系统，而不是单纯做一个大模型问答器。

因为审核人员真正需要的不是一句“AI 觉得有风险”，而是：

- 这是什么问题
- 违反或疑似违反哪类规则
- 风险证据在哪一段
- 风险程度如何
- 建议怎么改

这也是后续产品方案、规则库、提示词设计和审查工作流设计的共同基础。
