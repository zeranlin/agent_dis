### 2026-03-28 - 采用本体化思路讨论政府采购招标文件合规审查

- 状态：已采纳
- 背景：当前业务方向聚焦于政府采购招标文件合规性检查，需要一种既能承载法规约束，又能承载文档结构、风险分类和审查结论的讨论与建模方式。
- 决策：将该业务优先定义为“规则本体 + 文档结构本体 + 风险本体”的三层系统，并以帮助审核人员快速、系统、可追溯地发现风险点为核心目标。
- 影响：后续产品方案不应只做问答式输出，而应围绕规则依据、证据定位、风险等级和整改建议构建结构化能力。
- 相关资料：`docs/business/government-procurement-compliance-overview.md`

### 2026-03-28 - V1 技术实现采用最小闭环架构

- 状态：已采纳
- 背景：业务侧已经收口到单文件上传、自动审查、生成结果、人工查看下载的极简闭环，技术侧需要尽快形成可执行方案，同时避免过早平台化。
- 决策：V1 技术实现先采用“单体应用 + 异步审查执行器 + 持久化存储 + 对象存储”的最小闭环架构，并通过固定规则包、固定任务指令和有限对外状态保持主链路稳定。
- 影响：首版开发将优先围绕上传接口、任务状态、文件解析、审查编排和结果汇总展开；暂不引入复杂工作台、审批流或重型 Agent 编排框架。
- 相关资料：`docs/tech/v1/system-design.md`、`docs/tech/v1/state-machine.md`、`docs/tech/v1/api-spec.md`

### 2026-03-28 - V1 首个实现骨架采用 Python 标准库

- 状态：已采纳
- 背景：仓库当前还没有产品代码，需要尽快把上传接口和任务状态模型落到最小可运行实现，同时避免在早期引入过重依赖和工具链复杂度。
- 决策：V1 首个实现骨架先采用 Python 标准库构建最小单体服务与自动化测试，用本地文件系统承载原始文件、任务元数据和解析队列占位。
- 影响：当前可以在无额外依赖的前提下直接运行和验证上传接口与状态查询；后续如需替换为正式 Web 框架或外部存储，应保持当前对外接口和状态模型不变。
- 相关资料：`app/server.py`、`app/models.py`、`tests/test_upload_api.py`
