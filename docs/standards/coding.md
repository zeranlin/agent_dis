# 编码规范

## 通用原则

- 优先选择可读、朴素的默认方案，而不是炫技式实现。
- 保持模块足够小，边界和归属清晰。
- 只有在代码本身难以表达意图时才添加注释。
- 避免在脚本和自动化中引入隐蔽副作用。

## 面向智能体的约束

- 文件名和命令应当便于发现和检索。
- 初始化与验证命令应尽量短、可预测。
- 在增加新入口之前，优先复用现有入口。
- 如果某个新约束很重要，应把它编码进自动化，而不是依赖记忆。
- 任务上下文应写到足以让另一个智能体脱离聊天记录继续工作。
- 实验和决策记录优先采用追加式日志。

## 文档要求

- 当行为或工作流发生变化时，要在同一次变更里更新文档。
- 优先使用聚焦的小文档，而不是一个大而全的说明文件。
- 当一项改动引入新决策或推翻旧假设时，要更新对应日志。
- 本仓库中的 Markdown 文档内容默认使用中文。

## Mermaid 图规范

- Mermaid 流程图默认使用 `graph LR`，从左到右布局。
- Mermaid 图必须使用 `subgraph` 对内容分组，且至少包含 3 个分区。
- Mermaid 图整体层级应优先组织为“输入 -> 处理 -> 输出”。
- 每个节点文字保持简短，默认不超过 10 个字。
- 每个 `subgraph` 内节点数量不超过 8 个。
- 避免一次性展开全部细节，优先抽象主干结构。
- 优先保证图清晰可读，而不是追求完全展开。
- 主流程必须清晰，次要关系可以省略或减少连接。
- 尽量避免大量交叉线。
- Mermaid 代码块必须可直接渲染，不得包含语法错误。

## 讨论分层准则

- 业务方案讨论必须先完成主流程层，再进入子流程层。
- 在上一级流程未讨论完成前，不得提前展开下一级细节。
- 讨论顺序应遵循“1 -> 1.1 -> 1.1.1”这种逐层展开方式，不跳级讨论。
- 每轮讨论结束后，必须进行一次回顾，明确当前已完成到哪一层。
- 只有当同一层级的并列节点都完成讨论后，才进入下一层级。
- 任务管理也必须采用相同分层方式组织，主任务未完成前不提前拆到更深层任务。
