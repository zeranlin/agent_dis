# LLM V1 收缩方案草案

## 文档目的

这份文档用于说明当前 `LLM V1` 在技术实现上的收缩方向。

目标不是推翻“使用 `LLM` 做招标文件合规审查”的路线，而是回答以下问题：

1. 当前方案里哪些地方把 `LLM` 用重了
2. V1 应该把 `LLM` 收缩到什么边界
3. 收缩后怎样更快形成一版可验证成果

## 一、当前问题判断

当前主线已经暴露出一个很明确的风险：

- 正常样例在真实 `LLM` 审查执行阶段出现 `HTTP 504`

这说明当前方案的主要矛盾，已经不再是页面、规则库或结果展示，而是：

- `LLM` 在主流程中的负载过重
- 单次调用稳定性对整条主流程影响过大

因此，当前需要的不是继续扩写能力，而是先把 `LLM` 的使用边界收紧。

## 二、当前方案里 `LLM` 可能用重的地方

### 1. 把 `LLM` 放在主流程的重同步位置

当前正常样例一旦在审查执行阶段失败，整单任务就无法进入 `completed`。

这意味着：

1. `LLM` 不是普通能力组件，而是主流程关键单点
2. 一次超时会直接放大成整单失败
3. 页面验收、结果验收、下载验收都会被连带阻断

### 2. 单次请求负载可能过大

当前审查输入中容易叠加以下内容：

1. 条款片段文本
2. 条款上下文
3. 默认规则包
4. 固定审查任务指令
5. 输出结构要求

当片段数量、规则数量和上下文长度共同放大时，单次推理请求就容易变成“高时延、高失败率、高成本”的调用。

### 3. 让 `LLM` 同时承担太多职责

当前 `LLM` 实际上可能同时参与：

1. 风险识别
2. 风险解释
3. 证据抽取
4. 结果说明生成
5. 局部结论组织

如果这些责任在一次调用中叠加，就会把 `LLM` 从“判断器”变成“全能处理器”，显著增加不稳定性。

### 4. 对单次成功率依赖过高

当前如果没有足够轻量的：

1. 分批
2. 缩批
3. 最小重试
4. 断点续跑

那么一次 `504` 就会让任务整体失败，而不是只影响其中一个子批次。

### 5. V1 过早同时追求稳定性、精度和完整表达

当前技术链路已经逐步叠加了：

1. 高价值规则召回
2. 风险组口径
3. 长条款细切分
4. 结果页和 Markdown 下载

这些方向本身都合理，但如果它们都强依赖单次真实 `LLM` 审查执行稳定成功，V1 的验证节奏就会明显变慢。

## 三、V1 收缩原则

当前建议采用以下 5 条收缩原则。

### 1. 不取消 `LLM`，但收缩其职责

`LLM` 仍然是 V1 的核心审查引擎，但只负责最关键的“规则约束下的片段级判断”。

### 2. 不让 `LLM` 承担整单重处理

不要让 `LLM` 在一次重调用中负责整份文件的完整理解和完整输出。

### 3. 能交给代码做的事情，尽量交给代码

例如：

1. 批次调度
2. 结果归并
3. 风险组汇总
4. Markdown 拼装
5. 页面对象组装

这些应尽量由代码完成，而不是继续压给 `LLM`。

### 4. 优先保证正常样例进入 `completed`

V1 当前最重要的验证目标，不是“精度再提升 5%”，而是：

- 正常样例稳定进入 `completed`

### 5. 先形成可验证成果，再继续做精度优化

只有当主链路稳定、结果页可读、下载可用后，继续精调才有意义。

## 四、建议收缩后的 V1 边界

### 1. `LLM` 只负责片段级审查判断

收缩后，`LLM` 的核心职责应限定为：

1. 判断当前片段是否命中给定规则
2. 输出命中的风险说明
3. 输出对应证据文本
4. 输出结构化结果字段

不再让 `LLM` 负责：

1. 整份文件级自由总结
2. 报告全文拼装
3. 页面摘要最终成稿
4. 大段跨章节综合写作

### 2. 结果汇总继续由代码负责

收缩后仍然保持现有方向：

1. 风险组由代码汇总
2. 总体结论由代码生成
3. Markdown 报告由代码拼装
4. 页面对象由代码组装

这可以减少 `LLM` 的输出负担，并提高结果一致性。

### 3. 规则范围以默认规则包为主

V1 不继续扩大规则范围。

继续使用当前默认规则包作为主规则边界，避免因为规则面继续膨胀，进一步放大单次调用负载。

### 4. 页面与交互范围不再扩展

收缩期内不再扩以下内容：

1. 多文件
2. 工作台
3. 复杂筛选
4. 复杂结果流转
5. 复杂运营后台

## 五、建议的执行收缩方式

### 1. 分批执行

当前最优先的收缩方式，是把真实审查执行改成更轻量的分批调用。

建议方向：

1. 每次只处理少量条款片段
2. 每个批次只携带必要规则上下文
3. 让单次请求负载明确受控

### 2. 缩小单次请求内容

单次请求只保留必要输入：

1. 片段正文
2. 最小必要上下文
3. 当前规则集合
4. 固定输出格式要求

应尽量避免：

1. 过长上下文重复透传
2. 过多无关规则说明反复透传
3. 非必要的大段示例反复进入每一批

### 3. 增加最小重试

对 `504`、超时这类明显属于调用层波动的问题，建议先加入最小重试机制。

目标不是复杂容灾，而是让短时波动不直接变成整单失败。

### 4. 允许缩批

如果大批次不稳定，应允许自动或显式缩小批次大小。

收缩目标是：

- 宁可多调几次，也不要一次调得过重

### 5. 优先恢复 `completed`

当前修复优先级应按以下顺序：

1. 正常样例重新进入 `completed`
2. 结果页重新可读
3. Markdown 下载重新可用
4. 再回头做下一轮验收

## 六、建议的 V1 可验证版本

当前建议把 V1 验证版明确收缩成以下形态。

### 1. 输入边界

1. 单文件
2. `PDF` / `DOCX` / `DOC`
3. 单次上传触发

### 2. 审查边界

1. 使用当前默认规则包
2. 只做结构化风险审查
3. `LLM` 只做片段级判断

### 3. 输出边界

1. 风险点
2. 风险组
3. 总体结论
4. 结果页
5. Markdown 报告

### 4. 稳定性边界

1. 支持轻量分批
2. 支持最小重试
3. 支持缩批
4. 支持恢复正常样例进入 `completed`

## 七、当前不建议做的事

在收缩期内，不建议继续做以下事情：

1. 继续扩规则面
2. 继续叠加复杂结果层补丁
3. 继续扩大结构切分复杂度
4. 继续增加新的页面能力
5. 在未恢复稳定性前继续追求更高精度

这些动作大概率会继续拉长验证周期，而不是帮助 V1 更快形成成果。

## 八、验收口径

收缩后的 V1 是否可验证，建议只看以下口径：

1. 真实样例是否稳定进入 `completed`
2. 补充正常样例是否稳定进入 `completed`
3. 结果页是否可读
4. Markdown 下载是否可用
5. 风险组、结论和证据展示是否具备业务验证价值

如果这 5 项成立，就可以认为：

- 当前 `LLM V1` 已经达到“可验证版本”

## 九、当前正式建议

当前正式建议如下：

1. 不推翻 `LLM` 审查路线
2. 明确认定当前 `LLM` 在执行层有“使用偏重”的问题
3. 通过职责收缩、轻量分批、最小重试、缩批控制来收口
4. 先形成一版“稳定进入 `completed` 的可验证版本”
5. 在此基础上，再决定是否继续做下一轮精度优化
