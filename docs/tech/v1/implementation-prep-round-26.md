# V1 实现准备第二十六轮（首版）

## 文档目的

这份文档用于承接“继续进入下一轮解析链路深化”的最新指令，在保持 V1 极简边界的前提下，把本轮主题切换为“V1 收口判断与交付准备”，对当前主线是否已达到最小可交付门槛形成正式技术结论。

## 1. 产品追溯

本轮判断直接追溯以下已确认文档：

- `docs/status/summary.md`
- `docs/tasks/t-now.md`
- `docs/tasks/current.md`

- `docs/tech/v1/implementation-prep-round-25.md`

## 2. 本轮主题边界

本轮不再继续补零散缺口，而是统一回答 4 个问题：

1. V1 主线是否已经达到最小可交付门槛
2. 当前已收口范围是否与既定产品边界一致
3. 当前还剩哪些问题，但这些问题是否属于非阻塞项
4. 如果进入联调和验收，技术侧建议怎样组织交付准备

本轮明确不做：

- 新解析能力扩展
- 新规则能力扩展
- 新页面或新工作台扩展
- OCR、多文件、复杂表格恢复
- 任何超出既定 V1 边界的能力补充

## 3. 当前收口判断

### 3.1 主流程判断

对照 `docs/business/v1/acceptance-criteria.md`，当前主流程已经形成稳定闭环：

1. 文件进入系统：已支持单文件上传，覆盖 `PDF`、`DOCX`、`DOC`
2. AI 发起审查：上传后可进入解析、装配、审查执行最小链路
3. 生成审查结果：已可生成最终结论、风险列表、Markdown 报告与页面结果对象
4. 人工查看结果：已支持结果页查看和 Markdown 下载

技术判断：

- 当前主流程已经达到 V1 最小可交付门槛
- 当前主线不再存在阻塞性交付缺口

### 3.2 输入侧判断

对照 `docs/business/v1/parsing-and-segmentation-minimum.md`，当前输入侧已达到“可用于最小稳定审查”的程度：

- 已接入 `PDF` / `DOCX` / `DOC` 最小真实解析入口
- 已具备正文轻清洗、目录噪声过滤、可审查正文判定
- 已具备章节 / 条款 / 段落混合切分与切分块对象
- 已补齐 `DOCX` 表格文本提取与重复提取修复
- 已把章节上下文、条款类型与定位线索稳定传入审查输入

技术判断：

- 当前输入侧已满足 V1 最小审查闭环需要
- 解析质量仍是“最小可用”，但不再构成交付阻塞

### 3.3 结果消费侧判断

当前结果消费闭环已完成以下收口：

- 结果接口、报告输出与结果页展示口径一致
- 风险排序、上下文解释、下载对象字段已统一
- `completed` / `reviewing` / `failed` / `not_found` 场景均有稳定反馈
- 页面结果对象已具备规范字段、主操作按钮和辅助说明对象

技术判断：

- 当前结果消费侧已经达到 V1 交付要求
- 当前页面层没有新的阻塞性技术缺口

## 4. 已收口范围与非阻塞项

### 4.1 已收口范围

当前可以正式视为已收口的范围如下：

1. 上传接口与任务创建
2. 最小文件解析与失败分类
3. 结构切分与审查输入装配
4. 规则加载与最小审查执行
5. 结果汇总、最终结论生成与 Markdown 报告导出
6. 结果接口、下载接口与结果页展示
7. 结果页的失败态、审核中态、完成态与不存在态反馈

### 4.2 非阻塞项

当前仍存在但不构成 V1 阻塞的事项如下：

1. 真实复杂文档上的解析准确率仍有继续优化空间
2. 复杂表格结构恢复仍未进入 V1
3. 扫描件、图片型 `PDF` 与 OCR 仍未进入 V1
4. 规则覆盖度和执行精细度仍属于后续增强项
5. 页面体验仍可继续打磨，但当前已满足最小交付要求

这些事项应视为后续版本优化项，不应倒逼当前 V1 扩范围。

## 5. 交付准备建议

### 5.1 技术侧建议的交付入口

当前建议把 V1 交付准备收敛为以下 3 项：

1. 基于既有测试与 `make check` 继续作为基础门禁
2. 组织一轮真实样例文件联调，重点验证上传、解析、完成、失败 4 类路径
3. 组织一轮按业务验收口径的结果页和下载文件人工走查

### 5.2 联调建议清单

联调时优先覆盖以下样例：

1. 正常文本型 `PDF`
2. 正常 `DOCX`
3. 常规 `DOC`
4. 无可审查正文文件
5. 不支持格式文件

每类样例重点确认：

- 任务状态是否正确推进
- 失败分类是否准确
- 结果对象字段是否完整
- 页面展示与下载内容是否一致

### 5.3 交付边界提示

进入交付准备时，应明确向评审和验收方说明：

- 当前版本是 V1 最小闭环，不承诺复杂版式高精度恢复
- 当前版本不处理 OCR、多文件、附件联审和复杂表格精细结构
- 当前版本重点验证“可审、可看、可下载、可追溯”

## 6. 主要技术风险

本轮判断后，当前剩余风险主要集中在“预期管理”而不是“主链路缺失”：

1. 若用复杂扫描件或高度异形文档做验收，容易把非 V1 范围误判为缺陷
2. 若后续修改结果对象字段而不同步页面层，可能重新引入口径漂移
3. 若在未明确业务边界前继续扩规则或扩页面，容易破坏当前收口节奏

对应控制方式：

- 验收前再次明确 V1 非范围项
- 继续以统一 presenter / schema 字段作为结果消费入口
- 后续新增主题必须先写正式文档，再进入实现

## 7. 当前结论

当前技术判断是：

1. V1 主线已经基本收口
2. 当前实现已经达到最小可交付门槛
3. 当前可以从“解析链路深化”切换到“V1 收口复审与交付准备”
4. 本轮之后不建议再回头补零散点，除非复审发现新的阻塞性缺口

因此，本轮建议总负责人按“V1 收口判断与交付准备”整主题进行复审；如复审通过，可把后续主线切换为联调、验收和交付组织，而不是继续扩展 V1 范围。
