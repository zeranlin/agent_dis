## 一、主链路核查结论

### 1. 首轮新增 `3` 组规则已经进入主链路

本轮核查确认，以下能力已经实际进入可执行代码路径：

1. `R3` 资格条件违法限定
2. `R4` 资格业绩区域 / 数量定向
3. `R9` 评分因素违法与价格分规则合法性

已确认进入的位置包括：

1. 候选筛选阶段：规则候选片段排序已消费 `module_type / unit_label / unit_name`
2. 送审阶段：批次规则选择会优先保留当前批次存在候选片段的规则
3. 审查阶段：启发式预命中与 `LLM` 命中在同一主链路内并流
4. 落库阶段：`rule_code` 会进一步归一化为已知规则并落到正式 `rule_id`
5. 报告阶段：最终结果页与 Markdown 报告继续消费最新结果对象

### 2. `rule_code` 全空并不是风险真的没有规则

本轮已查明，上一轮“`rule_code` 全空”的判断来自读到了同一 `task_id` 的旧结果，而不是最新风险对象真的丢了规则编号。

已确认的根因如下：

1. 同一任务多次生成结果时，`results.json` 会保留多份结果对象
2. 旧的 `get_result_by_task()` 只返回命中的第一份结果，不保证是最新结果
3. 老报告仍然是旧骨架格式，因此复核时会误以为新链路没有生效

当前已修正为：

1. 保存结果时，先清理同一 `task_id` 的旧结果
2. 读取结果时，按 `generated_at` 返回最新结果

这意味着：

1. `risk.rule_id` 实际并未整批丢失
2. 上一轮看到的旧标题、旧报告口径和旧高风险数量，根因是结果串读

本轮只做了最小过滤，不扩复杂策略，不改变 `V1` 主流程。
