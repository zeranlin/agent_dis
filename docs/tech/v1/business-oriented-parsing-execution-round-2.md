# 业务化解析执行第二轮

## 1. 本轮目标

本轮继续落实 `docs/business/v1/government-procurement-business-oriented-parsing-plan.md` 中“四类高价值模块优先形成稳定业务单元入口”的要求。

本轮不扩成完整 `module_block -> review_unit -> unit_fragment` 三级对象，而是在现有 `ClauseRecord` 上继续前推一层，让四类高价值模块不再只停留在 `module_type + unit_type` 的粗粒度分类。

## 2. 本轮范围

本轮只做以下事情：

1. 把资格条件、采购需求、评分办法、合同条款四类模块推进到更稳定的业务单元入口
2. 给现有解析对象补单元入口标识与单元名称
3. 让执行载荷、结果说明、结果页和 Markdown 报告继续消费这些字段

## 3. 本轮实现

### 3.1 解析对象补业务单元入口字段

当前在 `ClauseRecord` 上新增以下字段：

1. `review_unit_id`
2. `unit_label`
3. `unit_name`

这三个字段的定位如下：

1. `review_unit_id`：表示当前片段归属的业务单元入口
2. `unit_label`：表示当前业务单元入口的业务类型
3. `unit_name`：表示当前业务单元入口的可读名称

当前仍然没有单独拆出独立 `review_unit` 表，但已经把“业务单元入口”的最小核心信息显式挂到现有 `clause` 上。

### 3.2 四类高价值模块进入更细单元标签

当前四类高价值模块已开始进入更细的 `unit_label`。

#### 资格条件

当前会优先识别：

1. `单条资格要求`
2. `单条业绩要求`
3. `单条资质要求`
4. `单条信用要求`

#### 采购需求

当前会优先识别：

1. `单条技术参数`
2. `单个参数项`
3. `单条功能要求`
4. `单条服务要求`
5. `单条交付要求`
6. `单个偏离项`

#### 评分办法

当前会优先识别：

1. `单个评分项`
2. `价格分规则`
3. `商务分规则`
4. `技术分规则`

#### 合同条款

当前会优先识别：

1. `单条合同条款`
2. `付款条款`
3. `验收条款`
4. `违约责任条款`
5. `质保条款`
6. `费用承担条款`

### 3.3 业务单元入口开始具备稳定名字

当前 `unit_name` 已开始按以下思路生成：

1. 表格类优先取首个有业务意义的数据列
2. 普通条款优先取块标题或锚点标题
3. 过长名称会做轻量截断，避免结果对象膨胀

这一步的价值在于：

1. 后续风险说明不再只剩“某章节某片段”
2. 同一业务单元被拆成多个子片段时，仍可共享同一个 `review_unit_id`
3. 结果页与报告可逐步从“片段视角”转向“业务单元入口视角”

### 3.4 执行与展示链路继续透传

当前以下链路已继续消费新字段：

1. 审查输入 `payload.clauses`
2. 输出 `schema`
3. `review_reasoning`
4. 结果页重点风险摘要
5. Markdown 报告风险组明细

这意味着本轮新增字段不是只停留在解析落库里，而是已经进入审查和展示主链路。

## 4. 真实样例快速观察

使用真实样例 `SZCG2025000300-A` 做解析快速观察时，当前已能看到以下更细业务单元标签进入结果：

1. `资格条件 / 单条资质要求`
2. `采购需求 / 单条技术参数`
3. `采购需求 / 单条功能要求`
4. `采购需求 / 单个偏离项`
5. `评分办法 / 单个评分项`
6. `合同条款 / 付款条款`
7. `合同条款 / 验收条款`
8. `合同条款 / 费用承担条款`

这说明当前实现已经不再只是“高价值模块字段分类”，而是开始接近“高价值业务单元入口”。

## 5. 自动化验证

本轮已补并通过以下验证：

1. 四类高价值模块能进入更细的 `unit_label`
2. 参数表数据行能提取稳定的 `unit_name`
3. 审查输入 `schema` 已包含 `review_unit_id`、`unit_label`、`unit_name`
4. 审查批次 `payload` 已携带新的业务单元入口字段

## 6. 当前结论

本轮正式结论如下：

1. 当前已把四类高价值模块从“字段分类后的 clause”继续推进到“更稳定的业务单元入口”
2. 当前仍采用 V1 最小落地方式，没有直接跳到完整独立 `review_unit` 对象体系
3. 当前结果页、Markdown 报告和执行链路已开始共享同一套业务单元入口字段

## 7. 下一步建议

下一步优先级建议如下：

1. 继续提升真实文件中的章节识别稳定性，减少 `默认章节` 残留
2. 继续压缩程序条款、采购需求、偏离项之间的边界误判
3. 在确认口径稳定后，再考虑是否单独拆出独立 `review_unit` 对象
