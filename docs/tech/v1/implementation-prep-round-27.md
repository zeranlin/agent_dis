# V1 实现准备第二十七轮（首版）

## 文档目的

这份文档用于承接“联调、验收与交付准备”的最新指令，在不扩 V1 范围的前提下，把交付视角下最需要直接执行的内容一次收口为可落地清单。

本轮不再继续补零散实现，除非联调或验收过程中发现新的阻塞性交付缺口。

## 2. 本轮主题边界

本轮只处理交付准备相关内容：

1. 联调样例与联调检查项
2. 验收入口与验收走查顺序
3. 交付物清单与技术交接说明
4. 非范围提醒与阻塞缺口升级规则

本轮明确不做：

- 新增解析能力
- 新增规则能力
- 新增页面交互
- 新增平台化能力
- 对已通过复审的旧主题回头补点

## 3. 联调准备

### 3.1 联调目标

联调阶段只验证 3 件事：

1. V1 主流程是否稳定跑通
2. 失败路径是否能给出正确反馈
3. 页面展示与下载结果是否保持一致

### 3.2 联调样例清单

建议至少准备以下 5 类样例：

1. 文本型 `PDF` 正常样例
2. 常规 `DOCX` 正常样例
3. 常规 `DOC` 正常样例
4. 无可审查正文样例
5. 不支持格式样例

如联调资源允许，可额外补 1 类“复杂但仍在 V1 边界内”的样例，用于观察最小解析能力的稳定性，但不应把复杂扫描件或 OCR 场景混入本轮主验收。

### 3.3 每类样例的检查项

每个联调样例都统一检查以下内容：

1. 上传后是否生成唯一任务标识
2. 任务状态是否按预期推进到 `reviewing`、`completed` 或 `failed`
3. 失败样例的失败分类是否正确
4. 完成样例是否生成最终结论、风险列表和 Markdown 结果文件
5. 结果页展示是否与下载文件中的结论口径一致
6. `reviewing`、`failed`、`completed`、`not_found` 页面反馈是否符合预期

## 4. 验收入口

### 4.1 验收前置

进入人工验收前，先完成以下基础动作：

1. 执行 `make check`
2. 准备联调样例文件
3. 明确本轮验收只按 V1 边界判断，不额外拉新范围

### 4.2 验收走查顺序

建议按以下顺序走查：

1. 先验证上传页能正常提交文件
2. 再验证审核中状态能体现任务处理中
3. 再验证完成态结果页能展示结论、风险与下载入口
4. 再验证失败态页面能体现失败原因
5. 最后验证任务不存在场景的页面级反馈

这样排序的原因是先锁定主流程，再补异常分支，避免在非主线异常里消耗过多时间。

### 4.3 验收通过口径

当前建议统一按以下口径判断是否通过：

1. 主流程 4 步均可跑通
2. 至少 1 个正常样例可完整走到结果查看与下载
3. 至少 1 个失败样例能在正确阶段失败并给出正确提示
4. 页面展示与下载结果的关键信息一致
5. 过程中未发现新的阻塞性交付缺口

## 5. 交付物与交接说明

### 5.1 当前交付物清单

本轮对交付对象的理解统一为以下 5 类：

1. 上传与结果查看最小服务骨架
2. 默认规则包与固定审查任务指令
3. 最小解析、审查执行、结果汇总闭环
4. 结果页与 Markdown 下载输出
5. 技术文档、状态板与通信记录

### 5.2 技术交接重点

交接给总负责人或后续联调同学时，优先说明以下内容：

1. V1 做什么：单文件上传、最小审查、结果查看与下载
2. V1 不做什么：OCR、多文件、附件联审、复杂表格恢复
3. 结果判断依赖什么：默认规则包、固定审查任务指令、最小解析输出
4. 验收重点看什么：主流程是否跑通、结果是否可读、失败是否可解释

### 5.3 建议引用入口

交接时建议优先引用以下文档：

1. `docs/business/v1/acceptance-criteria.md`
2. `docs/business/v1/parsing-and-segmentation-minimum.md`
3. `docs/tech/v1/implementation-prep-round-26.md`
4. `docs/tech/v1/implementation-prep-round-27.md`
5. `docs/tasks/current.md`

## 6. 阻塞缺口升级规则

从本轮开始，只有以下问题才应升级为新的技术实现工作：

1. 主流程无法跑通
2. 结果对象缺失导致页面或下载不可用
3. 失败分类错误导致用户无法判断如何处理
4. 页面与下载结果口径明显冲突

以下问题暂不升级为阻塞性交付缺口：

1. 个别复杂文档上的解析精度不够理想
2. 复杂表格结构没有恢复
3. 页面文案仍有继续润色空间
4. 后续规则能力仍可继续增强

## 7. 当前结论

当前技术侧已经正式进入“联调、验收与交付准备”阶段。

本轮之后的默认动作应为：

1. 先按本文件组织联调与验收
2. 只在发现阻塞性问题时再回到实现修改
3. 否则继续围绕交付说明、走查执行和最终交接推进
