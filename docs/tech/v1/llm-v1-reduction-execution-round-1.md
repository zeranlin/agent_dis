# LLM V1 收缩方案执行首包

## 1. 本轮背景

本轮不继续优先做真实 `completed` 长时间验收复跑，原因不是页面或流程阻塞，而是当前 `LLM` 仍然存在明显“用重了”的问题：

1. 单次请求仍重复携带整包规则
2. 单次条款文本仍偏长
3. 请求体仍包含一部分非必要字段
4. 真实联调、测试和验收时间被 `LLM` 吞吐拖长

结合当前联调环境的实际约束：

1. 当前瓶颈主要在 `tokens/s`
2. 当前并发能力不适合持续压大批量、长上下文请求

因此，本轮先回到 `docs/tech/v1/llm-v1-reduction-plan.md`，继续把执行层收轻、收小、收稳。

## 2. 本轮目标

本轮只做以下事情：

1. 进一步缩小单次 `LLM` 请求负载
2. 让每个批次只携带必要规则上下文
3. 保持现有结果汇总、页面与下载链路不扩散
4. 用自动化测试确认收缩没有打断主链路

## 3. 本轮实现

### 3.1 收紧默认执行参数

已将默认参数进一步收紧为更轻量口径：

1. `OPENAI_REVIEW_BATCH_SIZE` 默认值由 `8` 收紧到 `4`
2. `OPENAI_REVIEW_CLAUSE_MAX_CHARS` 默认值由 `1200` 收紧到 `800`
3. `OPENAI_REVIEW_BATCH_CHAR_BUDGET` 默认上限由 `1400` 收紧到 `1000`
4. 新增 `OPENAI_REVIEW_RULE_LIMIT`，默认每批最多携带 `6` 条规则

### 3.2 按批次筛选规则

此前每个批次都会把整包规则全部发送给 `LLM`。

本轮调整为：

1. 优先保留高价值规则：`R1`、`R3`、`R5`、`R9`、`R12`
2. 再按当前批次条款文本，对 `focus_terms`、`positive_examples`、`hit_definition`、`rule_name` 做轻量匹配
3. 最后只补足到本批上限，不再默认透传整包规则

### 3.3 精简传给 `LLM` 的字段

本轮从批次载荷中去掉了非必要字段：

1. 去掉 `task_id`
2. 去掉 `document_id`
3. 去掉 `file_name`
4. 去掉条款级 `location_label`
5. 不再继续透传 `negative_examples`

同时保留最小必要字段：

1. 规则编号与规则名称
2. 风险等级与执行等级
3. 命中定义
4. 重点关注词
5. 少量正例
6. 条款编号、章节标题、片段类型、条款正文

### 3.4 继续由代码侧负责结果组织

本轮没有把更多职责压回 `LLM`。

仍由代码侧继续负责：

1. 分批调度
2. 失败拆批
3. 风险对象落库
4. 结果聚合
5. 页面对象组装
6. Markdown 拼装

## 4. 自动化验证

本轮已通过以下关键验证：

1. 批次载荷仍保留必要规则指导字段
2. 批次规则选择会优先保留高价值规则和当前批次命中规则
3. 失败拆批后任务仍可继续推进
4. 审查执行、结果聚合和 worker 闭环测试仍通过

## 5. 本轮结论

本轮正式结论如下：

1. `LLM` 执行层已进一步向“片段级判断器”收口
2. 当前实现已不再默认按整包规则重复压给每个批次
3. 当前默认批量、条款长度和批次字符预算已进一步收紧
4. 当前更适合继续走“轻量执行优先”的主线，而不是回到长时间真实验收硬跑

## 6. 下一步建议

1. 先以当前收缩实现继续做小成本链路验证，不再默认做长时间真实样例全量复跑
2. 如需恢复真实验收，应优先选更短样例或更小批次口径
3. 后续若仍出现显著耗时，再继续从规则筛选和候选片段筛选两侧收缩
