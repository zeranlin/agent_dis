# V1 技术风险与实现假设（首版）

## 文档目的

这份文档用于显式记录 V1 当前技术假设、主要风险和对应缓解策略，避免实现过程中把关键前提只留在聊天上下文里。

## 1. 当前技术假设

### 1.1 输入假设

- 用户一次只上传 1 份招标文件
- 输入文件为 `PDF` 或 `Word`
- 主要处理对象是招标文件正文

### 1.2 运行假设

- 审查能力主要依赖 LLM
- 当前环境单并发能力约为 30 tokens/秒
- 审查链路允许使用异步任务执行

### 1.3 输出假设

- V1 结果以最终结论和审查报告为主
- 下载文件优先采用 `Markdown`
- 风险点、证据片段和规则依据作为支撑层输出

## 2. 主要技术风险

### 风险 1：长文上下文过大导致审查不稳定

- 背景：招标文件正文通常较长，整篇直接送入模型容易超出稳定上下文范围，且输出质量波动大。
- 影响：审查漏判、输出结构不稳定、响应耗时过高。
- 缓解策略：
  - 采用章节级初筛和条款级审查的两段式执行
  - 固定输出 schema
  - 控制单次送模文本大小

### 风险 2：PDF / Word 解析质量影响后续审查

- 背景：原始文档格式不规范时，目录、页码和条款切分可能不稳定。
- 影响：证据定位不准确，风险点无法可靠映射到原文位置。
- 缓解策略：
  - 先把解析状态作为独立阶段暴露
  - 保留原始文本和切分结果
  - 对切分失败文档直接进入失败态，不做模糊继续执行

### 风险 3：规则包与模型输出难以稳定对齐

- 背景：若规则定义过于抽象，模型可能无法稳定给出结构化命中结果。
- 影响：风险点对象字段缺失，结果汇总失败。
- 缓解策略：
  - 规则包字段统一收口到执行层对象
  - 任务指令固定输出格式和必填字段
  - 对模型输出增加 schema 校验和有限重试

### 风险 4：最终结论与中间结果不一致

- 背景：如果结果汇总阶段脱离中间结果单独生成，容易出现报告和风险明细不一致。
- 影响：结果可解释性下降，审阅人员不信任输出。
- 缓解策略：
  - 最终结论必须基于已落地的风险点和证据片段汇总
  - 报告模板固定结构
  - 结果生成前增加一致性校验

### 风险 5：失败态不可见导致用户误判系统卡住

- 背景：异步链路中若只展示“审核中”，用户无法判断是处理中还是失败。
- 影响：重复提交、排障困难。
- 缓解策略：
  - 明确 `failed` 状态
  - 所有关键阶段记录 `status_message`
  - 页面和接口都要允许展示失败信息

## 3. 工程实现假设

- V1 可以接受单体应用加异步执行器的部署方式
- 默认规则包和固定任务指令首版可先以文件资产形式管理
- 首版不强制要求复杂权限系统和多角色后台
- 首版允许通过最小数据持久化支持查询和排障

## 4. 待后续验证的问题

1. PDF 和 Word 的解析方案哪一种在条款切分上更稳定
2. 章节级初筛是否能显著减少条款级送模规模
3. 默认规则包 1.0 的字段是否足够支撑稳定审查输出
4. 报告 Markdown 是否需要同步产出 HTML 预览版本

这些问题适合在后续实现或试运行阶段写入 `docs/logs/experiments.md`。

## 5. 当前结论

V1 的主要风险不在页面复杂度，而在“解析质量、上下文控制、结构化输出稳定性和失败态可见性”。因此首版技术实现必须优先保证链路稳定和结果可追溯，而不是优先追求复杂交互或平台化能力。
