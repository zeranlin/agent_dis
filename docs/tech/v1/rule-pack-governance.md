# 规则库在总流程中的位置与维护策略

## 文档目的

这份文档用于正式说明 V1 阶段默认规则包在总流程中的位置、当前维护机制和版本策略。

目标不是把规则库描述成一个独立规则引擎，而是明确：

1. 规则库在当前 `LLM` 审查链路里到底承担什么职责
2. 规则库和固定审查任务指令、代码执行链路分别如何分工
3. 规则库后续应如何维护、升级和复盘

## 当前定位

V1 的规则库，本质上是“默认规则包”。

它不是：

1. 完整法规知识库
2. 可视化规则运营平台
3. 让运营人员日常手工维护的大型规则系统

它当前是：

1. `LLM` 审查的主规则边界
2. 高价值风险的默认审查范围
3. 稳定审查输出口径的约束层
4. 后续漏报、误报收口的主要低成本抓手

当前仓库中的默认规则包资产位于：

- `assets/review/rule-packs/default-rule-pack.v1.json`

## 在总流程中的位置

当前 V1 主流程是：

1. 文件进入系统
2. 文件解析与结构切分
3. 加载默认规则包和固定审查任务指令
4. 组装审查输入
5. `LLM` 按规则逐条审查
6. 生成命中结果
7. 汇总结论、结果页与 Markdown 报告

在这条链路中，规则库位于“解析之后、审查之前”，是审查执行层的核心输入之一。

### 1. 输入侧位置

规则库不直接处理原始文件。

它依赖前置链路先完成：

1. 文本提取
2. 章节 / 条款 / 片段切分
3. 审查片段装配

只有当文件被拆成可审查片段后，规则库才开始生效。

### 2. 审查侧位置

规则库会和以下对象一起进入审查执行器：

1. 条款或片段文本
2. 条款上下文
3. 默认规则包
4. 固定审查任务指令

执行器不是把整份文件直接丢给 `LLM` 自由发挥，而是把“片段 + 规则 + 指令”一起交给 `LLM`。

### 3. 输出侧位置

规则库不直接生成最终报告。

它首先约束 `LLM` 产出结构化命中结果，例如：

1. 命中了哪条规则
2. 为什么命中
3. 证据片段在哪里
4. 风险级别是什么

之后再由结果汇总链路生成：

1. 风险组
2. 总体结论
3. 页面展示对象
4. Markdown 报告

## 规则库当前起什么作用

当前规则库在总流程中主要承担 4 个作用。

### 1. 定义审查范围

规则库首先回答“审什么”。

V1 当前不是泛化阅读整份招标文件，而是围绕默认规则包定义的风险类型开展审查。

例如当前重点规则包括：

1. 地域限制
2. 所有制 / 规模歧视
3. 资格条件过高
4. 品牌 / 型号指向
5. 评分项未量化
6. 关键条款缺失 / 责任失衡

### 2. 约束命中边界

规则库不仅给出规则名称，还通过字段约束模型判断边界。

当前关键字段包括：

1. `rule_code`
2. `rule_name`
3. `rule_domain`
4. `execution_level`
5. `risk_level`
6. `hit_definition`
7. `focus_terms`
8. `positive_examples`
9. `negative_examples`
10. `review_focus`

这些字段的作用，是减少 `LLM` 出现以下问题：

1. 风险类型乱扩展
2. 规则标题错位
3. 看到关键词就误判
4. 忽略高价值风险表述

### 3. 稳定输出口径

规则库会约束命中的结构化输出，使不同文件、不同轮次的审查结果尽量使用相同规则口径。

这使得后续结果汇总时能够稳定处理：

1. 规则编码
2. 风险名称
3. 风险域
4. 风险级别
5. 审查说明

### 4. 成为后续优化抓手

当真实样例出现问题时，规则库是最重要的优化入口之一。

例如：

1. 漏报时，可先补命中定义、关注词和正例
2. 误报时，可先补反例并收紧边界
3. 标题错位时，可先修规则名称和审查重点

这意味着很多质量优化，不需要第一时间改主流程代码，而是先改规则包。

## 与其他层的分工

当前应明确区分以下三层。

### 1. 固定审查任务指令

固定审查任务指令负责：

1. 规定 `LLM` 的角色
2. 规定输出格式
3. 规定整体审查方式
4. 规定高优先级判断原则

它解决的是“怎么审”。

### 2. 默认规则包

默认规则包负责：

1. 规定审查范围
2. 规定各条规则的命中边界
3. 规定高价值风险关注点
4. 规定规则级输出口径

它解决的是“审哪些风险、每条风险如何判定”。

### 3. 代码执行链路

代码执行链路负责：

1. 文件解析
2. 片段切分
3. 输入装配
4. `LLM` 调用
5. 结果归并
6. 最终结论与报告输出

它解决的是“如何把规则和指令真正执行起来并变成结果”。

## 当前维护机制

V1 阶段不采用“重运营、重配置”的规则维护方式。

当前维护机制应理解为：基于真实审查结果，持续迭代默认规则包。

### 1. 维护触发来源

当前规则库维护主要来自以下 3 类输入：

1. 真实样例联调
2. 质量复盘结论
3. 规则命中前后差异对比

典型触发场景包括：

1. 高价值风险漏报
2. 规则标题错位
3. 误报过多
4. 同类表述命中不稳定

### 2. 当前维护动作

每次维护，优先只做以下三类动作：

1. 补充或收紧 `hit_definition`
2. 补充 `focus_terms`、`positive_examples`、`negative_examples`
3. 调整 `review_focus`、`rule_name`、`risk_level`、`execution_level`

默认不优先做的事情包括：

1. 设计复杂 DSL
2. 引入可视化规则编排
3. 让运营人员手工逐条配置
4. 同时大规模扩充几十条新规则

### 3. 当前维护节奏

建议按“样例复盘驱动”的方式维护：

1. 跑真实样例
2. 记录漏报、误报、错位
3. 将问题归类为提示词、规则包、切分、结果层问题
4. 若属于规则包问题，更新默认规则包
5. 用同一样例回归验证
6. 再决定是否进入下一轮主题

### 4. 当前维护责任

当前默认责任分工建议如下：

1. 总负责人：决定是否放行新的规则包优化主题
2. 技术负责人 T：实现规则包改动、联调、测试和回提
3. 产品经理 P：仅在需要时补业务口径，不负责日常配规则

## 版本策略

规则库必须版本化维护。

原因很简单：

1. 审查结果会直接受到规则边界变化影响
2. 没有版本就无法追踪一次精度变化到底来自哪里
3. 后续需要对不同样例结果进行回溯和比较

### 1. 版本对象

当前至少要对以下对象做版本化：

1. 默认规则包
2. 固定审查任务指令
3. 重要样例复盘结论

### 2. 版本表达

当前默认规则包已采用显式版本字段，例如：

1. `rule_pack_name`
2. `version`
3. 每条规则自己的 `version`

后续建议继续沿用语义化的小版本升级方式，例如：

1. `1.0.0`：首版可运行规则包
2. `1.1.0`：增强重点规则命中定义与正反例
3. `1.2.0`：扩充部分规则边界或新增高价值规则

### 3. 什么情况下升级版本

出现以下情况时，应升级规则包版本：

1. 新增规则
2. 删除规则
3. 明显改变某条规则的命中边界
4. 明显改变风险级别或执行级别
5. 明显改变规则关注重点，可能影响实际命中结果

### 4. 什么情况下只记回提不升版本

出现以下情况时，可以只记回提文档，不单独升规则包版本：

1. 仅补文档说明
2. 仅调整不影响运行的文字描述
3. 仅修复与规则语义无关的展示文案

## 后续维护路径

后续规则库建议分三层逐步推进。

### 1. 第一层：V1 默认规则包持续打磨

这是最近阶段的主线。

目标是：

1. 让默认规则包足够支撑真实文件验证
2. 让高价值规则召回更稳定
3. 让误报、错位和重复逐步下降

### 2. 第二层：专项规则扩展

当默认包稳定后，再考虑按专题扩展，例如：

1. 评分规则专题
2. 资格条件专题
3. 合同条款专题
4. 技术参数定向专题

### 3. 第三层：地方化规则扩展

后续若业务继续深入，再考虑引入：

1. 省级特色规则包
2. 市级特色规则包
3. 特定行业或专项采购规则包

但这一层不属于当前 V1 主线。

## 当前正式结论

当前对规则库的正式判断如下：

1. 规则库是 V1 `LLM` 审查链路中的主规则边界，而不是独立规则引擎
2. 规则库当前主要负责定义审查范围、收紧命中边界、稳定输出口径
3. 规则库当前维护方式应是“真实样例复盘驱动的默认规则包迭代”
4. 规则库必须版本化维护，不能在无版本前提下持续漂移
5. V1 当前追求的是“默认规则包可用”，而不是“完备规则库一次到位”
