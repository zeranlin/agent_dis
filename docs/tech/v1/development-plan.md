# V1 开发计划（首版）

## 文档目的

这份文档用于把 V1 技术方案转成可执行的开发顺序、阶段目标和交付检查点，帮助技术负责人组织实现和拆分任务。

## 1. 计划原则

- 先跑通主链路，再补精度和体验
- 先固定对象和状态，再补实现细节
- 每一阶段都要有可演示的增量结果
- 阶段拆分要直接对应当前验收标准

## 2. 阶段划分

V1 开发建议拆为 4 个阶段。

### 阶段 1：技术基线与资产准备

目标：

- 固定技术设计、状态机、接口和数据结构
- 准备默认规则包与固定任务指令的可加载资产

交付物：

- 技术文档首包
- 规则包资产文件
- 任务指令资产文件

完成标志：

- 技术设计已收口
- 开发成员可按文档开始编码

### 阶段 2：输入链路跑通

目标：

- 完成上传、存储、解析和结构切分

建议实现：

1. 上传接口
2. 任务创建
3. 文件存储
4. PDF / Word 解析
5. 章节与条款切分
6. 状态查询接口

完成标志：

- 上传后可得到 `task_id`
- 任务可进入 `parsed` 或明确失败态

### 阶段 3：审查链路跑通

目标：

- 完成规则加载、任务指令装配、审查执行和中间结果落地

建议实现：

1. 规则包加载器
2. 固定任务指令加载器
3. 审查编排器
4. 章节级初筛
5. 条款级审查
6. 风险点与证据落地

完成标志：

- 任务可从 `parsed` 推进到 `aggregating`
- 至少能稳定产出风险点和证据片段

### 阶段 4：结果层与页面闭环

目标：

- 生成最终结论和 Markdown 报告，并在页面完成展示与下载

建议实现：

1. 结果汇总器
2. 结论生成
3. 报告 Markdown 生成
4. 结果查询接口
5. 下载接口
6. 三态页面

完成标志：

- 结果页可查看最终结论与审查报告
- 两类 Markdown 文件可下载

## 3. 推荐任务拆分

| 任务包 | 内容 | 依赖 |
| --- | --- | --- |
| T1 | 接口骨架与任务状态模型 | 技术文档首包 |
| T2 | 文件上传与存储 | T1 |
| T3 | 文件解析与结构切分 | T2 |
| T4 | 规则包与任务指令加载 | T1 |
| T5 | 审查编排与执行 | T3、T4 |
| T6 | 风险与证据落地 | T5 |
| T7 | 结果汇总与 Markdown 导出 | T6 |
| T8 | 页面接入与结果展示 | T1、T7 |

## 4. 验证检查点

### 4.1 阶段 1 检查点

- 文档是否能支撑接口与对象定义
- 是否已明确失败态和状态流转

### 4.2 阶段 2 检查点

- 上传后是否能稳定创建任务
- 解析失败是否可见

### 4.3 阶段 3 检查点

- 审查输出是否有固定 schema
- 风险、证据、说明是否能落地

### 4.4 阶段 4 检查点

- 结果页是否只消费接口对象即可显示
- 下载文件内容是否与页面一致

## 5. 最小测试建议

V1 首版至少准备以下测试：

1. 上传接口文件格式校验测试
2. 状态机流转测试
3. 解析输出结构测试
4. 审查输出 schema 校验测试
5. 结果对象组装测试
6. Markdown 导出测试

## 6. 当前不优先做的事情

- 任务列表页
- 历史结果管理
- 在线规则编辑
- 复杂人工复核台
- 工程类采购适配

## 7. 当前结论

V1 开发应按“技术基线 -> 输入链路 -> 审查链路 -> 结果闭环”的顺序推进。这样拆分最符合当前产品边界，也最容易在每个阶段拿到清晰的可验证结果。
