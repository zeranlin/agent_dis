# V1 系统设计（首版）

## 文档目的

这份文档用于定义 V1 技术实现的最小系统设计，覆盖模块划分、主链路、边界职责和验证方式，作为开发启动的技术基线。

- `docs/business/v1/overall-solution.md`

## 2. 当前技术接手范围、约束和优先级

### 2.1 当前技术范围

技术侧当前只负责把以下主流程做成可开发闭环：

1. 接收 1 份 `PDF` 或 `Word`
2. 解析文件并形成章节、条款结构
3. 自动加载默认规则包和固定审查任务指令
4. 执行 AI 审查并产出风险点、证据片段、风险说明
5. 自动汇总为最终结论和审查报告
6. 提供在线查看和 Markdown 下载能力

### 2.2 当前硬约束

- 不扩展到多文件联审
- 不扩展到工程类采购
- 不引入在线规则运营后台
- 不纳入复杂审批流或结果流转
- 不依赖外部联网搜索或实时法规检索
- 必须适配当前单并发约 30 tokens/秒的 LLM 运行前提

### 2.3 当前优先级

按实现优先级排序如下：

1. 主流程跑通
2. 状态流转清晰可观测
3. 规则包和固定任务指令可加载
4. 结果对象稳定、可下载、可回溯
5. 在上述基础上再补异常处理、测试和后续扩展点

## 3. 设计目标

V1 技术设计只服务于当前已确认的产品闭环：

1. 上传 1 份 `PDF` 或 `Word`
2. 自动触发审查
3. 生成最终结论与审查报告
4. 在线查看并下载结果

因此系统设计优先保证：

- 主链路可跑通
- 状态可观测
- 结果可追溯
- 后续可逐步替换内部实现，而不打破外部对象

## 4. 设计原则

- 不预设复杂工作台、多租户或审批流
- 不引入重型自治 Agent 编排框架
- 优先固定输入、固定规则包、固定输出结构
- 优先使用异步任务链路隔离上传请求与审查执行
- 所有核心对象都要能回溯到 `project -> document -> clause -> risk -> evidence -> conclusion`

## 5. 最小模块划分

V1 先划分为 6 个模块。

### 3.1 接入层

负责：

- 接收上传文件
- 创建审查任务
- 提供状态查询和结果查询接口

输入：

- 原始文件

输出：

- 任务标识
- 初始状态对象

### 3.2 文件处理模块

负责：

- 文件格式校验
- 原始文件存储
- 文本解析
- 章节与条款切分

输入：

- 原始文件

输出：

- 文件对象
- 章节对象
- 条款对象

当前说明：

- 当前代码实现中的解析能力仅为“最小文本解码占位实现”
- 该实现用于先跑通输入链路，不等同于真实 `PDF` / `Word` 文件提取能力

### 3.3 审查编排模块

负责：

- 自动触发审查
- 装配审查输入
- 串联章节级初筛、条款级审查和结果汇总
- 驱动状态流转

输入：

- 已解析文件
- 规则包
- 固定任务指令

输出：

- 中间审查任务
- 审查状态更新

### 3.4 规则与指令资产模块

负责：

- 加载默认规则包
- 加载固定审查任务指令
- 对审查执行暴露稳定输入结构

输入：

- 规则配置文件
- 指令模板文件

输出：

- 可直接传给审查执行器的约束对象

### 3.5 审查执行模块

负责：

- 调用 LLM 执行章节级与条款级审查
- 输出结构化风险点、证据片段和说明

输入：

- 条款文本
- 规则约束
- 输出 schema

输出：

- 风险点对象
- 证据片段对象
- 风险说明

当前说明：

- 当前代码已先落“最小可跑通的执行骨架”
- 当前骨架使用少量规则关键词命中生成中间结果
- 当前实现用于打通状态推进与对象落地，不等同于正式 LLM 语义审查能力

### 3.6 结果汇总模块

负责：

- 风险归并
- 最终结论生成
- 报告 Markdown 生成
- 下载文件生成

输入：

- 风险点
- 证据片段
- 规则元数据

输出：

- 审查结果对象
- 最终结论文件
- 审查报告文件

## 6. 系统上下游边界

### 4.1 当前上游

- 审核人员上传单份招标文件

### 4.2 当前系统内部

- 文件接入与解析
- 审查任务编排
- LLM 审查执行
- 结果汇总与展示

### 4.3 当前下游

- 结果页在线查看
- Markdown 下载

V1 暂不接入：

- 外部审批系统
- 外部规则运营平台
- 外部任务调度平台

## 7. 主链路设计

```mermaid
graph LR

    A["上传文件"] --> B["创建审查任务"]
    B --> C["文件存储与格式校验"]
    C --> D["文本解析与结构切分"]
    D --> E["加载规则包与任务指令"]
    E --> F["章节级初筛"]
    F --> G["条款级审查"]
    G --> H["风险与证据落地"]
    H --> I["生成最终结论与报告"]
    I --> J["结果查询与下载"]
```

## 8. 主流程到模块的映射

| 产品主流程 | 技术主责任模块 | 关键产物 |
| --- | --- | --- |
| 文件进入系统 | 接入层、文件处理模块 | `task_id`、文件对象、章节、条款 |
| AI 发起审查 | 审查编排模块、规则与指令资产模块、审查执行模块 | 风险点、证据片段、风险说明 |
| 生成审查结果 | 结果汇总模块 | 最终结论、审查报告、结果对象 |
| 人工查看结果 | 接入层 | 状态查询接口、结果查询接口、下载接口 |

这个映射的目的，是保证主流程技术设计先收口，再进入各一级节点的细化设计。

## 9. 推荐部署形态

V1 推荐采用单体应用加异步任务执行器的形态。

### 6.1 单体应用

负责：

- 上传接口
- 状态查询接口
- 结果查询接口
- 页面服务

### 6.2 异步任务执行器

负责：

- 文件解析
- 审查调用
- 结果汇总

### 6.3 持久化存储

负责：

- 文件元数据
- 审查任务状态
- 风险点与证据
- 审查结论

### 6.4 对象存储

负责：

- 原始文件
- 导出的 Markdown 结果文件

这样做的原因是：

- 上传请求可快速返回
- 审查链路耗时长，适合异步执行
- 状态查询和结果查询可以保持简单稳定

## 10. 开发前置条件

在进入实现阶段前，至少应满足以下前置条件：

1. 默认规则包有可加载资产格式
2. 固定审查任务指令有版本化存放位置
3. 任务状态对象和结果对象字段已固定
4. 文件解析与结构切分的失败策略已明确
5. 开发和验证都以 `make check` 作为仓库基线检查入口

## 11. 模块职责边界

| 模块 | 必须负责 | 不负责 |
| --- | --- | --- |
| 接入层 | 上传、查询、对象返回 | 审查逻辑判断 |
| 文件处理模块 | 解析与切分 | 风险判定 |
| 审查编排模块 | 组织流程、推进状态 | 直接渲染页面 |
| 规则与指令资产模块 | 提供稳定规则输入 | 在线运营配置 |
| 审查执行模块 | 识别风险与证据 | 直接生成最终页面对象 |
| 结果汇总模块 | 生成结论与报告 | 文件解析 |

## 12. 当前不做的技术内容

为了避免范围漂移，V1 当前明确不做以下技术内容：

- 多文件任务编排
- 在线人工复核工作台
- 动态规则编辑器
- 审批流集成
- 外部法规检索增强
- 复杂多 Agent 自治协作

如果后续需要引入这些能力，应先由总负责人确认范围变化，再补技术设计。

## 13. 失败处理原则

- 上传失败直接返回，不创建成功任务
- 解析失败任务进入失败态，并记录失败原因
- 审查执行失败允许有限重试
- 汇总失败保持中间产物可追溯，并进入失败态
- 任一步进入失败态后，对外仍能通过任务接口看到明确状态和消息

## 14. 验证方式

这份设计通过以下方式验证：

1. 是否完整覆盖 `上传 -> 审查 -> 结果 -> 查看下载` 主链路
2. 是否与 `docs/business/v1/` 已确认范围一致
3. 是否能支撑 `state-machine.md`、`api-spec.md` 和 `data-schema.md` 继续细化
4. 是否满足 `docs/runbooks/handoff-t-to-owner.md` 中对模块、对象、状态、风险和开发顺序的交接要求

## 15. 当前结论

V1 适合采用“单体应用 + 异步审查执行器 + 持久化存储 + 对象存储”的最小闭环架构。这样既能尽快跑通主流程，也保留了后续把解析、审查和汇总拆成独立能力的空间。
