## 2. 本轮双小包边界

本轮只合并同一主题下的 2 个小包，不跨主题混包。

### 2.1 小包 A：下载对象字段口径统一

目标：

- 让结果接口中的 `downloadable_files` 与业务文档样例口径进一步对齐

本包只处理：

- 下载对象 `type` 统一为 `markdown`
- 下载对象补齐稳定 `file_key`
- 下载对象补齐按钮标签和用途说明

本包不处理：

- 新增下载格式
- 新增导出文件类型
- 下载权限与鉴权能力

### 2.2 小包 B：结果页消费结构化下载对象

目标：

- 让结果页“快速操作”区不再只消费原始链接名，而是直接消费结构化下载对象

本包只处理：

- 下载按钮标签展示
- 文件名称展示
- 文件类型展示
- 下载用途说明展示

本包不处理：

- 复杂下载中心
- 批量下载
- 新的页面交互编排

## 3. 本轮实现点

### 3.1 下载对象由统一 helper 生成

当前 `downloadable_files` 由统一结果展示 helper 生成，不再在结果对象里手写两套不同口径。

这样后续结果接口和结果页消费下载对象时，可以复用同一份结构定义，减少字段漂移。

### 3.2 下载对象口径向业务样例收齐

当前下载对象统一补齐：

- `type = markdown`
- `file_key`
- `label`
- `description`
- `url`

其中 `type = markdown` 直接向 `docs/business/v1/page-and-result-spec.md` 的下载对象样例收齐。

### 3.3 结果页快速操作区补齐最小交付信息

当前结果页“快速操作”区会直接展示每个下载对象的：

- 下载按钮
- 文件名称
- 文件类型
- 文件用途说明

这样审核人员在页面上可以更快判断：

- 先下载哪个文件
- 两个下载对象分别承载什么内容

## 4. 最小测试

本轮补齐以下验证：

1. 结果接口的 `downloadable_files[0].type` 为 `markdown`
2. 结果接口的下载对象包含 `file_key`
3. 结果接口的下载对象包含 `label`
4. 结果接口的下载对象包含 `description`
5. 结果页完成态包含“下载最终结论”
6. 结果页完成态包含“下载审查报告”
7. 结果页完成态包含“文件类型：markdown”

## 5. 风险与边界

本轮风险可控，原因如下：

- 只调整结果对象与结果页消费口径
- 不改解析主链路
- 不改任务状态流转

本轮明确不做：

本轮继续沿着解析链路深化主线推进，但聚焦点进一步收口到“下载对象和结果页交付面的最小一致性”。
