# V1 实现准备第三十一轮（首版）

## 文档目的

这份文档用于承接“回到唯一主线缺口，补齐上传页到结果页的页面闭环”的最新指令，在保持 V1 极简边界的前提下，完成上传页 -> 等待页 -> 结果页的最小用户页面闭环。

## 1. 产品追溯

本轮实现直接追溯以下已确认文档：

- `docs/status/summary.md`
- `docs/tasks/t-now.md`
- `docs/business/v1/main-flow.md`
- `docs/business/v1/page-and-result-spec.md`
- `docs/business/v1/acceptance-criteria.md`
- `docs/tech/v1/implementation-prep-round-27.md`

## 2. 本轮主题边界

本轮只补页面闭环，不扩其他能力。

本轮处理范围：

1. 补最小上传页
2. 补上传后的等待页
3. 让等待页自然衔接现有结果页
4. 保持结果页继续沿用当前结果读取和下载链路

本轮明确不做：

1. 多文件上传
2. 复杂工作台
3. 页面任务列表
4. 新解析能力
5. 新规则能力

## 3. 本轮实现点

### 3.1 上传页落地

当前新增上传页入口：

- `/`
- `/upload`

页面提供：

1. 页面标题
2. 文件选择控件
3. 支持格式说明
4. 开始审核按钮

上传页使用表单直接提交到：

- `/review-tasks/upload`

这样审核人员无需手工调接口，即可直接开始审查。

### 3.2 等待页落地

当前新增等待页入口：

- `/review-tasks/{task_id}/waiting`

等待页提供：

1. 等待审查状态提示
2. 基础文件信息
3. 状态接口查看入口
4. 结果页入口

上传成功后，页面会先跳转到等待页，再由等待页继续承接后续状态变化。

### 3.3 等待页到结果页衔接

当前等待页采用以下最小策略：

1. 首次进入等待页时先展示“等待审查”状态
2. 后续请求会推动现有最小 worker 链路继续执行
3. 当任务进入 `completed` 或 `failed` 时，等待页自动跳转到已有结果页

这样可以保持：

- 上传页 -> 等待页 -> 结果页

这一条页面版用户主链路真正闭合。

### 3.4 结果页链路保持不动

本轮没有重写结果页，而是继续沿用已有：

1. 结果页 payload 组装
2. 结果接口
3. 下载接口
4. `completed` / `reviewing` / `failed` / `not_found` 状态反馈

这样可以把改动控制在“补入口和衔接”，不破坏已通过复审的结果消费层。

## 4. 最小测试

本轮补齐并通过以下验证：

1. 上传页可正常渲染表单
2. 上传表单提交后会跳转到等待页
3. 等待页可正常展示等待反馈
4. 等待页可在任务完成后跳转到结果页
5. 原有结果页、下载接口和状态页面测试继续通过

## 5. 风险与边界

本轮风险可控，原因如下：

1. 只新增页面入口和等待页衔接
2. 不改审查规则和结果对象结构
3. 不改下载对象和结果页主展示逻辑

本轮边界仍然明确：

1. 当前只是最小页面闭环
2. 不是任务管理平台
3. 不是复杂前端工作台
4. 不是多角色后台

本轮完成后：

2. 上传后可以明确看到等待审查状态
3. 审查完成后可以自然进入结果页
4. 当前前台页面版 V1 最小闭环已补齐
