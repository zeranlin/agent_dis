# business 导向的总体技术约束

## 文档目的

这份文档用于明确一个前提：

- `LLM V1` 的后续技术实现，不能只以“跑通”“提速”或“调用成功”为目标，而必须服从 `business` 目录下已经确认的业务目标和审查场景。

这份文档用于统一约束后续技术实现、架构收口、解析策略、`LLM` 执行策略和验收口径。

## 一、约束来源

当前总体技术约束，直接追溯以下业务文档：

1. `docs/business/government-procurement-compliance-overview.md`
2. `docs/business/government-procurement-compliance-ontology.md`
3. `docs/business/government-procurement-review-framework.md`

后续若技术方案与上述文档冲突，应优先回到业务文档重新对齐，而不是在技术侧单独改写业务目标。

## 二、当前业务本质

从上述业务文档出发，当前产品本质不是：

1. 通用文档理解系统
2. 法规问答系统
3. 全文摘要系统
4. 大模型自由审阅器

当前产品本质是：

- 面向政府采购招标文件审核场景的风险识别系统

其核心目标不是替代审核人员作出最终裁定，而是帮助审核人员：

1. 快速找到风险点
2. 理解风险依据
3. 定位风险证据
4. 形成可复核、可留痕、可输出的审查结果

## 三、总体技术目标必须服从的四个业务要求

后续技术方案必须同时满足以下 4 个要求。

### 1. 要精准找到风险点

系统不能只给出模糊结论，而必须尽量输出：

1. 具体风险点
2. 命中的规则域或规则编号
3. 证据位置
4. 风险说明

因此，技术方案必须优先支持精准命中，而不是优先支持自由生成。

### 2. 要适应真实运行环境

当前真实运行环境已经明确：

1. 隔离环境
2. 本地 `LLM`
3. 单并发约 `30 tokens/s`
4. 最大并发约 `4`

因此，技术方案不能建立在“高吞吐云端大模型”假设上，而必须默认：

1. 单次请求成本敏感
2. 单次请求时延敏感
3. 大上下文调用风险高
4. 整单重同步执行不稳定

### 3. 要在较短时间内找出风险点

这个要求意味着：

1. 系统不必先把全文所有内容都做最高精度处理
2. 应优先找到高价值风险
3. 应优先让审核人员尽快看到值得复核的问题

因此，后续实现必须优先考虑：

1. 高风险模块优先
2. 高价值规则优先
3. 小批次执行优先

### 4. 要符合真实审核场景

“政府采购招标文件审核”不是泛化阅读任务，而是规则约束下的文档风险识别任务。

这意味着：

1. 输出对象必须是风险点，而不是泛化摘要
2. 风险必须能回溯到条款、表格行或评分项
3. 审核人员最终看到的应该是可复核证据，而不是模型自由发挥的解释

## 四、总体技术约束

基于上述业务要求，后续技术方案必须遵守以下总体约束。

### 1. 技术主链路必须服从“风险识别”目标

后续技术实现的核心问题，不是“全文理解得有多像人”，而是：

1. 风险点找得是否足够准
2. 证据定位是否清楚
3. 结果是否便于审核人员复核

如果某项实现更复杂，但让风险识别更慢、更碎、更难复核，则不应优先采用。

### 2. 解析层必须服从“业务结构”而不是“技术长度”

从业务文档看，招标文件是一个结构高度稳定的业务对象，不是普通长文。

因此，解析层必须优先识别：

1. 资格条件
2. 采购需求
3. 评分办法
4. 合同条款
5. 程序条款
6. 政策条款

技术上可以继续做轻量切分，但前提必须是：

1. 先有业务模块
2. 先有业务单元
3. 再在必要时做子片段切分

### 3. `LLM` 必须服从“执行层收缩”原则

后续实现中，`LLM` 不是总架构，而是执行层组件。

`LLM` 的职责应收口为：

1. 在规则约束下判断业务单元是否存在风险
2. 输出结构化风险说明
3. 输出证据引用

`LLM` 不应承担：

1. 全文级自由审阅
2. 报告拼装主逻辑
3. 页面摘要主逻辑
4. 复杂全文级综合组织

### 4. 代码侧必须承担稳定汇总职责

能交给代码做的事情，应尽量交给代码完成。

至少以下内容应优先由代码承担：

1. 批次调度
2. 风险归并
3. 风险组汇总
4. 总体结论生成
5. Markdown 报告拼装
6. 页面结果对象组装

### 5. 运行时必须优先“小请求、小批次、可恢复”

在当前本地弱算力环境下，后续技术实现必须优先满足：

1. 单次请求负载可控
2. 批次可以缩小
3. 批次失败可重试
4. 不因一次调用失败拖垮整单

因此，任何重新引入大上下文、整单重审或超长同步请求的方案，都应被视为高风险方案。

## 五、后续实现的优先级原则

后续主线应优先按以下顺序推进。

### 1. 第一优先：保证当前执行链路可验证

先完成：

1. `LLM` 收缩执行
2. `completed` 验证
3. 结果页与 Markdown 下载真实可用验证

### 2. 第二优先：让解析更贴近业务对象

在执行链路基本稳定后，再推进：

1. 资格条件业务化解析
2. 参数项业务化解析
3. 评分项业务化解析
4. 合同条款业务化解析

### 3. 第三优先：再讨论更高阶精度优化

只有在前两步都成立后，才适合继续推进：

1. 更深层规则扩展
2. 更复杂的跨章节逻辑
3. 地方规则扩展

## 六、当前不允许作为优先路线的做法

后续若出现以下倾向，应视为偏离业务目标：

1. 只追求“能跑通”而不关心是否符合真实审核对象
2. 只追求“调用成功”而不关心风险点是否精准
3. 只追求“切得更细”而不关心是否符合招标文件天然结构
4. 把更多汇总、组织、表达职责继续压给 `LLM`
5. 在未完成 `completed` 验证前就重新分散到更多实现主题

## 七、验收时必须回答的问题

后续任何关键技术改动，在验收时都至少要回答以下问题：

1. 这次改动是否更贴近真实政府采购招标文件审核场景
2. 这次改动是否更有利于精准识别高风险条款
3. 这次改动是否更适应当前本地弱算力环境
4. 这次改动是否让审核人员更快看到值得复核的风险点
5. 这次改动是否让结果更可追溯、更可解释

如果不能同时回答这些问题，就不应仅凭“代码更复杂”或“模型更智能”判断方案更优。

## 八、当前正式结论

当前正式结论如下：

1. 后续技术方案必须以 `business` 目录中的业务定义为最高约束
2. 项目目标不是“让大模型更自由地理解全文”，而是“让系统更准确、更快速地找出招标文件风险点”
3. 技术主链路必须收口为：业务化解析 + 收缩式 `LLM` 执行 + 代码侧汇总
4. 在当前环境下，“精准、较快、可复核”优先级高于“复杂、全面、自由生成”
5. 后续任何实现，如果偏离真实政府采购招标文件审核场景，都应视为方向偏差而不是技术进步
