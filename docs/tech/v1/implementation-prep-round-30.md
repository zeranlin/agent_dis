# V1 实现准备第三十轮（首版）

## 文档目的

这份文档用于承接“正式进入验收 / 交付，转向交付说明与交接材料整理”的最新指令，把当前阶段真正需要执行的支持材料一次收口为可落地清单。

本轮不再继续补实现，除非后续验收中发现新的阻塞性交付缺口。

## 1. 产品追溯

本轮交付支持直接追溯以下已确认文档：

- `docs/status/summary.md`
- `docs/tasks/t-now.md`
- `docs/tasks/current.md`
- `docs/business/v1/acceptance-criteria.md`
- `docs/tech/v1/implementation-prep-round-27.md`
- `docs/tech/v1/implementation-prep-round-29.md`

## 2. 本轮主题边界

本轮只处理以下 3 类内容：

1. 验收执行支持
2. 最终交付说明
3. 交接材料清单

本轮明确不做：

- 新增解析能力
- 新增规则能力
- 新增页面交互
- 回头补零散优化项
- 任何超出 V1 极简边界的扩展

## 3. 验收执行支持

### 3.1 验收前置动作

进入正式验收前，统一执行以下动作：

1. 执行 `make check`
2. 准备 5 类真实样例文件
3. 明确本轮验收只按 V1 边界判断
4. 以 `docs/tech/v1/implementation-prep-round-29.md` 作为最近一次联调记录基线

### 3.2 验收执行顺序

建议现场按以下顺序执行：

1. 先确认上传入口可正常提交文件
2. 再确认 worker 可把正常样例推进到 `completed`
3. 再确认失败样例能在正确阶段返回正确反馈
4. 再确认结果页与下载结果一致
5. 最后确认 `not_found` 页面兜底反馈

这样排序的目的，是先锁定主流程，再补异常路径，减少现场走查时的切换成本。

### 3.3 验收现场建议核对项

现场验收时优先核对以下内容：

1. 任务 ID 是否生成
2. 正常样例是否能看到最终结论和风险列表
3. Markdown 结论与报告是否可下载
4. 无可审查正文样例是否在解析阶段失败
5. 不支持格式样例是否在上传阶段直接拒绝
6. `reviewing`、`completed`、`failed`、`not_found` 四类页面是否反馈正确

## 4. 最终交付说明

### 4.1 当前版本做什么

当前 V1 正式交付范围统一为：

1. 单文件上传
2. `PDF` / `DOCX` / `DOC` 最小解析
3. 最小审查执行与结果汇总
4. 结果页查看
5. Markdown 结论与报告下载

### 4.2 当前版本不做什么

交付时必须明确说明以下内容不属于 V1：

1. OCR
2. 多文件联审
3. 附件联审
4. 复杂表格精细结构恢复
5. 平台化规则配置
6. 更复杂的页面工作台和交互编排

### 4.3 结果解释口径

对外说明时建议统一以下口径：

1. 当前版本重点验证“可审、可看、可下载、可追溯”
2. 当前解析能力是最小可用实现，不承诺复杂版式高精度恢复
3. 当前规则命中结果用于辅助审核，不替代最终人工判断

## 5. 交接材料清单

### 5.1 必备交接材料

当前建议交接给总负责人或验收执行同学的材料如下：

1. `docs/business/v1/acceptance-criteria.md`
2. `docs/tech/v1/implementation-prep-round-27.md`
3. `docs/tech/v1/implementation-prep-round-29.md`
4. `docs/tech/v1/implementation-prep-round-30.md`
5. `docs/tasks/current.md`
6. `docs/status/summary.md`

### 5.2 材料用途说明

各材料建议用途如下：

1. `acceptance-criteria.md`：对齐验收口径
2. `implementation-prep-round-27.md`：查看联调和验收执行清单
3. `implementation-prep-round-29.md`：查看真实样例联调记录
4. `implementation-prep-round-30.md`：查看当前交付说明和交接入口
5. `current.md`：查看当前阶段状态与下一步
6. `summary.md`：低成本快速恢复上下文

### 5.3 交接时建议直接说明的 4 件事

交接时建议直接说清以下内容：

1. 当前已经联调通过，可进入验收 / 交付
2. 当前没有阻塞性交付缺口
3. 当前只接受阻塞性问题回流到实现修改
4. 当前非范围项不应在验收现场临时拉进 V1

## 6. 异常处理口径

从本轮开始，现场若出现问题，建议先按以下口径分类：

1. 若主流程跑不通：视为阻塞性交付缺口
2. 若结果页或下载不可用：视为阻塞性交付缺口
3. 若失败分类和提示明显错误：视为阻塞性交付缺口
4. 若只是复杂文档精度一般：先记录，不直接拉回实现
5. 若只是文案仍可优化：先记录，不直接拉回实现

## 7. 本轮结论

本轮结论如下：

1. 当前已正式从“联调走查”切换到“验收执行支持与交接材料整理”
2. 当前已具备执行验收和交付所需的最小材料包
3. 当前无需回到实现修改
4. 后续只有在出现阻塞性交付缺口时，才回到代码层处理
