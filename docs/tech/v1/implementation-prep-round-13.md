# V1 实现准备第十三轮（首版）

## 文档目的

这份文档用于承接结果页交付说明与联调提示成果，继续补齐结果页最后一处页面级缺口，并给出当前线程是否可收口的技术判断。

## 1. 产品追溯

本轮实现直接追溯以下已确认文档：

- `docs/status/summary.md`
- `docs/tasks/t-now.md`
- `docs/tasks/current.md`
- `docs/business/v1/page-and-result-spec.md`
- `docs/tech/v1/implementation-prep-round-12.md`

若后续有人希望继续扩展复杂前端交互、权限控制或富文本样式，应先回提总负责人，不在本轮直接展开。

## 2. 本轮目标

本轮只处理两件事：

1. 补齐结果页不存在时的页面级反馈
2. 判断当前结果页线程是否已达到 V1 可交付门槛

## 3. 本轮范围

### 3.1 本轮要做

1. 将任务不存在时的结果页访问反馈收口为 HTML 页面
2. 补齐对应最小测试
3. 给出结果页线程收口判断

### 3.2 本轮不做

- 新增页面入口
- 增加前端框架
- 增加复杂重试、跳转或审批动作
- 扩展结果对象字段

## 4. 当前实现设计

### 4.1 页面不存在场景改为页面级反馈

当前 `GET /review-tasks/{task_id}/page` 在任务不存在时，统一返回：

- `404`
- `text/html; charset=utf-8`
- 可直接阅读的结果页不存在说明

这样可以让结果页链路在“完成、审核中、失败、任务不存在”四类常见访问结果下，全部保持页面级表达，避免同一路径下部分场景跳回 JSON。

### 4.2 页面反馈口径

当前不存在页面只承担最小告知职责，明确提示：

- 先确认任务标识是否正确
- 必要时重新提交文件
- 联调时先确认任务是否已经创建

这仍然符合 V1 极简边界，不引入新的操作按钮或额外工作流。

## 5. 收口判断

从技术实现角度，当前结果页线程已经基本满足 V1 最小可交付门槛，判断依据如下：

1. 主路径已完整覆盖上传后查看结果的最小阅读链路
2. 页面已稳定消费现有结果对象字段，没有额外扩展产品范围
3. `completed`、`reviewing`、`failed` 三类主状态页面都已有稳定反馈
4. 结果页不存在场景已补成页面级反馈，访问体验更完整
5. 已具备对应最小测试，可支撑后续联调和收口复审

因此，技术侧建议：

- 将结果页线程视为“已达到技术收口准备条件”
- 是否正式结束该线程，由总负责人最终拍板

## 6. 最小测试

本轮至少覆盖以下验证：

1. 任务不存在时访问结果页返回 `404 HTML`
2. 页面内容包含明确的缺失说明
3. 既有完成态、审核中、失败态页面能力保持不回退

## 7. 当前结论

实现准备第十三轮不是继续加页面能力，而是把结果页线程补到“边界完整、体验自然、可以判断是否收口”的状态。

完成这一轮后，当前结果页已经具备：

- 三类主状态的稳定页面反馈
- 页面不存在时的统一页面级兜底
- 面向联调和复审的最小验证依据
- 提交总负责人做最终收口判断的技术基础
