# 当前阶段整体再梳理说明

## 文档目的

这份文档用于把当前阶段已经完成的工作、真实联调暴露的问题、`LLM V1` 收缩方向和业务化解析方向，统一整理成一页总说明。

这份说明不是新的产品方案，而是对当前阶段整体判断的正式收口，回答以下问题：

1. 到目前为止项目真正完成了什么
2. 当前主矛盾到底在哪里
3. 为什么前一阶段推进时间变长
4. 接下来应如何从整体层面收口

## 一、当前阶段已完成的核心工作

从整体看，当前项目已经完成了三层关键工作。

### 1. 业务层已经基本定型

当前已经完成：

1. 政府采购招标文件合规审查业务边界收口
2. V1 产品范围收口
3. 审查对象、本体、主流程、数据模型和规则框架收口
4. 默认规则包、固定审查任务指令和报告驱动模式收口

这意味着项目已经不是一个概念讨论状态，而是有清晰产品边界的 V1。

### 2. 精度层已经完成三轮系统性收口

当前已经完成：

1. 提示词与规则包联合收口
2. 结果归并轻收口
3. 长条款细切分与重复块降权

这三轮工作的价值在于：

1. 高价值规则召回得到提升
2. 风险组口径已经收口
3. 长条款和重复块带来的明显重复命中已被压降

### 3. 真实联调已经把主问题逼出来

当前已经不再停留在“代码逻辑上可以跑”的阶段，而是进入了：

1. 真实模型联调
2. 真实招标文件联调
3. 页面闭环与报告闭环验证
4. `completed` 验收放行判断

这一阶段的最大价值，是让系统真正暴露出执行链路中的核心问题，而不是继续被假设掩盖。

## 二、当前真正暴露出的核心问题

经过这一轮推进，当前核心问题已经比较明确，不再是页面、文档或讨论方式问题，而是以下两点。

### 1. `LLM` 在执行层使用偏重

真实验收中出现的 `HTTP 504`，本质上说明：

1. 当前 `LLM` 在主流程中承担了过重的同步职责
2. 单次请求负载对整条链路影响过大
3. 一次真实调用失败会直接放大为整单无法进入 `completed`

因此，问题不在于“是否应该使用 `LLM`”，而在于：

- 当前 `LLM` 的职责和执行方式没有收得足够轻

### 2. 解析层还不够业务化

前面虽然已经做了长条款细切分和重复块降权，但进一步梳理后可以确认：

1. 当前解析仍带有较强的技术切分思维
2. 审查单元还不够贴近招标文件天然的业务结构
3. 参数表、评分表、偏离表、合同条款等高价值对象，还应更多按业务单元而不是普通文本块处理

因此，问题不在于“切分不够细”，而在于：

- 解析层还没有完全从技术切分切到业务结构切分

## 三、为什么这一阶段推进时间变长

从整体视角看，这一阶段推进时间变长，并不只是执行效率问题，而是因为系统进入了一个更深层的收口阶段。

### 1. 从“功能能跑”进入了“真实可验证”

在前一阶段，很多工作回答的是：

1. 功能有没有
2. 链路通不通
3. 页面出不出得来

而当前阶段回答的是：

1. 正常样例能不能稳定进入 `completed`
2. `LLM` 调用方式是否真的适合当前运行环境
3. 解析结果是否真的符合业务审查需要

后者天然比前者更慢，但也更真实。

### 2. 当前问题已经从功能缺口转成架构收口问题

现在已经不是简单补一个接口、补一个页面、补一个字段的问题，而是：

1. `LLM` 应该承担到什么程度
2. 解析层应该按什么结构组织
3. 哪些事情应该交给代码，哪些事情应该留给 `LLM`

这类问题虽然推进时间更长，但决定的是后续主链路能否稳定。

### 3. T 这次长时间推进，实际上逼出了更高价值的结论

这轮推进最终沉淀出来的，不只是一个稳定性修复结果，而是三条更重要的共识：

1. 不能继续把 `LLM` 当成重同步黑盒
2. 不能继续把解析主线理解成“切得更碎”
3. V1 的下一步目标不再是继续堆功能，而是尽快形成一版真正可验证的版本

## 四、当前整体架构应如何重新理解

当前更合理的整体理解，不应再是“上传文件 -> 解析成大量块 -> 扔给 `LLM` -> 再勉强汇总”，而应重构为以下四层。

### 1. 业务层

负责定义：

1. 审什么
2. 哪些对象最重要
3. 哪些规则属于 V1 范围

这一层当前已经比较稳定。

### 2. 业务化解析层

负责把招标文件先转成自然业务单元，例如：

1. 资格条件
2. 参数项
3. 评分项
4. 合同条款
5. 偏离项

这一层回答的是：

- 什么样的单元值得被审

### 3. 收缩后的 `LLM` 执行层

负责在规则约束下，对业务单元做片段级判断。

这一层回答的是：

- 这个单元有没有命中风险，为什么命中，证据在哪里

这一层不再承担：

1. 整份文件级自由总结
2. 整份报告拼装
3. 过重的同步执行责任

### 4. 代码汇总与展示层

负责：

1. 风险归并
2. 风险组汇总
3. 总体结论生成
4. Markdown 报告拼装
5. 页面对象组装

这一层回答的是：

- 怎样把已命中的结果稳定变成最终输出

一句话收口：

- 业务化解析决定“审什么单元”，收缩后的 `LLM` 决定“怎么审这些单元”，代码层决定“怎么把结果拼起来”

## 五、当前两个关键调整方向

### 1. `LLM V1` 收缩方向

这部分已经正式沉淀在：

- `docs/tech/v1/llm-v1-reduction-plan.md`

当前已明确的方向是：

1. `LLM` 收口为片段级判断器
2. 结果归并、结论生成、报告拼装由代码负责
3. 执行层优先支持轻量分批、缩批和最小重试
4. 先形成更轻、更稳的可验证版本，再继续追求更高精度

### 2. 业务化解析方向

这部分已经正式沉淀在：

- `docs/business/v1/government-procurement-business-oriented-parsing-plan.md`

当前已明确的方向是：

1. 先按招标文件天然业务结构解析
2. 再按技术需要做轻量切分
3. 审查单元优先是资格条件、参数项、评分项、合同条款等自然业务单元
4. 表格不只当作普通文本，而应尽量恢复为行级或项级业务单元

## 六、当前阶段的正式总判断

当前正式总判断如下：

1. 项目方向没有走偏，业务方案和产品边界总体成立
2. 默认规则包、三轮精度收口和最小结果闭环都已经为 V1 奠定了基础
3. 当前主要问题已经不再是“功能不够”，而是“解析方式和 `LLM` 执行方式需要进一步收口”
4. 当前真正需要调整的，不是是否继续做这个产品，而是技术主链路的组织方式
5. 当前技术主链路应从“重同步 `LLM` + 技术碎片切分”切换为“业务化解析 + 收缩式 `LLM` 执行”

## 七、下一步总方向

从整体上看，当前最合理的下一步不是继续散做，而是保持单主线推进。

当前建议顺序如下：

1. 先执行 `LLM V1` 收缩方案
2. 同步让解析策略逐步转向业务化
3. 再恢复进入 `completed` 验证
4. 以真实文件、结果页和 Markdown 下载为准重新做验收判断

## 八、当前一句话结论

当前一句话结论是：

- 项目没有走偏，但必须把技术主链路从“功能实现逻辑”收口到“业务化解析 + 收缩式 `LLM` 执行逻辑”。
