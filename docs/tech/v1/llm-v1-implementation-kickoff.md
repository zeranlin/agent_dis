# LLM V1 首轮实现启动说明

## 文档目的

这份文档用于把当前主线从“页面闭环与验收支持”正式切回到 `LLM V1` 首轮实现，并给技术负责人提供一版可以直接开工的实现范围、顺序与验收口径。

## 当前判断

当前已经具备以下基础：

- 上传页、等待页、结果页、下载链路已打通
- 文件解析、正文提取、结构切分已具备最小可用基础
- 审查输入装配已存在最小骨架
- 结果汇总、总体结论和 Markdown 输出已存在最小骨架
- `LLM` 审查主流程、输入模型、输出模型、分片策略、风险归并和结论输出已形成方案骨架

当前正式判断为：

- 不需要继续把 `8.x / 9.x` 无限下钻后才开发
- 当前应冻结到现有设计层级
- 直接进入 `LLM V1` 首轮实现

## 首轮实现目标

首轮目标只有一个：

`用真实 LLM 调用替换当前关键词占位执行层，打通 LLM V1 最小可用审查闭环。`

## 首轮范围

本轮只做以下 `4` 件事：

1. `LLM` 调用接入
2. `LLM` 审查执行链路替换
3. 结果汇总链路接轨
4. 最小端到端联调

## 1. LLM 调用接入

### 1.1 目标

- 支持通过环境变量读取模型服务地址、模型名称和鉴权信息
- 提供一个最小稳定的 `LLM` 客户端或适配层
- 先完成单次调用打通

### 1.2 约束

- 不在仓库代码或文档中写明文密钥
- 优先兼容 OpenAI 风格接口
- 优先支持超时、失败和基础错误分类

### 1.3 最小验收

- 本地能发起一次真实模型调用
- 能拿到有效响应
- 失败时能给出清晰错误而不是静默失败

## 2. LLM 审查执行链路替换

### 2.1 目标

- 基于当前结构切分结果构造真实 `LLM` 审查输入
- 用真实 `LLM` 输出替换当前关键词命中执行器
- 输出最小结构化风险项

### 2.2 当前建议

- 先不要追求一次把输出结构做到最复杂
- 先保证最小结构化字段稳定
- 当前建议最小输出字段至少包括：
  - `risk_title`
  - `risk_level`
  - `risk_category`
  - `rule_code`
  - `evidence_text`
  - `review_reasoning`
  - `need_human_confirm`

### 2.3 最小验收

- 结构切分后的片段能够被送入 `LLM`
- `LLM` 能返回最小结构化风险结果
- 当前执行链路不再依赖关键词占位命中作为主路径

## 3. 结果汇总链路接轨

### 3.1 目标

- 把 `LLM` 输出接入现有结果汇总骨架
- 继续复用现有结果页和 Markdown 下载链路
- 让总体结论和报告仍然能正常生成

### 3.2 当前建议

- 当前先不追求完整实现全部复杂归并策略
- 可先保留最小归并或轻归并
- 但结果对象必须稳定，不影响页面和下载链路

### 3.3 最小验收

- `LLM` 输出能落到最终结果对象
- 结果页可以展示
- Markdown 可以下载

## 4. 最小端到端联调

### 4.1 目标

- 上传一个真实文件
- 完成解析、切分、`LLM` 审查、结果汇总、结果展示
- 至少验证一个成功样例

### 4.2 最小验收

- 上传后任务能推进到完成态
- 结果页可查看
- 下载文件可生成
- 当前链路确实走了真实 `LLM` 调用

## 本轮不做什么

- 不继续扩页面范围
- 不继续扩多文件能力
- 不继续扩复杂运营能力
- 不继续深挖 `8.x / 9.x` 更下一层设计
- 不追求一次做完全部精细归并和高级评估

## 建议实施顺序

建议按以下顺序推进：

1. 先打通 `LLM` 连通性
2. 再替换审查执行器主路径
3. 再接到现有结果汇总和页面链路
4. 最后做最小端到端联调

## 当前验收口径

本轮只按以下两类结论回提：

- 通过：`LLM V1` 最小可用闭环已打通，可进入下一轮精度和结构优化
- 不通过：存在阻塞主线的实现缺口，需要先回到实现修正

## 当前建议

当前建议技术负责人按“单主题整包回提”方式推进，不再回到过细的小包节奏。
