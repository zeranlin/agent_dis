# `SZCG2025000300-A` 风险候选收口第一轮

## 文档目的

本文件记录在当前唯一主题“围绕 `SZCG2025000300-A` 打磨可用风险审查结果”下，本轮针对“解析 -> 审查执行”之间候选送审链路所做的最小收口。

本轮只解决一个问题：

- 让已经识别出的业务单元入口，真正影响高价值规则的候选片段排序

本轮明确不做：

- 不扩页面
- 不扩复杂策略引擎
- 不回头拉长提示词与规则包主题
- 不做规则硬过滤
- 不做新的大范围 `LLM` 联调

## 约束来源

本轮实现直接追溯以下已确认文档：

1. `docs/comms/inbox-t.md`
2. `docs/business/v1/government-procurement-business-oriented-parsing-plan.md`
3. `docs/business/government-procurement-review-framework.md`
4. `docs/tech/v1/business-driven-technical-constraints.md`

对应关系如下：

- 业务文档要求“先按业务结构解析，再按技术需要轻量切分”
- 技术约束要求“高风险模块优先、高价值规则优先、小请求优先”
- 当前唯一主题要求“先把 `SZCG2025000300-A` 真正有价值的风险片段送进现有审查链路”

## 本轮问题判断

在本轮改动前，现有系统已经具备：

1. 业务模块识别
2. 业务单元识别
3. 规则级候选片段预筛
4. 业务模块优先筛选

但仍存在一个收口缺口：

- 候选排序虽然已经知道 `module_type` 和 `unit_label`，但在规则级排序里还没有充分利用这些信息

这会导致两类问题：

1. 某些高价值规则虽然已经能拿到正确模块，但排在前面的仍可能是过泛片段
2. `LLM` 本轮即使不滥用，也会被更多“能解释但不够关键”的片段占用预算

## 本轮实现

本轮只在 `app/review_executor.py` 做了一个最小增强：

### 1. 为高价值规则补“业务单元优先”软提示

新增 `RULE_CODE_UNIT_LABEL_HINTS`，但只作为排序加分使用，不作为硬过滤。

当前覆盖规则包括：

- `R1` 到 `R12`

其中重点加强以下路径：

- `R3` 优先看 `单条资格要求 / 单条资质要求 / 单条业绩要求 / 商务要求项`
- `R5` 优先看 `单条技术参数 / 单个参数项 / 单条功能要求 / 单个偏离项`
- `R9` / `R10` / `R11` 优先看 `单个评分项 / 价格分规则 / 商务分规则 / 技术分规则`
- `R12` 优先看 `付款条款 / 验收条款 / 违约责任条款 / 质保条款 / 费用承担条款 / 商务要求项`

### 2. 对过泛单元做轻量降权

对以下标签做轻量降权，而不是直接剔除：

- `普通条款`
- `普通表格行`
- `不确定审查对象`

这意味着：

1. 泛条款仍然可以进候选池
2. 但在存在更明确业务单元时，会自然让位给更适合当前规则的片段

### 3. 不做规则硬映射

本轮明确没有采用：

- “某类单元只能命中某些规则”的硬过滤

这样做是为了保持当前主题聚焦：

- 先提升真实样例的高价值候选送审质量
- 不把本轮扩成新的“动态规则映射”主题

## 自动化验证

本轮新增并通过了以下验证：

1. 规则候选仍能保持原有“按业务模块优先”能力
2. 在同模块内，具体业务单元会优先于泛条款进入候选前列
3. 高价值章节与关键词命中优先级未被破坏

## 使用 `SZCG2025000300-A` 的离线观察

本轮按照“少用 `LLM`”约束，没有新增完整真实审查联调，而是对真实样例做了解析后候选池观察。

观察结果如下：

### `R3 资格条件过高检查`

候选前列已稳定出现：

- `商务要求项 / 免费保修期`
- `商务要求项 / 培训方案`
- 含 `原厂保修`、`厂家升级`、`培训资源` 的真实片段

这说明：

- 样例中“原厂保修 / 培训资源偏严”已经进入高优先候选区

### `R5 品牌/型号指向检查`

候选前列已稳定出现：

- `提供与影像处理平台同品牌内窥镜主机`
- `具备与内窥镜控制单元同品牌 30 度内窥镜`
- `与内窥镜控制单元同品牌`

这说明：

- 样例中最关键的“同品牌约束”片段已经稳定进入候选前列

### `R9 评分项未量化检查`

候选池已进一步收口到：

- `单个评分项`
- `资格性审查表`

当前仍可见一些“综合评分法”类泛评分条款，但已不再主要落在普通段落和杂项片段上。

### `R12 关键条款缺失/责任失衡检查`

候选前列已稳定出现：

- `商务要求项 / 检测验证`
- `商务要求项 / 免费保修期`
- `商务要求项 / 违约责任`

这说明：

- 样例中“第三方检测费用承担”“合同责任与履约约束”已经进入更合适的送审入口

## 本轮结论

本轮正式结论如下：

1. 当前已把“业务单元入口”真正接入规则候选排序，而不只是停留在解析输出字段
2. 当前高价值规则更容易拿到与业务语义一致的候选片段
3. 当前改动保持了 `V1` 极简边界，没有扩流程、页面或复杂策略
4. 当前未发现新的阻塞性交付缺口

## 下一步建议

若继续围绕当前唯一主题推进，建议顺序如下：

1. 在现有收缩配置下，仅对 `SZCG2025000300-A` 做一次受控真实 `LLM` 复跑
2. 重点核对最终结果里是否已更稳定出现 `R5` 同品牌约束、`R3` 原厂保修 / 培训资源偏严、`R12` 检测费用承担 / 合同责任失衡
3. 若结果已基本可读，则进入结果复核与报告口径收口
4. 若仍明显漏报，再继续做同主题内的候选排序或输入裁剪微调

本轮不建议直接跳回：

- 复杂策略
- 大规模提示词扩张
- 新的页面主题
