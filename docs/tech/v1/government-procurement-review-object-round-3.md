# 政府采购审查对象解析扩展第三轮

## 1. 本轮目标

本轮继续按 `docs/comms/inbox-t.md` 当前唯一有效消息推进“政府采购审查对象解析扩展”。

本轮重点不是继续细化既有四类高价值模块，而是把总负责人明确要求的其余审查对象入口补进现有解析链路，包括：

1. 资格性审查表
2. 符合性审查表
3. 商务要求项
4. 政策适用条款
5. 不确定区补判

## 2. 本轮实现

### 2.1 新增五类对象入口标签

当前在现有 `unit_label` 基础上新增以下解析入口：

1. `资格性审查表`
2. `符合性审查表`
3. `商务要求项`
4. `政策适用条款`
5. `不确定审查对象`

这一步的定位是：

1. 先让现有解析结果更像真实政府采购审查对象
2. 避免把所有未稳定识别内容继续硬塞到 `普通条款`、`普通表格行` 或错误模块
3. 为后续继续压缩边界误判留出更明确的观察口径

### 2.2 补“近邻上下文”透传

为避免表格行只看本行文本而丢失前置标题，本轮给业务单元分类补了最小“近邻上下文”透传。

当前做法是：

1. 解析时把同章节最近两条文本行作为 `context_text`
2. 分类时把 `chapter_title + block_title + context_text + 当前文本` 合并判断

这一步的收益主要体现在：

1. `资格性审查表`
2. `符合性审查表`
3. 部分 `商务要求` / `政策` / `偏离表`

这类“本行标题弱、上一行标题强”的对象上。

### 2.3 不确定区先显式落标签

当前对以下内容先落到 `不确定审查对象`：

1. `默认章节` 下的内容
2. `特别警示条款`
3. `政府采购违法行为风险知悉确认书`

这样做的原因是：

1. 这部分内容当前确实还没稳定归到更准确对象
2. 显式标成“不确定审查对象”比误标为“采购需求参数项”更安全
3. 便于后续继续围绕真实样例做边界收口

## 3. 自动化验证

本轮已补并通过以下验证：

1. `资格性审查表` 可进入稳定标签
2. `符合性审查表` 可进入稳定标签
3. `商务要求项` 可进入稳定标签
4. `政策适用条款` 可进入稳定标签
5. `默认章节` 内容会先落到 `不确定审查对象`

## 4. 真实样例观察

使用 `SZCG2025000300-A` 做本轮快速校准时，当前已出现以下标签分布：

1. `资格性审查表`：`5`
2. `符合性审查表`：`1`
3. `商务要求项`：`23`
4. `政策适用条款`：`36`
5. `不确定审查对象`：`56`

这说明当前八类审查对象入口已经开始在真实样例中落地，不再只停留在测试夹具。

## 5. 当前问题

本轮也进一步暴露出当前未收口问题：

1. `资格性审查表` 的首个样本仍可能吃到“招标文件信息”上下文
2. `符合性审查表` 当前在真实样例中的稳定样本仍偏少
3. `商务要求项` 与 `评分办法` / `偏离项` 的边界仍有交叉
4. `政策适用条款` 与 `程序条款` 的边界仍未完全收稳
5. `不确定审查对象` 数量仍偏多，说明章节识别与前置对象识别仍需增强

## 6. 当前结论

本轮正式结论如下：

1. 当前已经把总负责人要求的八类政府采购审查对象入口基本补进现有解析链路
2. 当前仍属于 V1 最小扩展，不是完整对象体系重构
3. 当前下一步应继续围绕真实样例压缩边界误判，而不是回头扩页面或扩大 `LLM` 用量

## 7. 下一步建议

下一步优先级建议如下：

1. 继续提升章节识别，优先减少 `默认章节`
2. 继续收口 `资格性审查表 / 符合性审查表` 的前置标题污染问题
3. 继续收口 `商务要求项 / 评分办法 / 偏离项` 之间的边界误判
