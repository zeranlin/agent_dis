# 业务化解析执行首包

## 1. 本轮目标

本轮用于落实 `docs/business/v1/government-procurement-business-oriented-parsing-plan.md` 的最小可执行版本。

目标不是一次性完成完整解析体系，而是先把当前主链路从“纯技术切分片段”推进到“带业务语义的审查单元”。

## 2. 本轮实现范围

本轮优先落以下三件事：

1. 让现有解析结果显式带上业务模块
2. 让现有解析结果显式带上业务单元类型
3. 让表格行不再只作为普通连续文本进入后续审查

## 3. 本轮实现

### 3.1 解析对象增加业务字段

当前在现有 `ClauseRecord` 上补充以下字段：

1. `module_type`
2. `unit_type`
3. `parent_unit_id`

这一步的作用是先把现有 `clause` 从“技术片段对象”提升为“可过渡承载业务审查单元的对象”。

### 3.2 先落模块级业务分类

当前解析阶段已开始优先识别以下业务模块：

1. 项目基础信息
2. 资格条件
3. 采购需求
4. 评分办法
5. 合同条款
6. 程序条款
7. 政策条款
8. 其他

当前分类策略不是简单全文关键词命中，而是：

1. 章节标题优先
2. 块标题次之
3. 正文文本辅助

这样可以避免正文偶发关键词把整条业务单元误分到错误模块。

### 3.3 先落单元级业务分类

当前解析阶段已开始识别以下业务单元类型：

1. 条款
2. 表格行
3. 评分项
4. 参数项
5. 合同项

当前规则如下：

1. 评分办法模块中的表格行优先归为 `评分项`
2. 采购需求模块中的表格行或需求条款优先归为 `参数项`
3. 合同条款模块中的条款优先归为 `合同项`
4. 含“偏离 / 偏差 / 响应情况”等特征的内容优先归为 `偏离项`
5. 其他模块保留为更一般的 `条款` 或 `表格行`

### 3.4 表格行开始进入业务单元

当前 `DOCX` 表格提取后，不再只被理解为一段普通文本。

本轮已让表格行在解析阶段进入：

1. 对应业务模块
2. 对应业务单元类型

这意味着：

1. 采购需求章节中的表格行可进入 `参数项`
2. 评分办法章节中的表格行可进入 `评分项`
3. 偏离表中的表格行可进入 `偏离项`

### 3.5 子片段开始保留父单元归属

当前长条款进入轻量子片段切分后，已开始为同源子片段补 `parent_unit_id`。

这一步的作用是：

1. 保留子片段与原业务单元的归属关系
2. 为后续独立拆出 `review_unit` / `unit_fragment` 铺垫
3. 避免后续结果汇总和证据回溯时丢失“同一原条款”的关系

## 4. 与原业务方案的关系

这次实现还不是 `module_block -> review_unit -> unit_fragment` 完整三级对象。

当前更准确的定位是：

1. 已先把 `review_unit` 的核心业务字段折叠进现有 `clause`
2. 已为后续真正拆出 `module_block` 和独立 `review_unit` 做字段铺垫
3. 已让“业务结构优先”开始进入真实运行链路，而不再只停留在文档层

## 5. 自动化验证

本轮已补并通过以下验证：

1. 资格条件正文条款会被归到 `资格条件 / 条款`
2. 采购需求章节中的表格行可归到 `采购需求 / 参数项`
3. 评分办法章节中的表格行可归到 `评分办法 / 评分项`
4. 偏离表中的表格行可归到 `采购需求 / 偏离项`
5. 长条款子片段会保留同一个 `parent_unit_id`
6. 业务字段进入解析结果后，审查执行与 worker 闭环仍可通过

## 6. 当前结论

本轮正式结论如下：

1. 原业务化解析要求已开始进入真实实现，不再只停留在业务文档
2. 当前主链路已从“只有技术切片”推进到“带业务模块和业务单元类型的审查单元”
3. 当前已经覆盖原方案中的 `偏离项` 和“子片段保留父单元归属”这两个最小缺口
4. 这为后续继续实现独立 `module_block`、`review_unit` 和更细的表格行对象打下了基础

## 7. 下一步建议

1. 继续把高价值模块从“字段分类”推进到“独立解析对象”
2. 优先把资格条件、评分办法、采购需求做成更稳定的业务单元入口
3. 后续再补 `module_block` 和真正独立的 `review_unit` 数据结构
