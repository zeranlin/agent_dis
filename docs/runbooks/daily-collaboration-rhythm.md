# 三线程日常协作节奏

## 目的

这份文档用于固化总负责人、产品经理 P、技术负责人 T 的日常协作节奏，帮助项目持续推进时保持：

- 角色清晰
- 节奏稳定
- 交接明确
- 状态可见

## 适用范围

当前节奏适用于本仓库内的三线程协作方式：

1. 总负责人线程
2. 产品经理 P 线程
3. 技术负责人 T 线程

三者共享同一个仓库，但各自承担不同职责。

## 总体原则

- 先看仓库状态，再开新讨论
- 先写入文档，再视为正式结论
- 先完成当前层级，再进入下一层级
- 优先形成单一下一步动作，避免三边同时发散

## 1. 每次开始工作前

每个线程开始工作前，都应先做以下动作：

1. 优先阅读 `docs/status/summary.md`
2. 阅读与自己角色对应的短状态文件
3. 阅读对应收件箱中的待处理消息
这里的“待处理消息”默认指当前唯一有效消息，不包含历史归档。
4. 必要时再阅读 `docs/tasks/current.md`
5. 明确本轮只推进一个优先主题

对应角色重点查看内容：

- 总负责人：当前任务状态、P/T 最新交付、是否存在待拍板问题
- 产品经理 P：当前产品主题、边界是否已收口、是否有 T 回提问题
- 技术负责人 T：当前技术主题、产品输入是否齐全、是否存在待确认范围

短状态入口：

- 总负责人：`docs/tasks/owner-now.md`
- 产品经理 P：`docs/tasks/p-now.md`
- 技术负责人 T：`docs/tasks/t-now.md`

## 2. 标准推进顺序

默认情况下，三线程应按以下顺序协作：

1. 总负责人决定当前优先主题
2. 产品经理 P 先补齐产品层结论
3. P 向 T 交接已收口内容
4. 技术负责人 T 输出技术方案
5. T 向总负责人交接技术结论
6. 总负责人决定是否进入实现阶段或继续细化

如果当前主题已经完成产品收口，则可以直接由 T 继续推进技术层。

## 3. 日常协作节奏

### 3.1 总负责人节奏

总负责人每轮应做：

1. 先看 `docs/status/summary.md`
2. 再看 `docs/tasks/owner-now.md`
3. 再看 P 或 T 最新提交与对应 inbox
4. 必要时再看 `docs/tasks/current.md`
5. 判断当前谁应该继续推进
6. 给出唯一优先动作
7. 把判断写回仓库

总负责人不应同时给 P 和 T 派大量并行主题。

更稳妥的做法是：

- 同一时间只推进一个主主题
- 明确当前是产品推进轮，还是技术推进轮

### 3.2 产品经理 P 节奏

P 每轮应做：

1. 看当前主题是否已明确到自己这一层
2. 输出当前层级最佳方案
3. 收口范围、非范围、输入、输出、验收口径
4. 写入 `docs/business/` 或 `docs/business/v1/`
5. 在 `docs/tasks/current.md` 中更新交接状态

P 完成当前层级后，应主动判断：

- 是否已达到交接给 T 的条件
- 是否还有待总负责人拍板的问题

### 3.3 技术负责人 T 节奏

T 每轮应做：

1. 确认产品输入是否齐全
2. 基于已确认边界输出技术方案
3. 优先补主流程方案，再补子流程方案
4. 写入 `docs/tech/v1/`
5. 在 `docs/tasks/current.md` 中更新当前技术推进状态

T 如发现问题，应先区分：

- 是技术实现问题
- 还是产品边界问题

如果是技术实现问题，T 自行收口。
如果是产品边界问题，T 应回提总负责人，不自行改产品结论。

## 4. 每轮交接动作

### 4.1 P 交给 T

P 完成一轮后，至少应补：

- 当前交接主题
- 已确认范围
- 暂不纳入范围
- 输入对象
- 输出对象
- 当前验收口径
- T 下一步动作

### 4.2 T 交给总负责人

T 完成一轮后，至少应补：

- 当前技术主题
- 技术方案范围
- 不做内容
- 模块划分
- 状态流转
- 接口或对象草案
- 风险与假设
- 是否可进入开发

### 4.3 总负责人收口

总负责人每轮收口时，应至少判断：

- 当前层级是否已完成
- 是否可以进入下一层级
- 下一步由谁继续推进
- 当前唯一优先动作是什么

## 5. 任务状态更新规则

`docs/tasks/current.md` 是三线程共享的当前状态总板，但在低 token 模式下，它不再是每轮默认第一入口。

进一步约定：

- `docs/tasks/current.md` 是全局状态面
- `docs/tasks/owner-now.md`、`docs/tasks/p-now.md`、`docs/tasks/t-now.md` 是角色工作面
- 技术负责人 T 可以更新 `docs/tasks/current.md`
- 但技术负责人 T 只应在“一轮实现完成并准备回提”时更新 `docs/tasks/current.md`
- 日常推进过程中，技术负责人 T 应优先更新 `docs/tasks/t-now.md`
- 总负责人在复审后负责统一收口 `docs/tasks/current.md`、`docs/status/summary.md` 与 `docs/comms/inbox-*.md`
- 当前默认节奏改为“同主题整包回提优先”
- 只有主题明显过大、一次回提风险过高时，才退回到同主题分包回提

建议每轮至少更新以下内容：

- 当前目标
- 当前进展
- 当前结论
- 当前任务列表
- 下一步建议
- 阻塞或待确认

如果本轮发生正式交接，还应更新：

- 当前交接主题
- 交接给谁的内容
- 对方下一步要做什么

如果本轮需要异步提问或反馈，也可同步写入：

- `docs/comms/` 下对应 inbox 文件

如果历史消息已经处理完成，应定期移入：

- `docs/comms/archive/`

如果当前状态文件已经明显变长，应定期整理到：

- `docs/tasks/archive/`

更严格的低 token 做法是：

- `inbox-*.md` 只保留当前唯一有效消息
- 历史消息不继续保留在 inbox 中
- 只有在复盘时才读取 `archive/` 下归档文件

## 6. 推荐工作节奏

如果你日常是人工切换多个线程，推荐采用以下节奏：

1. 先打开总负责人线程
2. 先看 `docs/status/summary.md`
3. 再看 `docs/tasks/owner-now.md`
4. 判断先去 P 还是先去 T
5. 在对应线程推进一轮
6. 回到总负责人线程收口
7. 必要时再切到另一个线程

简单理解就是：

- 由总负责人决定节奏
- 由 P 或 T 执行当前轮次任务
- 再回到总负责人统一收口

## 7. 什么时候找谁

日常可直接按以下规则路由：

- 产品问题找 P
- 技术问题找 T
- 范围和优先级问题找总负责人

具体对照可见：

- `docs/runbooks/question-routing.md`

异步消息通道可见：

- `docs/comms/README.md`

## 8. 异常情况处理

出现以下情况时，不要继续推进下一级，而应先停下来收口：

- 当前层级还没收口
- P 和 T 对范围理解不一致
- 技术方案明显超出当前产品边界
- 当前主题出现多个并行优先级冲突
- `docs/tasks/current.md` 与仓库实际状态不一致

## 当前结论

三线程协作的关键，不是让三个线程同时发散，而是让它们围绕同一个仓库、同一块状态板、同一条主线，轮流推进并持续收口。
